A Course · Advanced → Expert · Scaling Snip across many machines

Some part is always failing.
Build a system that stays up anyway.

A single machine is easy to reason about: one clock, one memory, and when it dies, everything stops together. A distributed system is the opposite on every count — many machines, no shared clock, and partial failure as the normal state, where some nodes are up, some are down, and some are merely slow in a way you can't tell from dead. That one fact reshapes everything. You partition data to scale and replicate it to survive failure; you need consensus to agree without a leader you can trust; you reason about time without a global clock; and under a network partition, the CAP theorem forces you to choose between staying consistent and staying available. Ten labs turn these abstractions into things you can watch happen — as we scale Snip from one box to a global fleet.

AdvancedModule 00

Why distributed is different

Everything you know about programming assumes a single machine: one clock, one address space, and a simple failure model where the whole thing crashes at once. A distributed system breaks all three assumptions, and the consequences are the entire field. The defining property is partial failure: at any moment some nodes are running, some have crashed, and some are simply slow — and you fundamentally cannot tell a slow node from a dead one, because the only evidence you have is the absence of a reply. Add that there is no global clock to order events across machines, and that the network can drop, delay, duplicate, or reorder any message, and you have a world where the old certainties are gone.

The classic "fallacies of distributed computing" name the traps: the network is not reliable, latency is not zero, bandwidth is not infinite, and the topology is not stable. Every technique in this course is a response to those facts. You partition data across nodes to scale beyond one machine, and replicate it so a failure doesn't lose it. You use consensus to make a group of unreliable nodes agree on a single value. You reason about ordering with logical clocks instead of wall time. And you accept the CAP tradeoff: when the network splits, you cannot have both perfect consistency and full availability. We'll build all of it by scaling Snip — one URL shortener, from a single box to a fault-tolerant global fleet.

the shift in assumptions 
// single machine:
var hits = store.Get(code);     // one memory, one clock, all-or-nothing failure

// distributed:
var hits = await replicas.Quorum(code);  // which replica? is it current? is that node even alive?
// the network may drop this call, or the node may be slow, not dead — you can't tell

Hold the core truth the whole way through: in a distributed system, failure is not an exception you handle — it is the steady state you design for. The goal is never to prevent failure (you can't); it's to build a system that keeps serving correctly while pieces of it are broken.

↳ The Capstone · Mastering ASP.NET Core · SDLC
Snip was built and deployed on a single server in the earlier courses. This one asks the next question every successful system faces: what happens when one machine isn't enough, or when that one machine fails? The lifecycle course treated outages as incidents; here, tolerating them is the architecture.
A distributed system is just a program running on several computers.
It's a program that must stay correct while parts of it fail independently and messages between them are lost, delayed, or reordered — with no shared clock or memory. The hard part isn't spreading work across machines; it's the coordination and failure-handling that spreading work forces on you.
If a node stops responding, it has crashed.
You can't know that. A slow network, a long GC pause, or an overloaded node looks identical to a dead one — the only signal is silence. This impossibility of distinguishing slow from dead is at the root of timeouts, false failure detection, and much of the difficulty in the whole field.
ScaleAdvancedModule 01

Partitioning & scale

To scale past one machine, you partition (shard) data across nodes — each node owns a slice of the keys. The question is how to map keys to nodes. The obvious answer, hash(key) % N, has a fatal flaw: change N (add or remove a node) and the modulus shifts for almost every key, so nearly all your data has to move — a catastrophic reshuffle. Consistent hashing fixes this by placing both nodes and keys on a ring; each key belongs to the next node clockwise. Now adding a node only steals the keys in one arc — roughly K/N of them — and everything else stays put. It's the foundation of every scalable data store.

sharding.cs 
// naive — remaps almost everything when N changes:
int node = Hash(code) % N;              // add a node → ~all keys move

// consistent hashing — only one arc moves:
int node = ring.NextClockwise(Hash(code));  // add a node → ~K/N keys move

Twelve Snip codes are sharded across the nodes. Pick a strategy, then add or remove a node and watch how many keys have to move — modulo reshuffles almost everything, consistent hashing barely disturbs the ring:

Modulo hashing scatters keys by remainder, so changing N moves almost all of them. Consistent hashing puts keys and nodes on a ring; a new node only takes over the arc before it — a small fraction move, the rest stay home.
↳ Data Structures · Advanced SQL (partitioning)
The ring is a hash structure from Data Structures repurposed for placement, and the sharding idea is the distributed cousin of table partitioning from Advanced SQL — split one big dataset into pieces so no single node (or partition) holds it all.
hash(key) % N is a fine way to shard across nodes.
It works until N changes — then the remainder shifts for nearly every key and you reshuffle almost the entire dataset, a huge, disruptive data movement. Real systems use consistent hashing (often with virtual nodes) so scaling moves only a small, bounded slice.
Sharding just spreads load; it's otherwise transparent.
It introduces hard questions: how to route a request to the right shard, how to run a query that spans shards, and how to rebalance when nodes come and go. Partitioning buys scale but complicates everything that used to be a single local lookup.
ReplicationAdvancedModule 02

Replication & quorums

Partitioning scales you out; replication keeps you alive. Store each piece of data on N nodes so that losing one doesn't lose the data — but now the copies can disagree, and you must decide how consistent to keep them. The elegant tool is quorums: require a write to reach W replicas and a read to consult R of them. The key inequality is W + R > N: when it holds, every read quorum is guaranteed to overlap every write quorum by at least one node, so a read always sees the latest write — strong consistency. Loosen it (W + R ≤ N) and reads get faster and more available, but may return stale data. Tuning N, W, and R is how you dial the consistency/availability trade directly.

quorum.cs 
const int N = 5;   // replicas per key
await WriteTo(W: 3, value);     // acknowledged once W replicas store it
var v = await ReadFrom(R: 3);   // W + R = 6 > 5 → read overlaps the write → current
// with W=1, R=1: W + R = 2 ≤ 5 → the read may miss the latest write (stale)

Five replicas hold a value. Tune W and R, write a new value v2 to W replicas, then read from R — and see whether the read is guaranteed to observe v2 or can come back stale:

When W + R > N, the write set and read set must share a node, so the read always sees v2 — strong consistency. When W + R ≤ N, the sets can be disjoint and the read can return the old v1. Same replicas, different guarantee, set entirely by the numbers.
↳ Advanced C# · Vol. I (the memory model)
The read/write overlap is the distributed echo of the memory-model lesson: just as an unsynchronized count++ can miss an update on one machine, disjoint read/write quorums can miss a write across machines. Correctness in both comes from forcing an overlap — a happens-before that guarantees the reader sees the writer.
More replicas automatically means stronger consistency.
Replicas give durability and availability; consistency comes from the quorum rule. Five replicas with W=R=1 are highly available but happily return stale reads. It's W + R > N — the overlap — that buys consistency, and you pay for it in latency and availability.
You have to choose one consistency level for the whole system.
Quorums let you tune it per operation. A financial write might demand a full quorum while an analytics read tolerates staleness for speed. The N/W/R knobs make the consistency/availability trade a dial you set per use case, not a single global switch.
ConsensusExpertModule 03

Consensus & Raft

How does a group of unreliable nodes agree on a single value — a leader, the next log entry, a config change — when any of them can fail and messages can be lost? That's the consensus problem, and Raft is the algorithm that made it teachable. Raft elects a single leader for a term: a candidate requests votes, and whoever wins a majority leads. The leader accepts new entries and replicates them; an entry is committed once a majority of nodes have stored it — so it survives any minority failure. If the leader dies, its followers time out, a new term begins, and a new leader is elected. Majority (quorum) everywhere is the trick: any two majorities overlap, so decisions can't contradict.

raft.txt 
1. leader election  → candidate wins a MAJORITY of votes → leader for term T
2. log replication  → leader appends entry → committed once a MAJORITY store it
3. leader failure   → followers time out → term T+1 → elect a new leader
// any two majorities of N nodes overlap → no two conflicting decisions can commit

Run an election across five nodes, replicate a log entry (committed only when a majority acknowledges), then kill the leader and watch the cluster elect a new one in a fresh term:

A candidate needs a majority (3 of 5) to lead. An entry commits only once a majority stores it, so it can't be lost to a minority failure. Kill the leader and a follower wins the next term — the cluster heals itself.
↳ Replication & quorums (m02) · Design Patterns
Consensus is the quorum idea from m02 turned into agreement: majorities overlap, so no two conflicting values can both win. It's how a replicated store elects which node leads and which writes commit — the coordination layer beneath the replication you just tuned.
Nodes can just vote and take the majority answer — consensus is easy.
The difficulty is doing it while nodes fail and messages drop, without ever committing two conflicting values. Raft's terms, majority votes, and commit rules exist precisely to stay correct through failures — and even so, the FLP result proves no consensus algorithm can guarantee termination in a fully asynchronous network.
A committed entry can still be lost if nodes crash.
No — "committed" means a majority already stored it, and any future leader must be elected by a majority, which therefore includes a node that has the entry. That overlap guarantees committed data survives any minority failure. It's the whole safety argument in one sentence.
TimeExpertModule 04

Time & ordering

There is no global clock, and wall-clock timestamps from different machines can't be trusted to order events — clocks drift, and "later" by one node's clock may be "earlier" by another's. So distributed systems track logical time. A Lamport clock is a per-node counter: increment it on every local event, attach it to every message you send, and on receiving a message set your counter to max(local, received) + 1. This guarantees that if event A causes B (a message flows A→B), then A's timestamp is less than B's — capturing the happens-before relation without any real clock. Vector clocks go further, detecting when two events are concurrent (neither caused the other) — exactly the situation that produces conflicting updates.

lamport.txt 
local event  →  clock = clock + 1
send msg     →  clock = clock + 1;  attach clock to the message
receive msg  →  clock = max(local, received) + 1      // causality: cause < effect

↳ Bits, Pointers & Overflow · Advanced C# (the memory model)
Happens-before is the same relation the memory model used to order reads and writes across threads — here generalized across machines. Both replace "what time is it?" with "what provably came before what?", because that causal ordering is the only kind you can actually guarantee.
Just use timestamps to order events across servers.
Wall clocks drift and skew; two servers can disagree on which of two events was first, and NTP only bounds the error. Logical clocks order events by causality instead, which is what you actually need — and what physical time can't reliably give you across machines.
If two events have different Lamport timestamps, one caused the other.
Not necessarily — Lamport clocks guarantee cause < effect, but the reverse doesn't hold: unrelated (concurrent) events can still get different numbers. To actually detect concurrency you need vector clocks, which is why conflict detection uses them rather than a single counter.
ConsistencyExpertModule 05

Consistency models

"Consistency" isn't one thing — it's a spectrum of guarantees about what a read is allowed to return, and the stronger the guarantee, the more coordination (and latency) it costs. Linearizable is the strongest: every operation appears to take effect at a single instant, so a read always returns the most recent write, matching real-time order. Sequential keeps a single order consistent with each client's program order, but not necessarily real time. Causal guarantees only that causally-related writes are seen in order (and reads never go backward). Eventual is the weakest: reads may return stale values, with the sole promise that if writes stop, all replicas converge. Choosing a model is choosing how much reality you're willing to blur for speed.

a writer sets x = 1 then x = 2 · a reader reads three times 
linearizable → 2, 2, 2   // always the latest write
causal       → 1, 2, 2   // monotonic — never goes backward
eventual     → 0, 1, 2   // stale reads allowed; converges in the end

Linearizable forces every read to the latest write. Causal only forbids going backward, so an early read can still be stale but the sequence never regresses. Eventual permits any stale value, promising only that the replicas converge once writes stop.
Eventual consistency means the data is basically correct.
It means reads can return arbitrarily stale values until replicas converge — and "eventually" has no time bound. It's the right choice for many workloads (likes, view counts), but treating it as "almost strong" leads to bugs where the application acts on data that's simply out of date.
Stronger consistency is always better; use it everywhere.
Strong consistency costs coordination and latency and reduces availability under partitions. The engineering skill is matching the model to the operation — linearizable where correctness demands it, weaker where staleness is harmless — because paying for linearizability everywhere makes a system slow and fragile.
CAPExpertModule 06

The CAP theorem

The most famous result in the field, and the most misunderstood. CAP says that when a network partition splits your nodes into groups that can't talk, you must choose between consistency (every read sees the latest write) and availability (every request gets a non-error response) — you cannot have both across the split. A CP system refuses to serve on the side that can't reach a quorum, staying consistent by becoming partly unavailable. An AP system keeps serving on both sides, staying available by letting the copies diverge (to be reconciled later). The crucial clarification: partitions are rare but inevitable, so CAP is really about what you do during one — and PACELC adds that even without a partition, you still trade latency against consistency.

during a partition 
CP  → minority side returns errors  // consistent, but not available
AP  → both sides keep serving       // available, but the copies diverge
// no partition (PACELC): still choose lower Latency or stronger Consistency

The cluster is split into two partitions. Choose CP or AP, induce the partition, and issue writes to each side — then heal it and see the consequence: a rejected write and consistency, or two writes and a divergence to reconcile:

CP freezes the minority side — writes to B are rejected, so nothing diverges and consistency holds at the cost of availability. AP serves both sides, so A and B accept different writes and diverge, and healing forces a conflict to reconcile. That's the whole theorem, lived.
↳ Replication & quorums (m02) · Consistency models (m05)
CAP is the quorum inequality under stress: a partition can make a quorum unreachable on one side, and then W + R > N can't be satisfied there. CP chooses to fail that side; AP chooses to serve it with weaker (eventual) consistency from m05. The knobs you tuned earlier are where you sit on CAP.
CAP means you pick two of consistency, availability, and partition-tolerance.
Partition-tolerance isn't optional on a real network — partitions will happen — so you don't get to "pick" P. The real choice is C vs A when a partition occurs. Framing it as "two of three" obscures that every distributed system must tolerate partitions and decide how to behave during them.
A good system is both consistent and available, full stop.
Only when there's no partition. During a split, physics forces the choice — and even when connected, PACELC says you still trade latency for consistency. "Consistent and available" is achievable in the happy path; the design work is deciding what happens when the happy path breaks.
TransactionsExpertModule 07

Distributed transactions

A single-database transaction is atomic for free. Spanning a change across several services or shards is far harder, and the classic answer is two-phase commit (2PC). A coordinator runs a prepare phase — asking every participant "can you commit?" — and only if all vote yes does it send commit; any no triggers a global abort. It gives atomicity, but it has a notorious weakness: if the coordinator fails after prepare but before deciding, the participants are blocked — they've promised to commit, hold their locks, and can't safely proceed without the coordinator. Because of that, many systems avoid 2PC in favor of sagas: a sequence of local transactions, each with a compensating action to undo it if a later step fails — trading atomicity for availability.

two-phase commit 
phase 1 (prepare) → coordinator asks all: "can you commit?"
phase 2 (commit)  → all voted yes → COMMIT   |   any voted no → ABORT
// coordinator dies between phases → participants are BLOCKED, holding locks

Prepare then commit gives all-or-nothing atomicity across services. But a coordinator crash after prepare leaves participants holding locks with no safe way to decide — the blocking problem that pushes many systems toward sagas and compensating actions instead.
↳ Mastering EF Core · Advanced SQL (transactions)
The single-node ACID transaction from the database courses is the guarantee 2PC tries to stretch across machines — and the difficulty of doing so is exactly why microservice architectures lean on sagas and eventual consistency instead of distributed ACID. Atomicity is cheap in one database and expensive across many.
Two-phase commit gives you ACID transactions across services.
It gives atomicity, but at the cost of availability: it's a blocking protocol that stalls when the coordinator fails, and it doesn't scale well. Distributed ACID is possible but expensive — which is why most large systems choose sagas plus idempotency and accept eventual consistency.
A saga is just a transaction with extra steps.
A saga has no global atomicity — intermediate states are visible, and instead of a rollback it runs compensating actions to semantically undo completed steps. That's a genuinely different model: you design the undo for each step and tolerate temporary inconsistency, in exchange for staying available and scalable.
MessagingExpertModule 08

Messaging & delivery

Services talk through messages, and because the network can lose them, you must choose a delivery guarantee — each a different answer to "what if the acknowledgment is lost?" At-most-once never retries, so a message may be lost but never duplicated. At-least-once retries until acknowledged, so it's never lost but may be duplicated (the retry arrives after the original was already processed). True exactly-once delivery is impossible over an unreliable network — but you get its effect by combining at-least-once with idempotency or deduplication: give each message an id, and a consumer that has seen an id simply drops the repeat. That's how Snip can ingest a burst of click events, retry safely, and still count each click once.

delivery.txt 
at-most-once   → send, don't retry        // may LOSE, never duplicates
at-least-once  → retry until acked         // never lost, may DUPLICATE
exactly-once*  → at-least-once + dedup(id)  // duplicates dropped → processed once

At-most-once processes it once but risks losing messages whose send is dropped. At-least-once retries and the consumer processes a duplicate. Exactly-once retries too, but the consumer recognizes the message id and drops the repeat — processed once, safely.
↳ Advanced C# · Vol. I (channels & backpressure)
The message queue is the distributed version of the bounded channel from Advanced C#: it decouples producer from consumer and applies backpressure when the consumer falls behind — but now across machines, where retries and duplicates make idempotency essential rather than optional.
Use exactly-once delivery and you never worry about duplicates.
Exactly-once delivery is impossible on a lossy network; what's real is exactly-once processing, built from at-least-once plus idempotent consumers or dedup by id. The duplicates still arrive — you design the consumer so a repeat has no effect, rather than pretending the network won't repeat.
Idempotency is a nice-to-have optimization.
In an at-least-once world it's a correctness requirement. Retries are guaranteed to redeliver messages, so any non-idempotent handler will eventually double-charge, double-count, or double-send. Designing operations to be safely repeatable is one of the most important habits in distributed systems.
FailureExpertModule 09

Failure & circuit breakers

Because you can't tell a slow dependency from a dead one, calling a failing service naively is dangerous: requests pile up waiting on timeouts, threads and connections exhaust, and the failure cascades to callers that were themselves healthy. The circuit breaker is the standard defense, a small state machine wrapping a remote call. In closed state calls flow normally while failures are counted; once failures cross a threshold the breaker trips open and calls fail fast without even trying — protecting both caller and the struggling dependency. After a cooldown it goes half-open and lets a single probe through: success closes it, failure opens it again. Paired with retries with backoff, it turns a cascading outage into a contained, self-healing one.

circuit-breaker.txt 
CLOSED     → calls pass, count failures → threshold reached → trip
OPEN       → fail fast, don't call the dependency  // stop the cascade
HALF-OPEN  → allow one probe → success ? CLOSED : OPEN   // self-healing

Three consecutive failures trip the breaker open, and further calls fail instantly instead of hanging on timeouts — stopping the cascade. A cooldown allows one half-open probe; if the dependency has recovered, the breaker closes and normal traffic resumes.
↳ Mastering Testing in .NET · SDLC
Circuit breakers, timeouts, and backoff are resilience patterns you'd verify with the fault-injection and chaos testing from the testing and lifecycle courses — deliberately failing a dependency to prove the system degrades gracefully instead of collapsing. Resilience is a property you test, not assume.
Retrying a failed call is the right way to handle failure.
Naive retries make things worse — they add load to a service that's already struggling, amplifying the outage (a "retry storm"). Retries need backoff and jitter, and a circuit breaker to stop retrying entirely once a dependency is clearly down. Blind retries turn a small failure into a large one.
A single failing service only affects that service.
Without isolation, its callers block on timeouts waiting for it, exhausting their own threads and connections, so they fail too — a cascade that can take down an entire system from one weak link. Bulkheads, timeouts, and circuit breakers exist to contain failure to where it started.
ConvergenceExpertModule 10

Conflict resolution & CRDTs

An AP system lets replicas accept writes independently, which means they will diverge — and something has to merge them back. The naive fix, last-write-wins (keep the value with the highest timestamp), is simple but lossy: two concurrent increments to a counter merge to one, silently dropping the other. Conflict-free replicated data types (CRDTs) solve this by designing the data so that merging is commutative, associative, and idempotent — replicas can merge in any order, any number of times, and always converge to the same correct result with no coordination. A grow-only counter (G-Counter) keeps a per-replica count and merges by taking the max of each entry, then summing — so two concurrent increments both survive. It's how you count Snip's clicks across regions without a global lock and without losing a single one.

merge two replicas that each got increments 
last-write-wins → keep max(a, b)   // one side's increments are LOST

G-Counter       → per-replica: { A: a, B: b };  merge = max each;  total = sum
                // both sides survive → converges to a + b, no coordination

Last-write-wins keeps only the larger counter, discarding the other replica's increments — data loss on every concurrent update. The G-Counter tracks each replica separately and sums them, so both sides' increments survive and every replica converges to the same total.
↳ Time & ordering (m04) · Advanced C# (the memory model)
CRDTs are the constructive answer to the concurrency m04 could only detect: instead of flagging concurrent updates as conflicts, they define a merge that makes conflicts impossible. It's the distributed, coordination-free cousin of the atomic increment — correctness by design rather than by locking.
Last-write-wins is a reasonable default for merging replicas.
It's easy but silently loses data whenever writes are concurrent — the "losing" write just vanishes, which for counters, sets, or carts is a real bug. LWW is only safe when losing concurrent updates is genuinely acceptable; otherwise you need a CRDT or explicit conflict resolution.
Merging divergent replicas always needs coordination or a conflict resolver.
Not if the data type is a CRDT. By making merge commutative, associative, and idempotent, replicas converge automatically no matter the order or repetition of merges — no locks, no coordinator, no manual resolution. The cleverness is moved into the data structure's design.
ExpertModule 11

The whole system

Every technique in this course is a response to the same two facts — partial failure and no global clock — and together they compose into a system that scales and survives. You partition to grow, replicate with quorums to stay durable and tune consistency, run consensus to agree, order events with logical clocks, choose your spot on CAP during partitions, coordinate change with 2PC or sagas, move work through idempotent messaging, contain failure with circuit breakers, and merge divergence with CRDTs. Cluster config is agreed by Raft (m03); events are ordered by logical clocks (m04). Link creation runs CP for correctness while click counting runs AP for availability (m05–06). Clicks flow through an idempotent queue so retries never double-count (m08), dependencies are wrapped in circuit breakers (m09), and per-region hit counts merge with a CRDT that can't lose a write (m10). No single machine is essential, no single failure is fatal, and the system stays correct while pieces of it break — which is the entire point.

That's the distributed-systems mindset — assume failure, design for partial breakage, coordinate only as much as you must, and prefer techniques that converge on their own over ones that need a fragile central authority.
