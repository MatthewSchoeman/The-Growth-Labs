You work in objects.
EF speaks SQL.

Entity Framework Core is the object-relational mapper for .NET. You define plain C# classes, query and change them as ordinary objects, and EF does the database conversation for you — mapping classes to tables, translating your LINQ into SQL, tracking every change you make, and working out the exact INSERT/UPDATE/DELETE to save them. This course is the data layer that completes the track: the C# you learned, served over the web with ASP.NET, now backed by a real database.

BeginnerModule 00

What is EF Core

There's a fundamental mismatch at the heart of most applications. Your code thinks in objects — classes with properties, references between them, whole graphs in memory. Your database thinks in tables — flat rows, columns, and foreign keys. Bridging that gap by hand means writing endless SQL strings, mapping result columns back into objects, and tracking what changed. An object-relational mapper (ORM) automates the bridge, and EF Core is the ORM for .NET.

With EF Core you define plain C# classes, and it handles the rest: it generates the SELECT from your LINQ, materializes rows into objects, watches those objects for changes, and produces the INSERT/UPDATE/DELETE to persist them. It's provider-based — the same code runs against SQL Server, PostgreSQL, SQLite, and more by swapping one package. You trade a measure of low-level control for an enormous reduction in boilerplate.

↳ SQL course
EF Core doesn't replace what you learned in SQL & PostgreSQL — it sits on top of it. Everything EF generates is SQL hitting tables, indexes, and joins. Understanding the SQL underneath is exactly what separates someone who uses EF well from someone who fights it.

So the mental model for this whole course is the thesis: you work in objects, and EF speaks SQL on your behalf. You rarely write a query string. You write LINQ and manipulate objects; EF translates. The skill is knowing what that translation does — which is what every widget here makes visible.

An ORM means I never have to learn SQL.
The opposite, in practice. EF writes the SQL, but you have to read it to spot a missing index, an accidental table scan, or an N+1 storm. EF is a productivity multiplier on top of SQL knowledge, not a replacement for it.
EF Core is just slow — real apps use raw SQL.
Well-used EF is fast enough for the vast majority of work, and you can always drop to raw SQL for the hot 1%. Most "EF is slow" stories are really "this query was written carelessly" — fetching whole entities to read one column, or triggering hundreds of round-trips (Module 09).
ModelBeginnerModule 01

Your model & DbContext

An EF model is just plain C# classes — your entities — plus a DbContext that ties them to a database. The context exposes a DbSet<T> for each entity type, and each DbSet is your query gateway and your handle for adding and removing rows. The DbContext is three things at once: a query gateway, a change tracker, and a unit of work that batches everything into one SaveChanges.

c#public class Blog                       // an entity — a POCO
{
    public int Id { get; set; }         // convention: Id is the primary key
    public string Title { get; set; } = "";
}

public class BloggingContext : DbContext
{
    public DbSet<Blog> Blogs => Set<Blog>();   // maps to the Blogs table
    protected override void OnConfiguring(DbContextOptionsBuilder o)
        => o.UseSqlite("Data Source=blog.db");     // pick a provider
}

Using it is ordinary C#. You add an object, save, and query — no SQL in sight:

c#using var db = new BloggingContext();
db.Blogs.Add(new Blog { Title = "Hello EF" });   // staged as Added
db.SaveChanges();                              // INSERT runs here

var blogs = db.Blogs.Where(b => b.Title.Contains("EF")).ToList();
↳ ASP.NET course
Remember service lifetimes? In ASP.NET Core the DbContext is registered as a scoped service — one per request — precisely because it's a short-lived unit of work that isn't thread-safe. The "2 instances, one per request" behavior from the DI widget is the DbContext's natural home (Module 10).
A DbContext is meant to live a long time and be shared.
It's deliberately short-lived and single-threaded — create one, do a unit of work, dispose it. Its change tracker accumulates state, so a long-lived shared context leaks memory, returns stale data, and breaks under concurrent use.
A DbSet holds the table's data in memory.
A DbSet is a query gateway, not a collection. Touching db.Blogs loads nothing; only when you enumerate a query (or call ToList) does EF run SQL and fetch just the rows that query asked for.
MappingBeginnerModule 02

Mapping classes to tables

EF figures out most of your schema by convention, so a clean class needs almost no configuration. A class becomes a table; each property becomes a column; a property named Id (or BlogId) becomes the primary key; C# types map to sensible SQL types; and a non-nullable reference becomes a NOT NULL column. When conventions aren't enough, you override them with data annotations on the property or the Fluent API in OnModelCreating.

c#public class Product
{
    public int Id { get; set; }
    [MaxLength(200)] public string Title { get; set; } = "";
    [Column("Cost")]   public decimal Price { get; set; }
    [NotMapped]      public string Slug { get; set; } = "";  // no column
}

↳ SQL course
The right-hand side is exactly the CREATE TABLE world from SQL & PostgreSQL — column types, NULL/NOT NULL, primary keys. EF is deciding that DDL for you from your C#, which is why knowing what good schema looks like still matters.
You have to configure every property explicitly.
Conventions cover the large majority — keys, column names, types, nullability. You only reach for annotations or the Fluent API for the exceptions: a length limit, a renamed column, a composite key, a relationship EF can't infer.
EF needs a database that already exists.
In the common code-first flow, your classes are the source of truth and EF creates the schema (via migrations, Module 08). There's also a database-first flow that scaffolds classes from an existing schema — EF supports both directions.
QueryBeginnerModule 03

Querying with LINQ

This is where EF earns its keep. You query your tables with the exact same LINQ you learned in the C# course — Where, OrderBy, Select — but instead of running over an in-memory list, EF translates the whole chain into one SQL statement and runs it on the database server. The rows that come back are materialized straight into your entity objects.

↳ C# course → SQL course
The LINQ builder from Mastering C# ran over a List in memory. The widget below is its EF sibling: the same clauses, but now you also see the SQL EF generates — the bridge between the two earlier courses made literal. LINQ in, SQL out.

Each clause becomes part of one SQL statement. EF runs it once and turns the rows into objects.

Because LINQ is deferred, none of this runs until you enumerate the query — with ToList(), foreach, First(), and so on. That deferral is exactly what lets EF fold every clause into a single round-trip rather than fetching early and filtering in memory.

db.Products.Where(...) fetches rows, then filters them in C#.
The Where becomes a SQL WHERE — the filtering happens on the database, and only matching rows ever travel back. Fetching-then-filtering in memory is the anti-pattern EF is built to avoid.
Writing the query executes it.
Defining a query builds an expression tree; nothing hits the database yet. Execution happens on enumeration. You can keep composing — add another Where, an OrderBy — and it all collapses into one statement when you finally read the results.
QueryIntermediateModule 04

Translation & projection

EF can only run what it can translate. Operators and expressions it understands — comparisons, string functions like Contains, aggregates — become SQL and run on the server. But call a method EF can't turn into SQL inside a query, and modern EF throws rather than silently fetching everything and running your code in memory. That refusal is a feature: it stops an accidental full-table download.

c#// ✓ translates — runs as SQL WHERE
db.Products.Where(p => p.Title.StartsWith("K") && p.Price < 50);

// ✗ MyCustomCheck() can't become SQL → throws (not a silent client scan)
db.Products.Where(p => MyCustomCheck(p));

The other half of writing efficient queries is projection. Selecting whole entities pulls every column; projecting with Select into a small shape tells EF to fetch only those columns. For read-only screens this is one of the biggest, cheapest wins available:

c#// pulls every column of every product
var all = db.Products.ToList();

// SELECT p.Id, p.Title only — less data over the wire
var slim = db.Products
    .Select(p => new { p.Id, p.Title })   // projection
    .ToList();

var page = db.Products.OrderBy(p => p.Id)
    .Skip(20).Take(10).ToList();    // OFFSET 20 FETCH 10
Any C# I write inside a LINQ query will just work.
Only translatable expressions work; arbitrary C# methods can't become SQL. EF throwing on an untranslatable query is protecting you from quietly pulling the whole table into memory to run that code client-side.
Projecting with Select doesn't really matter.
It changes the actual SELECT column list. Loading a 30-column entity to display two fields wastes bandwidth and memory; Select new { ... } fetches just what the screen needs — and projected results aren't tracked, which is lighter still (Module 09).
RelationshipsIntermediateModule 05

Relationships & loading

Real models have relationships. A Blog has many Posts; each Post belongs to one Blog. You express that with navigation properties (blog.Posts, post.Blog) and a foreign key (post.BlogId). EF infers the relationship and the join from these by convention.

c#public class Blog { public int Id; public List<Post> Posts = new(); }
public class Post { public int Id; public int BlogId; public Blog? Blog; }

By default, loading a blog does not load its posts — EF won't silently fan out queries behind your back. You ask for related data explicitly. The most common way is eager loading with Include, which adds a JOIN so the posts come back in the same round-trip. Toggle it below and watch both the loaded data and the SQL change:

↳ SQL course
Include is EF writing a JOIN for you — the same join from SQL & PostgreSQL. And just like a hand-written join, it can multiply rows, so EF sometimes splits it into separate queries instead. Knowing joins tells you what Include will cost.
Navigation properties load automatically when I touch them.
Only if you've opted into lazy loading (proxies) — and that convenience hides round-trips. By default a navigation is empty until you Include it, project it, or explicitly Load it. EF makes related-data fetching a deliberate choice.
Include everything — more data loaded up front is safer.
Each Include is a join that widens the result and can duplicate parent rows. Pull only the relationships a screen actually uses; over-including is a common source of bloated, slow queries.
TrackingIntermediateModule 06

The change tracker

Here's the piece that makes EF feel like magic. When you load entities, the context keeps a private snapshot of each one and assigns it a state. As you change those objects with ordinary C# — set a property, add a new one, remove another — EF records the new EntityState: Unchanged, Modified, Added, or Deleted. It does not touch the database yet. It's just remembering what you did.

↳ Design Patterns
The change tracker is the Unit of Work and Identity Map patterns from Design Patterns, built in. Unit of Work batches all your changes into one commit; Identity Map guarantees that two queries returning the same row hand you the same object instance, so your edits stay consistent.

Change the entities below as plain objects and watch their state shift. Nothing is saved — EF is simply tracking what SaveChanges will need to do:

Watching your objects
every change updates an EntityState
Each action changes an entity's state in the tracker — but the database is untouched until you save.
Setting a property writes to the database right away.
It only updates the in-memory object and its tracked state. The database changes nothing until SaveChanges. That gap is the whole point — it lets EF batch many edits into one atomic commit.
Change tracking is free, so always query with it on.
Tracking costs memory and a snapshot comparison. For read-only queries that won't be saved, AsNoTracking() skips all of it and is meaningfully faster (Module 09). Track when you intend to write; don't when you only intend to read.
SaveIntermediateModule 07

Saving changes

Calling SaveChanges commits the unit of work. EF walks every tracked entity and generates the right statement for its state: an INSERT for each Added, an UPDATE for each Modified (touching only the columns that actually changed), and a DELETE for each Deleted. It sends them as a batch inside a transaction — all succeed or all roll back — then flips every entity back to Unchanged and flows database-generated keys back into your objects.

↳ Design Patterns + SQL
One SaveChanges = one Unit of Work commit. The UPDATE sets only changed columns because the tracker knows the before-snapshot; the whole batch runs in a transaction — the same atomicity guarantee from the SQL course.
An UPDATE rewrites the entire row.
EF compares against the snapshot and writes only the columns you changed. Edit one field of a wide entity and the UPDATE SET lists exactly that one column — smaller, faster, and less likely to clobber a concurrent edit.
Ten changes mean ten separate database calls.
EF batches them and wraps the batch in one transaction. Ten edits across entities become a small number of round-trips that all commit together — or, if anything fails, all roll back, leaving the database consistent.
MigrationsIntermediateModule 08

Migrations

Your model is going to change — new properties, new tables, new indexes. Migrations are how EF evolves the database schema to match your classes, safely and in version control. You change the model, run dotnet ef migrations add <Name>, and EF diffs the model against its snapshot to generate a migration with an Up method (apply the change) and a Down method (reverse it). dotnet ef database update then runs the pending migrations.

Pick a change to the Product model and see the migration EF generates, then apply it to the schema:

Products schema
A model change generates an Up/Down migration; applying it alters the real schema.
EnsureCreated() is a fine way to set up the database.
It's for prototypes and tests — it creates the schema once and can't evolve it. Production needs migrations, which apply incremental, reversible, version-controlled changes. The two don't mix; pick migrations for anything real.
Adding a migration changes the database.
Two separate steps. migrations add only generates the code; the database changes when you run database update (or apply migrations at startup). That separation lets you review and edit the migration before it ever touches a database.
PerformanceIntermediateModule 09

Performance & pitfalls

The single most common EF performance bug has a name: the N+1 query problem. You run one query to load a list of parents, then loop over them touching a navigation property — and each access quietly fires another query. Load 1 list of blogs and read each blog's posts in a loop, and you've issued 1 + N round-trips instead of one. It's invisible in the C# and devastating at scale.

↳ Big O course
N+1 is a complexity problem from Big O Complexity, hiding in your data layer: the number of database round-trips grows linearly with the rows you loaded — O(n) trips where O(1) would do. The fix collapses n+1 trips into a single query, the same instinct as picking the better-scaling algorithm.

The naive loop fires one query per blog; Include fetches everything in a single round-trip.

The other everyday wins: use AsNoTracking() for read-only queries to skip the change tracker's overhead; project with Select so you fetch only needed columns; lean on database indexes for the columns you filter and sort on; and watch out for multiple Includes producing a row-multiplying cartesian join (EF can split those into separate queries).

N+1 is a rare edge case I'll probably never hit.
It's the most common EF performance problem, and it usually looks completely innocent — a foreach over results that reads a related property. It stays quiet in development with three rows and falls over in production with three thousand.
AsNoTracking is a risky micro-optimization.
For queries whose results you only read and never save, it's the right default — it skips snapshotting and identity management entirely. Reserve tracking for entities you actually intend to modify and persist.
IntegrationIntermediateModule 10

EF in ASP.NET Core

This is where the whole track converges. In an ASP.NET Core app you don't new up a context — you register it once with AddDbContext and let dependency injection hand a fresh one to each request. Because the DbContext is registered scoped, every request gets its own unit of work, created and disposed around that request automatically.

Program.csbuilder.Services.AddDbContext<BloggingContext>(o =>
    o.UseSqlite(connectionString));        // registered as a SCOPED service

app.MapGet("/blogs", async (BloggingContext db) =>   // injected per request
    await db.Blogs
        .AsNoTracking()
        .Select(b => new { b.Id, b.Title })
        .ToListAsync());                       // async all the way down
↳ ASP.NET course
Two ideas from Mastering ASP.NET Core land here exactly. The scoped lifetime from the DI widget is why a DbContext is one-per-request. And async all the way from the async module is why every EF call in a web app is ToListAsync/SaveChangesAsync — never block a request thread on the database.

You'll hear debates about wrapping EF in a repository layer. It's worth knowing that the DbContext already is a unit of work and its DbSets already are repositories, so an extra layer is optional — add one for testing seams or to hide EF from the rest of the app, skip it when it's just ceremony.

I should create a DbContext with new inside my controller.
Inject it instead. AddDbContext wires it as scoped, so the framework gives each request its own context and disposes it for you. Manual construction means manual lifetime management and loses the request-scoped guarantee.
Synchronous EF calls are fine on a web server.
Blocking a request thread on database I/O throttles throughput under load — the exact lesson from the ASP.NET async module. Use the ...Async methods and await them, all the way up the call chain.
IntermediateModule 11

Putting it together

You can now see the full loop. You define a model of plain classes; a DbContext maps them to tables and serves as your gateway; you query with LINQ that EF translates to SQL; the change tracker watches the objects you edit; SaveChanges commits them as a batched transaction; and migrations evolve the schema as the model grows. Every step kept you in C# while EF handled the SQL.

The habits that make EF pleasant are small and consistent: keep the context short-lived (one per request), go async for every database call, use AsNoTracking and projection for read-only screens, load relationships deliberately to dodge N+1, and keep your migrations in source control. None of it is exotic — it's just respecting what the SQL underneath is doing.

The track is complete — C# → ASP.NET → EF Core

Step back and the three courses form one stack. C# gave you the language: types, classes, generics, LINQ, async. ASP.NET Core turned it into a web server: a request flows through middleware to an endpoint and a response flows back. EF Core gave that server a memory: the endpoint runs C#, queries and saves through EF, and your LINQ becomes SQL against a real database. A request comes in, your code runs, the database answers, a response goes out — and it's all the same language, end to end.

From here, depth comes from building. Watch the SQL EF generates, profile the queries that matter, and reach for raw SQL on the rare hot path that needs it. You have the whole picture now — language, web, and data — and the judgment to use each part well.

You work in objects. EF speaks SQL.

From your first DbSet to change tracking, transactions, and migrations, one idea carried through: you manipulate plain C# objects and write LINQ, and EF Core conducts the entire database conversation — generating the SQL, tracking the diffs, and saving them safely. The skill was never avoiding SQL; it was knowing what EF does with it.

That closes the loop the C# track set out to draw. The language you learned, served over the web, backed by a database — one coherent stack, written in C# from the request all the way down to the row.
