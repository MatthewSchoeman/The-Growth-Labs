AdvancedModule 00

The pull-based model

You already know what a signal does. This course is about how — and it starts with a distinction most tutorials skip: reactivity can be push or pull. A push system (classic RxJS) sends new values downstream the instant they change; every subscriber is told immediately, whether it needs the value now or not. A pull system inverts this: writing a value doesn't compute anything — it just marks dependents stale. The actual recomputation happens later, lazily, only when someone reads. Angular's signals are pull-based, and that single choice explains almost everything that follows.

Two properties fall out of pull-based design for free. First, laziness: a computed nobody reads never runs, no matter how often its inputs change. Second, glitch-freedom: because values are pulled on demand, you can never observe an inconsistent intermediate state where one derived value has updated and another hasn't. Push systems fight both problems with operators and scheduling; pull systems don't have them to begin with.

the shape of it 
const count   = signal(1);              // a producer
const doubled = computed(() => count() * 2);  // producer + consumer

count.set(2);   // ← does NOT recompute doubled. It marks it dirty.
doubled();     // ← THIS recomputes (pull), then caches until the next write

Every node in this graph is a producer (something can read it), a consumer (it reads others), or both. Reading a producer inside a consumer records a dependency edge — automatically, no dependency arrays. That graph, and the version counters that drive its dirty-checking, is the machine you'll watch in the next module. Hold one sentence in mind for the whole course: writes mark; reads compute.

↳ Mastering Angular · Front-End Engineering
The intro course showed signals as UI = f(state) with a sharper pencil. Here we sharpen the pencil itself: the "graph knows who to recompute" claim becomes a mechanism you can single-step — producers, consumers, versions, and lazy pull.
Signals are just observables with nicer syntax.
They're the opposite propagation direction. Observables push values at subscribers; signals let readers pull values on demand. That's why signals need no unsubscribe, never glitch, and skip work nobody observes — none of which is a syntax difference.
Setting a signal immediately updates everything that depends on it.
Setting a signal updates nothing — it only bumps a version and marks consumers stale. The recompute is deferred until a read (for computed) or a scheduled flush (for effects). "Writes mark; reads compute" is the entire model.
Reactive graphAdvancedModule 01

Signals in depth

Now the machinery. Each writable signal carries a version — a counter bumped on every change. Each computed remembers the versions of the producers it last read; if any differ when the computed is pulled, it's dirty and recomputes, otherwise it returns its cached value. Effects are consumers with no readable output; they're scheduled to run after a change rather than run inline. The knobs you'll actually use: a custom equal to suppress no-op writes, untracked() to read without recording a dependency, and linkedSignal() for writable state that resets when a source changes.

reactive-depth.ts 
const count = signal(1, { equal: Object.is });   // equal → skip no-op sets
const doubled = computed(() => count() * 2);   // caches; recomputes only when pulled dirty
const label = computed(() => `badge ×${doubled()}`);  // pulls doubled transitively

effect(() => log(label()));      // scheduled consumer — runs on flush, not inline
const total = untracked(() => count());  // read WITHOUT a dependency edge

↳ Bits, Pointers & Overflow
Dirty-checking by version counter is the same trick as a monotonic sequence number: don't compare the (expensive) values, compare a cheap integer that only ever increases. Reference equality plus a version is how Angular decides "did this actually change?" in O(1).
A computed recalculates every time you read it.
It recalculates only when it's dirty — when a producer's version moved since its last pull. Read a clean computed a thousand times and it returns the same cached reference a thousand times, zero recomputation. Caching is the default, not an optimization you add.
effect() runs synchronously the moment a dependency changes.
Effects are scheduled, then run once per flush — which is how ten writes in a row cause one effect run, not ten. Effects are for the outside world (DOM, logging); deriving values belongs in computed, which the lab keeps strictly separate.
Async signalsAdvancedModule 02

Async as signals: resource()

Signals hold synchronous state; the network is asynchronous. Modern Angular bridges the gap with resource() (and rxResource()): you give it a request derived from signals and an async loader, and it hands back the whole async lifecycle as signal state — .value(), .status() (idle → loading → resolved | error), .error(), and a reload(). Crucially, when the request signal changes while a load is in flight, the resource aborts the stale one via an AbortSignal — the stale-response bug from the intro course, handled by the primitive itself.

link-detail.resource.ts 
const id = signal('4C92');

const link = resource({
  request: () => ({ code: id() }),                 // re-runs when id() changes
  loader: async ({ request, abortSignal }) => {      // stale loads get aborted
    const res = await fetch(`/api/links/${request.code}`, { signal: abortSignal });
    return res.json();
  },
});
// template: @if (link.status() === 'loading') { … } @else { {{ link.value()?.clicks }} }

Drive the lifecycle below. Load a link, then change the request while it's loading and watch the first load abort; flip the server to failing to see the error state render honestly:

The async lifecycle, as state
idle · loading · resolved · error · abort
IDLErequest = —
link.value() is undefined — nothing requested yet
status(), value(), and error() are all signals — the template reads them like any other state. Change the request mid-load and the resource cancels the work that no longer matters.
↳ Mastering Angular · ASP.NET Core
This is the intro course's fetch lifecycle and switchMap cancellation, folded into one primitive. The loader still calls your ASP.NET endpoint — but loading, error, and abort are no longer yours to wire; resource() owns them and exposes them as signals.
resource() is just fetch() wrapped in a signal.
It's a state machine with cancellation: it tracks status, exposes value and error as signals, re-runs when its request signal changes, and aborts the previous load through abortSignal. The wrapper you'd hand-roll — and get subtly wrong — is the point.
You should call the API inside an effect and store the result in a signal.
That's the anti-pattern resource() exists to replace: effects that write signals build hidden async cascades with no cancellation and no status. Declare the request as derived state; let the resource own the lifecycle. Effects stay for genuine side effects.
RxJSAdvancedModule 03

Higher-order streams

Signals own state; RxJS owns time. And the heart of RxJS is a problem signals never face: what do you do when each value from an outer stream starts an inner stream — a value triggering an HTTP call, say? You have to flatten outer-of-inner back into one stream, and there are exactly four strategies. Choosing the wrong one is the single most common RxJS bug in production, so learn them as a set:

mergeMap runs every inner concurrently and interleaves results — maximum throughput, no ordering. concatMap queues inners and runs them strictly in order, one finishing before the next starts — ordering preserved, latency accumulates. exhaustMap ignores new outer values while an inner is running — perfect for "submit" buttons that must not double-fire. switchMap cancels the current inner when a new outer arrives — perfect for typeahead, where only the latest query matters.

flattening.ts 
clicks$.pipe(mergeMap(()  => save$()))   // all in parallel
clicks$.pipe(concatMap(() => save$()))   // queued, in order
clicks$.pipe(exhaustMap(()=> save$()))   // ignore while busy
clicks$.pipe(switchMap(() => save$()))   // cancel the previous

Below, one impatient triple-click feeds all four operators at once — each inner task takes the same time, so the only variable is the flattening strategy. Watch four different truths emerge from identical input:

Four strategies, one input
3 rapid clicks → 3 inner tasks, four ways
mergeMapall concurrent
concatMapqueued, in order
exhaustMapignore while busy
switchMapcancel previous
fire the clicks and compare — same input, four outcomes
merge finishes all three (overlapping); concat finishes all three (one after another); exhaust finishes one and ignores two; switch finishes one and cancels two. The operator is the behavior.
↳ Mastering Angular
The intro course's typeahead used switchMap and moved on. Here you can see why it was switchMap and not the other three — and why a save button wants exhaustMap, an audit log wants concatMap, and parallel prefetch wants mergeMap.
mergeMap is the general-purpose one — default to it.
mergeMap's concurrency is exactly what breaks ordered or single-flight work: overlapping saves, out-of-order writes, duplicate submits. Pick by intent — cancel (switch), ignore (exhaust), queue (concat), or parallelize (merge). "Default to merge" is how race conditions ship.
These are interchangeable; results are the same anyway.
The lab proves otherwise from identical input: same three clicks yield three results, three results, one result, one result. The operator determines how many inners run, in what order, and which get cancelled — not a detail, the whole behavior.
RxJSAdvancedModule 04

Multicasting & interop

One more RxJS truth that bites hard: most observables are cold. A cold observable re-runs its producer for every subscriber — subscribe an HttpClient call twice and you fire two HTTP requests. Templates make this easy to do by accident: two async pipes on the same stream are two subscriptions. The fix is multicasting — share() / shareReplay() — which runs the producer once and fans the result out to all subscribers, turning a cold stream hot. And the bridges to the signal world, toSignal() and toObservable(), are how you move between the two reactive systems deliberately.

interop.ts 
const links$ = this.http.get<Link[]>('/api/links').pipe(
  shareReplay({ bufferSize: 1, refCount: true })   // one request, many readers
);
const links = toSignal(links$, { initialValue: [] });   // stream → signal
const query$ = toObservable(this.querySignal);          // signal → stream (for operators)

Add subscribers to a cold stream and to a shared one, and watch the execution counters diverge — the cold source re-runs every time, the shared source runs exactly once:

Cold re-runs · hot is shared
one producer execution, or many?
cold source · http.get()
0 producer runs
shareReplay() · multicast
0 producer runs
each cold subscription is another HTTP request — each shared one just joins the first
Three cold subscribers → three executions. Three shared subscribers → one execution, replayed. Two async pipes on a cold stream is the bug this hides.
Subscribing twice just reads the same result twice.
For a cold observable it re-runs the producer twice — two HTTP requests, two side effects. Observables are recipes, not values; each subscription cooks the recipe again. shareReplay is what turns the recipe into a shared dish.
toSignal / toObservable are rarely needed.
They're the seams of a real app: router events, form valueChanges, and HTTP arrive as observables; your components increasingly think in signals. Convert at the boundary — toSignal to consume, toObservable when you need time-based operators — and each system does what it's best at.
DI internalsAdvancedModule 05

Dependency injection, deeply

The intro course showed that DI resolves a token; this one shows how the walk works. Injectors form a tree — an environment hierarchy (root → lazy routes) and an element hierarchy (component → parent component). When you inject(TOKEN), resolution starts at the requesting injector and walks up until someone provides it, throwing NullInjectorError if no one does. Four resolution modifiers bend that walk: @Self (look only here), @SkipSelf (start at the parent), @Optional (return null instead of throwing), and @Host (stop at the host element's injector).

resolution-modifiers.ts 
a = inject(Cfg);                          // walk up until found (or throw)
b = inject(Cfg, { self: true });            // only THIS injector
c = inject(Cfg, { skipSelf: true });        // start at the parent
d = inject(Cfg, { optional: true });        // null instead of NullInjectorError
e = inject(Cfg, { host: true });            // stop at the host element injector

Toggle which injectors provide the token, pick a modifier, and resolve from the deepest component — the highlighted node is the one the walk lands on (or a thrown error when nothing qualifies):

The resolution walk
who answers inject(Cfg)?
provides Cfg: modifier:
root · EnvironmentInjectorCfg = 'root'
Dashboard · element injectorCfg = 'dashboard'
LinkEditor · requesterCfg = 'editor'
pick providers and a modifier, then resolve
Default walk climbs LinkEditor → Dashboard → Root and takes the first provider. @SkipSelf ignores the requester; @Self forbids climbing; @Host stops at the host; @Optional turns a miss into null.
↳ ASP.NET Core · Design Patterns
The environment hierarchy is your ASP.NET service lifetimes again — root singletons, route-scoped providers. The modifiers add what a server container rarely needs: a positional walk (self / parent / host) because Angular's injectors mirror a live component tree, not a flat container.
There's one injector and DI just finds the service.
There's a tree of injectors, and "finds" means a directed walk up it. Where you provide a token decides which subtree sees which instance — the root singleton, a route's copy, or a component-scoped one. Placement is the design decision; the walk is the mechanism.
@Optional, @Self, @Host are exotic edge-case decorators.
They're how reusable directives stay safe: @Optional @Host to cooperate with a host component's control if present, @SkipSelf to find a parent instance while providing your own. Library authors reach for them constantly — they bend the walk to the component's real shape.
Change detectionAdvancedModule 06

The change-detection tick

The intro course counted which components get checked. Now the mechanism. Angular holds every component instance as an LView — a flat array of its bindings — linked into a tree that mirrors the component tree. A change-detection tick walks that tree from the root and refreshes each view. Under OnPush and zoneless, a view is refreshed only if it's dirty, and here's the subtle part: when something marks a leaf dirty (a signal it reads changed, or markForCheck() fires), the dirty flag propagates up the ancestor chain — because the tick reaches a node only by walking through its parents. Mark the leaf, flag the path.

The other half is the scheduler. Ten signals changing in one turn don't schedule ten ticks — they coalesce into a single tick on the next microtask. Between the up-propagation of dirtiness and the coalescing of ticks, Angular turns "something changed somewhere" into "walk once, refresh only the flagged path."

the two motions 
signalInLeaf.set(next);   // 1. mark LeafA dirty → flag DashA → flag App (upward)
                       // 2. scheduler coalesces → ONE tick on the next microtask
// tick(): walk from root, refresh only nodes whose dirty flag is set, clear flags

Fire a change in a leaf and watch the dirty flag climb its ancestors — nothing refreshes yet. Then schedule the tick and watch a single downward pass refresh exactly the flagged path. Fire both leaves before ticking to see coalescing — two changes, one tick:

Mark upward · refresh downward
the LView tree · dirty flags · one coalesced tick
Apprefreshed ×0
DashArefreshed ×0
LeafArefreshed ×0
DashBrefreshed ×0
LeafBrefreshed ×0
ticks: 0 · refreshed last tick: 0 mark a leaf, then schedule a tick
Marking flags the path but refreshes nothing — the tick does the refreshing. Fire both leaves, then tick once: two dirty paths, a single coalesced pass. That's the zoneless scheduler.
↳ Data Structures · Big O
The LView tree is the tree from Data Structures holding your running UI, and the tick is a traversal of it. Up-propagating dirtiness plus refreshing only flagged nodes is what keeps a tick near O(dirty path) instead of O(all components) — the Big O reason OnPush scales.
markForCheck() re-renders the component immediately.
It sets a flag and schedules — nothing renders synchronously. The name is literal: it marks the view (and its ancestors) as needing a check on the next tick. Confusing "mark" with "render" is why people call it in loops and wonder why nothing changed until later.
Every state change triggers its own change-detection pass.
The scheduler coalesces changes within a turn into one tick — the lab's "both leaves, one tick" is exactly this. That's why a handler can update ten signals cheaply, and why zoneless Angular stays fast without Zone.js patching every async API.
View APIAdvancedModule 07

Directives & the view API

Structural directives look like magic (*ngIf, @for) but rest on a small, explicit API you can wield yourself. The asterisk is pure sugar: *x="e" desugars to an <ng-template> wrapping the element, with the directive attached. The directive receives two tools by injection — a TemplateRef (the blueprint of the wrapped content) and a ViewContainerRef (an insertion point in the DOM) — and it renders by calling createEmbeddedView, clear, move, or remove. That's all @if and @for do under the compiler.

repeat.directive.ts 
@Directive({ selector: '[repeat]' })
export class Repeat {
  private tpl = inject(TemplateRef);          // the blueprint
  private vcr = inject(ViewContainerRef);     // where views go
  count = input.required<number>({ alias: 'repeat' });
  constructor() { effect(() => {
    this.vcr.clear();
    for (let i = 0; i < this.count(); i++)
      this.vcr.createEmbeddedView(this.tpl, { $implicit: i });   // render
  }); }
}

The desugaring on the left is what the compiler sees. On the right, drive a ViewContainerRef directly — create, insert, move, and remove embedded views, and watch the live view list and its indices track every call:

The view container, by hand
*sugar desugared · createEmbeddedView & friends
what you write*repeat="3" renders a row
what the compiler sees<ng-template repeat="3"> <row/> </ng-template>
the ViewContainerRef is empty — createEmbeddedView() to fill it
vcr.length = 0
Every button is one real ViewContainerRef method. The indices are the container's — insert, move, and remove renumber exactly as the DOM does.
↳ OOP & Coding Principles
Attribute directives are the Decorator pattern made literal — add behavior to an element without subclassing it. And hostDirectives (composing directives onto a component) is composition-over-inheritance, the OOP principle, promoted to a first-class Angular feature.
*ngIf and @if are built-in specials the compiler hardcodes.
They're ordinary uses of TemplateRef + ViewContainerRef — the same API your own *repeat gets. The asterisk is a desugaring rule, not a privileged keyword. Knowing that, you can build any structural directive the framework didn't ship.
To render a component dynamically you need a complex factory.
It's one call: vcr.createComponent(Cmp), or ngComponentOutlet in a template. The old ComponentFactoryResolver ceremony is gone — the view container creates components and embedded views through the same small interface you just drove by hand.
FormsExpertModule 08

Custom form controls

Reactive forms bind to native inputs effortlessly — but the moment you build a custom control (a rating widget, a segmented picker, a rich editor) you need it to speak the forms API. That contract is ControlValueAccessor: implement four methods and your component becomes indistinguishable from an <input> to formControl. writeValue pushes a model value into your view (programmatic changes); registerOnChange hands you a callback to fire when the user changes the value (view → model); registerOnTouched reports blur; setDisabledState reflects disable(). Two directions, cleanly separated — the same data-down / events-up split, at the forms boundary.

status-picker.component.ts 
@Component({ providers: [{ provide: NG_VALUE_ACCESSOR,
  useExisting: StatusPicker, multi: true }], … })   // register as a control
export class StatusPicker implements ControlValueAccessor {
  value = signal<string | null>(null);
  private onChange = (v: string) => {};
  writeValue(v: string)      { this.value.set(v); }        // model → view
  registerOnChange(fn: any) { this.onChange = fn; }         // view → model hook
  pick(v: string)            { this.value.set(v); this.onChange(v); }  // user picked
}

The segmented control on the left is a CVA. Click it (user edit) and watch the form's value, dirty, and touched flags update through onChange; press patch from code to see writeValue update the control without marking it dirty; toggle disabled to see setDisabledState:

A control that speaks forms

the FormControl · status

validINVALID
dirtyfalse
touchedfalse
User clicks flow up through onChange and mark the control dirty + touched. Patch-from-code flows down through writeValue and marks nothing — exactly the native-input contract.
A custom control can just take a FormControl as an @Input.
That leaks the control everywhere and skips validation, dirty/touched, and disabled wiring. ControlValueAccessor is the contract that makes your widget a first-class citizen of formControlName — driven, validated, and status-tracked like any native input.
writeValue and onChange are two names for the same update.
They're opposite directions, and conflating them causes loops. writeValue is model→view (programmatic, marks nothing); onChange is view→model (user intent, marks dirty). The lab's "patch from code" vs "click" is precisely this asymmetry — keep them separate and forms behave.
CompilerExpertModule 09

Compilation & @defer

Angular is a compiled framework. Ahead-of-time, the compiler turns every template into a pair of functions — a creation block that builds the DOM once and an update block that patches bindings on each tick — emitted as static instructions on the component class. There's no template interpreter shipped to the browser; your {{ }} and @if become straight-line calls like ɵɵtext, ɵɵproperty, and ɵɵconditional. That's why Angular can tree-shake unused features and why bindings are fast: the "framework" at runtime is mostly your compiled instructions plus a small renderer.

The most visible payoff is @defer. A deferred block is compiled into a tiny state machine with its own bundle: it shows @placeholder until a trigger fires (on idle, on viewport, on interaction, on timer…), streams the lazy chunk while showing @loading, then swaps in the real content — with optional prefetch to fetch the chunk early without rendering it:

dashboard.html 
@defer (on viewport; prefetch on idle) {
  <analytics-chart [data]="stats()"/>     <!-- its own lazy bundle -->
} @placeholder { <chart-skeleton/> }   <!-- shown until the trigger -->
  @loading (minimum 200ms) { <spinner/> }  <!-- shown while streaming -->
  @error { <p>chart failed to load</p> }

Pick a trigger, optionally prefetch, then fire it and watch the block move through placeholder → loading → loaded — and notice how prefetch removes the wait:

@defer, in motion
placeholder → trigger → loading → loaded
trigger:
@placeholder
<chart-skeleton/> — the real chart hasn't been downloaded yet
The deferred chart is a separate bundle that doesn't touch the first load. Prefetch fetches it early so that when the trigger fires, there's nothing left to wait for.
↳ Front-End Engineering · SDLC
This is the front-end course's build step — bundling, code-splitting, lazy loading — pulled into the template as one keyword. What used to be webpack configuration and route-level splitting is now @defer, compiled per block, and it ships through the same CI pipeline the Lifecycle course built.
The browser downloads a template engine and interprets my HTML.
There's no template interpreter at runtime. AOT compiles each template to static instruction calls baked onto the component; the browser runs those directly. That's the source of the tree-shaking, the binding speed, and the compile-time errors in your templates.
@defer is just lazy loading with nicer syntax.
It's a compiled state machine per block: placeholder, loading (with a minimum to prevent flicker), error, and a menu of triggers plus prefetch — all declarative, all in the template, all tree-shaken into their own chunk. Route-level lazy loading can't express "load when this scrolls into view."
StateExpertModule 10

A signal store from scratch

You now have every piece to build the thing teams reach for a library to get: a reactive store. The recipe is small. Hold state in one private signal. Expose computed selectors that derive read-only views of it. Offer methods that update it immutably through a patchState helper. That's the whole pattern behind @ngrx/signals and friends — and because selectors are computed, each one recomputes only when the slice it reads actually changes, which you'll see counted live.

snip-store.ts 
function snipStore() {
  const state = signal({ links: [...], filter: 'all' });
  const patch = (p) => state.update(s => ({ ...s, ...p }));   // immutable write
  return {
    visible: computed(() => state().links.filter(byFilter(state().filter))),
    total:   computed(() => state().links.reduce(sumClicks, 0)),   // ignores filter
    count:   computed(() => this.visible().length),
    addLink: (l) => patch({ links: [...state().links, l] }),
    setFilter: (f) => patch({ filter: f }),
  };
}

The store drives three views. Add a link and all three selectors recompute; switch the filter and watch total skip its recompute — it never reads filter, so the graph knows it can't have changed:

three subscribing views

setFilter recomputes visible and count — but total ignores filter, so its counter holds and StatsView never re-renders. Selectors recompute only when their own inputs move.
↳ Design Patterns · Mastering Angular
This is the intro course's one-way store, rebuilt from primitives: the Observer pattern (selectors subscribe to state) with immutable patchState as the single writer. No Redux, no library — just a signal, some computeds, and methods. You could publish it in twenty lines.
Real state management needs Redux or NgRx.
For most apps a signal store — private state signal, computed selectors, patchState methods — is the whole thing, and it's what the modern libraries are now built on. Reach for a bigger tool when you need its specific features (devtools, effects orchestration), not by default.
Every view re-renders whenever any store state changes.
Views re-render only when a selector they read produces a new value — the lab's total holding steady through a filter change is the proof. Fine-grained selectors mean a store update touches exactly the views whose data actually moved, and no others.

The expert dashboard

The intro course assembled Snip's dashboard from the framework's surface. Here it is assembled from its depths — every technique in this course, in one zoneless component:

dashboard.component.ts 
bootstrapApplication(App, { providers: [provideZonelessChangeDetection()] });  // m06
