The screen is a function.
UI = f(state).

Everything the user ever touches is the front end — and under every button, feed, and dashboard sits one loop: state renders the view, events update the state, and the view renders again. This course teaches that loop and the architecture around it: how the browser turns markup into pixels, the living DOM, events and how they travel, reactivity, components, reconciliation, one-way data flow, server state, routing, and performance. The running project is the dashboard UI for Snip — and there's a twist: this course is written in its own subject. Every lab below is front-end code, reacting in your browser right now.

BeginnerModule 00

The front end's job

Every course so far has lived on the server side of an HTTP line: the API, the database, the tests, the pipeline. The front end is everything on the other side — the code that runs in the user's browser, on hardware you don't control, over a network you can't trust, in front of a human who will click things in an order you never imagined. Its job: turn data into an interface, and interaction back into data.

The browser is a runtime with three languages sharing the work: HTML declares structure, CSS declares appearance, and JavaScript provides behavior. Our project is the missing piece of Snip — a dashboard UI over the API you built in ASP.NET and the analytics query you wrote in Advanced SQL:

index.html 
<!DOCTYPE html>
<html>
  <head><link rel="stylesheet" href="app.css"></head>
  <body>
    <main id="dashboard"></main>   <!-- the UI grows here -->
    <script src="app.js"></script>  <!-- fetches GET /stats, renders -->
  </body>
</html>

One warning before we start: the front end looks simple from a distance — "just make the button do the thing" — and that's exactly why so much of it ends up as tangled, untestable glue. The whole discipline of this course is a single idea applied relentlessly: the screen is a function of state. Hold onto that; every module sharpens it.

↳ ASP.NET Core · The Capstone
The front end is the other end of every request from Mastering ASP.NET Core: your dashboard calls GET /stats, Snip's API answers with JSON, and everything in this course happens between that response and the user's eyes.
Front end is the easy part — it's just making things look nice.
It's a distributed-systems problem in disguise: untrusted network, wildly varied devices, a concurrent human generating events, and state that must stay consistent across all of it. Styling is the visible 10%; the other 90% is exactly what this course covers.
HTML and CSS aren't real programming.
They're declarative languages the browser executes with sophisticated engines — a constraint solver for layout, a cascade resolver for style. Declaring what and letting an engine compute how is the same idea that made SQL powerful — and it's the shape reactivity will take in Module 04.
RenderingBeginnerModule 01

From markup to pixels

Before anything reacts, something must render. The browser runs a pipeline: it parses HTML into the DOM (a tree of elements) and CSS into the CSSOM (the resolved style rules), combines them into a render tree (only what's visible — display: none never makes the cut), then computes layout (every box gets a geometry — this is where the box model, flexbox, and grid do their work), paints pixels into layers, and composites the layers on the GPU.

Walk Snip's dashboard header through the pipeline, one stage at a time:

The rendering pipeline
parse → style → layout → paint → composite
Every visual change re-enters this pipeline somewhere. Changing geometry re-runs layout; changing only color skips straight to paint; transform and opacity can ride the compositor alone.

This is also why responsive design is a pipeline concept, not a styling trick: resize the viewport (or rotate a phone) and layout simply re-runs with new constraints — media queries and flexible units are instructions to that stage. And it's why performance advice keeps saying "animate transform, not top": one rides the cheap compositor, the other drags layout and paint back into every frame.

The browser repaints the whole page on every change.
Engines are aggressively lazy: they invalidate only what changed and re-run only the necessary stages on the affected region. Your job is to avoid forcing the expensive stages — geometry changes cost layout; color changes cost only paint.
CSS is decoration applied after the real work.
CSS is the layout engine's input — a declarative geometry program. Flexbox and grid are constraint systems computing every box's position each frame. Treating that as an afterthought is how interfaces end up fighting their own foundation.
The DOMBeginnerModule 02

The DOM is a tree

The DOM isn't your HTML file — it's the live tree the browser built from it, and JavaScript's window into the page. Every element is a node with a parent, children, attributes, and text; the API lets you find nodes (querySelector), make them (createElement), move them (append), and change them — and the pipeline from Module 01 reacts to every mutation.

dom.js 
const list = document.querySelector('#links');   // find
const li = document.createElement('li');          // make
li.textContent = 'snip.ly/promo';
list.append(li);                                    // attach → pipeline runs
card.classList.toggle('dark');                     // restyle → style/paint

Run those exact operations against a mini dashboard card — the tree on the left is the truth, the render on the right is its shadow:

the rendered page (shadow)
Every button is one real DOM API call. The page never changes on its own — something always mutated the tree first.
↳ Data Structures
This is the tree from Mastering Data Structures, running your interface. Traversal, parent/child pointers, subtree insertion — querySelector is a tree search and append is a subtree graft. The whole front end stands on one data structure.
The DOM is my HTML source file.
The DOM is the parsed, live object model — scripts mutate it constantly, so it drifts from the source the moment JavaScript runs. "View source" shows what arrived; the DevTools inspector shows what is. Front-end debugging lives in the second one.
Touching the DOM is slow — avoid it entirely.
Single mutations are fast. The real costs are churn (hundreds of scattered writes, each triggering pipeline work) and layout thrashing (alternating reads and writes, forcing synchronous re-layout). The cure is batching writes — which is precisely the service frameworks sell in Module 06.
EventsIntermediateModule 03

Events & propagation

Nothing reacts until something happens. Events are the browser's report of the world — clicks, keys, scrolls, submits — and addEventListener is how code subscribes. The subtle part is that an event doesn't fire on one element: it travels. First the capture phase walks down from the document to the target, then the bubble phase walks back up — and any ancestor can listen at either leg.

Bubbling powers the front end's favorite trick, delegation — one listener on the list handles clicks from every row, including rows that don't exist yet:

events.js 
list.addEventListener('click', (e) => {
  const row = e.target.closest('[data-code]');   // which row bubbled?
  if (row) select(row.dataset.code);              // one listener, every row
});

Click the button below and read the journey — then cut the return trip with stopPropagation():

The event's journey
capture ↓ · target · bubble ↑
document · listens ↓↑
section#links · listens ↓↑
▸ snip.ly/4C92 — click me
Five stops per click: down through the ancestors, the target itself, then back up. Delegation is just listening on the way up.
An event fires only on the element you clicked.
It visits every ancestor twice — capture down, bubble up. That's not trivia: it's why a listener on document can see every click on the page, why modals close on outside-click, and why delegation scales to lists of ten thousand rows.
Attach a listener to each item in the list.
A thousand rows don't need a thousand listeners — they need one, on the container, reading e.target. Less memory, and rows added later are handled automatically because bubbling doesn't care when the row was born.
ReactivityIntermediateModule 04

UI = f(state)

Here is the module the whole course orbits. The naive way to build UI is imperative: an event happens, so you reach into the DOM and adjust things — update this list, also that counter, oh and the empty-state message. Every feature multiplies the places you must remember to touch, and the day you forget one, the screen lies: it shows something the data no longer says.

The reactive way inverts it. Keep all facts in one state object. Events don't touch the DOM at all — they update state, then one render function redraws the view from state. The view can't drift, because it's never edited — only recomputed:

reactive.js 
let state = { links: [...], draft: '' };

function render(state) {              // the whole view, from state alone
  view.innerHTML = state.links.map(LinkRow).join('');
  count.textContent = state.links.length;
}
function dispatch(action) {           // events change state, never the DOM
  state = update(state, action);
  render(state);                      // UI = f(state)
}

Below, Snip's link manager runs this loop for real. Watch the pattern — every interaction logs as event → new state → render. Then switch to manual mode, where handlers patch the DOM directly, and watch the view quietly fall out of sync with the truth:

In declarative mode the view is recomputed from state after every event — it cannot disagree with the data. That guarantee is what every modern framework is selling.
Reactivity is a framework feature you install.
It's a discipline — twelve lines of vanilla JS above, no library. React, Vue, and friends add efficiency (Module 06) and ergonomics, but the contract is the one you just watched: events → state → render. Learn the loop and every framework becomes syntax.
Updating the DOM directly is simpler for small apps.
It's simpler for the first feature. By the fifth, every event must remember every DOM spot that displays the data — the combinatorial trap the manual mode just demonstrated. One render function is the only version of the UI you ever have to get right.
ComponentsIntermediateModule 05

Components

One render function for a whole app would be a monster — so we split the UI into components: small render functions, each owning one piece of screen, composed into a tree that mirrors the interface. A component receives props (data flowing down from its parent) and reports back by emitting events (flowing up). Data down, events up — the contract that keeps a hundred components comprehensible.

components.pseudo.jsx 
// framework-flavored pseudocode — the shape, not the syntax
function LinkRow({ link, selected, onSelect }) {
  return <li class={selected ? 'sel' : ''}
             onClick={() => onSelect(link.code)}>   // event ↑
           {link.code} · {link.clicks}
         </li>;
}
Dashboard → Header(selected)  LinkList(links) → LinkRow × n   // props ↓

Click a row in Snip's component tree and trace the round trip — the event climbs to the owner of the state, the new prop flows back down:

Data down, events up
click a row · watch the round trip
The row doesn't change its own color — it reports upward, the owner updates state, and the highlight arrives back as a prop. Even selection obeys UI = f(state).
↳ OOP & Coding Principles
Components are encapsulation and composition from OOP, wearing a UI costume: each one hides its internals behind a small interface (props in, events out) and complex screens are composed from simple parts rather than inherited from god-widgets.
Components are about reusing visual widgets.
Reuse is the bonus; the real product is isolation. A component you can reason about alone — these props in, this markup out — is a component you can test, refactor, and delete. Most components are used once and are still worth it.
A child should just update the parent's data directly.
Then the parent's state has many writers and no owner, and debugging becomes archaeology. Emitting an event keeps one writer per piece of state — the whole reason the tree stays comprehensible at 3 a.m.
ReconciliationIntermediateModule 06

Reconciliation & keys

Module 04 left a question hanging: if the view is recomputed after every event, isn't rebuilding the DOM each time wasteful? It would be — so frameworks don't. Your render function produces a cheap description of the UI (a virtual tree); the framework diffs it against the previous description and applies only the minimal DOM operations. You write "the whole view, from state"; the browser receives "change this one text node."

Diffing lists has a famous trap. Compared by position, moving one item makes every row look changed. A key — a stable identity like link.code — lets the differ recognize the same item at a new position and just move it:

keys.pseudo.jsx 
{state.links.map(l =>
  <LinkRow key={l.code} link={l} />   // identity, not position
)}
The differ at work
old tree vs new tree → the patch
previous render
next render
computed DOM patch
Without keys, one move reads as "every row changed." With keys, it's one MOVE — and the rows keep their real DOM nodes, focus, and animation state.
↳ Big O
Tree diffing in general is expensive; framework reconciliation is the Big O mindset applied — compare level-by-level with keyed identity and it's O(n) per render instead of combinatorial. That trade is what makes "re-render everything" affordable.
The virtual DOM is fast because it's faster than the DOM.
A description in memory does less work, not faster work — the win is skipping real DOM operations that would trigger the Module 01 pipeline. The virtual tree is a planning step; the savings are the layouts and paints that never happen.
key={index} satisfies the framework, so it's fine.
An index is the position — exactly the identity that breaks when lists reorder. Rows swap contents while keeping their nodes: inputs keep the wrong text, animations play on the wrong item. Keys must survive reordering: use the domain identity, like link.code.
ArchitectureIntermediateModule 07

One-way data flow

Components solved the small scale. The app scale has a different problem: some state is shared — the link list matters to the header, the list, and the stats bar at once. First answer: lift state up to the closest common ancestor and pass it down. When lifting means threading props through five layers, the next answer is a store: one owner of shared state, outside the tree, that components subscribe to.

What keeps a store sane is the rule around it — one-way flow: views dispatch actions, a reducer computes the next state (immutably: a new object, not an edit), the store notifies subscribers, views re-render. Every change has one entrance and an audit trail:

store.js 
function reduce(state, action) {
  switch (action.type) {
    case 'ADD_LINK':   return { ...state, links: [...state.links, action.link] };
    case 'SET_FILTER': return { ...state, filter: action.filter };
    default:           return state;      // unknown actions change nothing
  }
}

one writer, many readers

⇄
the store (one writer)
Both components update from the same dispatch — neither talked to the other, and neither touched the store directly. The reducer is the only writer in the whole app.
↳ Design Patterns
A store is the Observer pattern from Design Patterns at architectural scale — subject (store), subscribers (components), notifications (re-renders) — with the Command pattern's shape in every action object. Old patterns, new stage.
Put all state in a global store — that's the "proper" architecture.
Global-by-default turns every keystroke into ceremony. The working rule: state lives as close as possible to where it's used, and is lifted only when genuinely shared. An input's draft belongs to the input; the link list belongs to the store.
Immutability is wasteful — just mutate the object.
A new object per change is what makes change detectable (compare references, not deep values), time-travel debuggable, and race-free. The differ from Module 06 leans on it too. Modern engines make the copies cheap; the guarantees are the point.
Server stateAdvancedModule 08

Server state

Everything so far lived in memory. Real data lives across a network — Snip's links come from GET /stats, and the network is slow, flaky, and asynchronous. The front-end move is to treat all of that as state like any other: a request isn't a moment, it's a lifecycle — loading → success | error — and the view renders each phase honestly. No blank screens, no silent failures.

api.js 
async function loadLinks() {
  dispatch({ type: 'LOAD_START' });                 // → spinner renders
  try {
    const res = await fetch('/api/stats');
    if (!res.ok) throw new Error(res.status);
    dispatch({ type: 'LOAD_OK', links: await res.json() });
  } catch (err) {
    dispatch({ type: 'LOAD_FAIL', err });            // → retry UI renders
  }
}

Load Snip's links below — then break the server and load again. Finally, try an optimistic add: the UI trusts the request, shows the row instantly, and rolls back if the server says no:

nothing loaded yet — the view says so, honestly
Loading, success, and error are all states with renders of their own. The network is unreliable; the UI's account of it never is.
↳ ASP.NET Core · Advanced SQL
The endpoint on the other end is the one you built: ASP.NET's GET /stats running the Advanced SQL dashboard query — CTE, FILTER pivot, window rank — and returning JSON. This module is that JSON's last mile to a human.
Await the fetch, then render — that's the whole story.
That story has no chapter for the 800 ms in between or the request that never lands. Users live in those chapters. Loading and error are first-class states; render them or the interface goes silent exactly when the user needs it to speak.
Optimistic updates are lying to the user.
They're a bet with a receipt: show the expected result instantly, keep enough information to roll back, and reconcile with the server's answer. For high-success actions it's the difference between an app that feels instant and one that feels dial-up.
RoutingAdvancedModule 09

Routing & the app's shape

Zoom out to the whole app. The classic web is a multi-page app: every navigation asks the server for a fresh document. A single-page app loads once and navigates in JavaScript — instant transitions, state that survives navigation — while server-side rendering hybrids send real HTML first and hydrate it into an app for fast first paint and rich interaction. Different shapes, one shared insight: the URL is state. It should drive the view like any other state — and update like any other state, so back, refresh, and share-this-link keep working.

router.js 
function navigate(path) {
  history.pushState({}, '', path);   // change the URL — no reload
  render(routeFor(path));             // URL is just more state
}
window.addEventListener('popstate',   // back button = state change too
  () => render(routeFor(location.pathname)));
A router in miniature
URL as state · no reloads
Each click is pushState plus a render — the document never reloads, yet the URL, the view, and the back button all agree, because they all read the same state.

The other whole-app concern is the build: modern front ends are compiled — bundled into few files, minified, tree-shaken of dead code, and code-split so the settings page's JavaScript doesn't tax the dashboard's first load. The artifact you deploy is static files on a CDN; the app-ness happens in the browser.

↳ The Capstone
Snip now has two routers: the server one from the capstone answering GET /{code} with a 301, and this client one swapping dashboard views without a reload. Same concept — a path chooses what runs — on opposite ends of the wire.
SPAs are simply better — make everything one.
SPAs buy fluid interaction with a real bill: bigger first load, client-side complexity, SEO and accessibility work the browser used to do free. Content sites often want MPA or SSR; dashboards like Snip's earn the SPA. Shape follows use, not fashion.
The back button breaking is just how web apps are.
It breaks only when the URL stops being state. Route with pushState, render from the location, handle popstate — and back, refresh, and shared links all work, because they're all just reads of the same value.
PerformanceAdvancedModule 10

Performance & the main thread

Front-end performance is two promises: load fast (small bundles, lazy loading, caching — mostly Module 09's build doing its job) and stay responsive. The second one has a hard constraint: JavaScript, style, layout, and paint largely share one main thread, drained by the event loop one task at a time. At 60 fps a frame gets ~16 ms; any task longer than that and scrolling stutters, clicks queue, the page janks. The user can feel your for loop.

The classic offender is an event that fires far faster than its work deserves — input, scroll, resize. The fixes are rate-limiters: debounce (act after the burst goes quiet — right for search-as-you-type) and throttle (act at most every N ms — right for scroll effects):

debounce.js 
function debounce(fn, ms) {
  let t;
  return (...args) => {
    clearTimeout(t);                    // the burst resets the clock
    t = setTimeout(() => fn(...args), ms);
  };
}
search.addEventListener('input', debounce(runSearch, 250));

Type a Snip search below — every keystroke is an event, but watch how few of them become work:

Fifteen keystrokes, one search. The event loop only has 16 ms per frame — spend them on work the user can see.
JavaScript is async, so it's basically multithreaded.
Async means interleaved on one thread, not parallel: awaiting a fetch frees the loop, but your handler still runs alone and unpreemptable. A 200 ms computation freezes every animation on screen — that's what Web Workers exist to take elsewhere.
Optimize when users complain — premature optimization, etc.
Load weight and main-thread budgets aren't micro-optimizations; they're architecture, nearly unfixable after the fact. Wisdom is measuring before optimizing details — but budgets (bundle size, frame time) belong in the design from day one.
ExpertModule 11

Accessibility & the dashboard

One promise remains: the interface must work for everyone — keyboard users, screen-reader users, low vision, motor constraints, the person on a train with one bar of signal. Accessibility is mostly won by using the platform honestly: a real <button> is focusable, keyboard-operable, and announced correctly for free; a clickable <div> is none of those until you rebuild what you threw away. Semantic HTML first, visible focus, sufficient contrast, labels on inputs, and ARIA only to fill genuine gaps. Respect prefers-reduced-motion — this very page does.

And now the assembly. Everything this course taught, in one component — Snip's dashboard:

Dashboard.pseudo.jsx 
function Dashboard() {
  const { links, status, filter } = useStore();      // m07 · one-way flow
  useEffect(loadLinks, []);                           // m08 · server state
  const route = useRoute();                           // m09 · URL is state

  if (status === 'loading') return <Spinner />;      // m08 · honest phases
  if (status === 'error')   return <Retry />;

  return (                                            // m04 · UI = f(state)
    <main>                                            // m11 · semantic, focusable
      <StatsBar links={links} />                     // m05 · props down
      {links.map(l => <LinkRow key={l.code}        // m06 · keyed identity
                        onSelect={select} />)}     // m03/m05 · events up
    </main>
  );
}
The loop is the app

Strip the acronyms away and the front end is one loop wearing many coats: the browser renders state into pixels (Modules 01–02), the user fires events (03), events become new state (04), components split the render (05), the differ keeps it cheap (06), the store scales it (07), the server feeds it (08), the URL joins it (09), the budget disciplines it (10), and accessibility makes it everyone's (11). UI = f(state) — learn the formula once and every framework is just an accent.
