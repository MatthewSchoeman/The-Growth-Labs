Complexity is just
the shape of growth.

Two programs can return the same answer, yet one finishes before you blink and the other never finishes at all. The difference is how the work grows as the input grows. Master that, and you can predict performance from code you've never run. This is that mastery, built one idea at a time.

BeginnerModule 00

Why complexity matters

Imagine two functions that both check whether a list contains any duplicate values. The first compares every pair of elements. The second sorts the list, then walks through it once looking for neighbours that are equal. On a list of 10 items, you'd never notice a difference. On a list of 100,000 items, the first takes minutes and the second takes a fraction of a second.

Same input, same output, same language, same machine. The only thing that changed is how the number of steps scales with the size of the input. That scaling behaviour is what we call complexity, and it is the single most reliable predictor of whether a program will hold up when the data gets big.

Why not just time it with a stopwatch?

Measuring wall-clock time is useful, but it answers a narrower question than you think. A stopwatch tells you how this code performed, on this machine, with this input, today. It can't tell you what happens when the input is ten times larger, or on a server twice as fast, or next year. Those factors change the absolute time but not the underlying growth pattern.

Complexity analysis strips away the hardware, the language, and the specific input, and asks the one question that survives all of them: as the input grows, how does the amount of work grow? The answer is a property of the algorithm itself, which is why it transfers across machines and lasts across years.

The two resources we count

Time complexity — how the number of operations grows with input size. Space complexity — how the amount of memory grows with input size. They are analysed the same way and often trade off against each other: you can frequently spend memory to save time, or vice versa.

What "input size" means

We measure growth against a number we call n: the size of the input. Usually that's obvious — the length of an array, the number of records, the count of nodes in a graph. Sometimes there are two independent sizes (say a graph with V vertices and E edges), and then complexity is expressed in terms of both. The discipline is always the same: pick what grows, call it n, and describe everything relative to it.

That's the whole game in one sentence. The rest of this course is about doing it precisely, for any code you encounter, all the way up to the subtle cases that trip up experienced engineers.

BeginnerModule 01

Counting operations

Before any notation, build the intuition the notation is shorthand for. The core move is to count how many "basic steps" a piece of code performs as a function of n — then watch what that count does as n grows.

A first count

pythondef sum_list(nums):
    total = 0            # 1 step
    for x in nums:        # the loop runs n times
        total += x         # 1 step, repeated n times
    return total         # 1 step

Counting carefully: one step to initialise, n steps inside the loop, one step to return. Total work is n + 2 steps. As n grows, that +2 becomes irrelevant — at a million elements, the difference between 1,000,000 and 1,000,002 is noise. The work grows in direct proportion to n. That's the pattern we'll later name "linear".

What counts as a "step"?

A basic step is any operation that takes a fixed amount of time regardless of n: an arithmetic operation, a comparison, assigning a variable, accessing an array element by index, following a pointer. We deliberately treat them all as "roughly one unit" because we don't care about the exact constant — we care about how the total scales.

A trap worth flagging early

Not every line that looks like one step is one step. In many languages, x in some_list secretly scans the whole list (that's n steps hiding inside one line), while x in some_set is a constant-time lookup. Reading complexity correctly means knowing what your operations actually cost underneath — a theme we'll return to constantly.

When loops nest, counts multiply

pythondef all_pairs(nums):
    for i in nums:        # runs n times ...
        for j in nums:    # ... and for each, runs n times
            print(i, j)   # n × n = n² steps total

The inner work happens once for every combination of i and j, so the count is n × n = n². Double the input and the work quadruples. This is the difference that makes the duplicate-checker from Module 00 fall over: pairwise comparison is this shape.

You now have the only mental model you need: count the steps as a function of n, then ask what dominates as n gets large. Big O, next, is simply a precise vocabulary for "what dominates".

BeginnerModule 02

Big O notation

Big O is the language for describing how the step-count grows, with the irrelevant detail thrown away. Informally: Big O captures the dominant term of the growth and ignores constant factors. We'll make this formal in Module 04 — but the working intuition comes first, because that's what you'll actually use day to day.

The two simplification rules

To turn a raw step-count into a Big O expression, apply two rules until nothing changes:

  1. Drop constant factors. 5n, n/2, and 100n are all O(n). We're describing the shape of growth, not the exact slope. A constant factor shifts the line but doesn't change that it's a line.
  2. Keep only the fastest-growing term. In n² + n + 5, the n² term eventually dwarfs everything else, so the whole thing is O(n²). For large n, lower-order terms are rounding error.
Why we're allowed to throw so much away

Big O describes asymptotic behaviour — what happens as n heads toward infinity. The whole point is to compare how algorithms scale, and at scale the dominant term and only the dominant term decides the outcome. (This does not mean constants never matter in practice — they can. We'll treat that honestly in Module 09.)

The complexity classes you'll meet constantly

A handful of growth shapes cover the overwhelming majority of code. Learn these by feel — what each looks like, and a canonical example of each:

ClassNameFeelCanonical example
O(1)ConstantDoesn't grow at allArray index; hash lookup
O(log n)LogarithmicGrows painfully slowlyBinary search

The ordering from best to worst is worth memorising as a single chain:

The hierarchy
O(1) < O(log n) < O(n) < O(n log n) < O(n²) < O(n³) < O(2ⁿ) < O(n!)

Anything left of O(n log n) scales beautifully. O(n²) is fine for modest inputs but a liability at scale. Everything from O(2ⁿ) rightward is intractable beyond very small n and signals you need a fundamentally different approach.

The intuition to keep

Check yourself

An algorithm performs 3n² + 200n + 5000 operations. What is its Big O, and roughly at what input size does the n² term overtake the 200n term?

▶ Reveal answer

The Big O is O(n²) — drop the constant factor 3, and discard the lower-order 200n and 5000 terms.

The n² term overtakes 200n when 3n² > 200n, i.e. n > 200/3 ≈ 67. Below that, the linear term actually dominates the real work — a concrete reminder that Big O describes large-n behaviour, and that for small inputs the "lower-order" terms and constants can be the ones that matter.

IntermediateModule 03

Reading code for complexity

Now we turn intuition into a repeatable procedure. Given an arbitrary block of code, you should be able to read off its complexity with confidence. There are only a few combination rules, and everything else is applying them carefully.

Rule 1 — Sequential code adds

Two blocks that run one after the other have their costs added, and then the dominant term wins:

pythonfor x in nums:        # O(n)
    process(x)
for i in nums:        # O(n²)
    for j in nums:
        compare(i, j)
# total: O(n) + O(n²) = O(n²)

The first loop is real work, but it's swamped by the second. O(n) + O(n²) collapses to O(n²).

Rule 2 — Nested code multiplies

When one loop runs inside another, multiply their counts. A loop of n wrapping a loop of m is O(n·m). If both are n, that's O(n²); three deep is O(n³).

Rule 3 — Independent inputs stay separate

If two loops range over different inputs, keep both variables — don't collapse them:

pythondef merge_reports(users, orders):
    for u in users:      # a = len(users)
        index(u)
    for o in orders:     # b = len(orders)
        index(o)
    # O(a + b) — NOT O(n). They grow independently.
A common and costly mistake

Calling both inputs "n" when they're independent hides real behaviour. If you process every user against every order, that's O(a·b) — and if one is small and one is huge, knowing which drives the product is the difference between a fast feature and an outage.

Rule 4 — A loop isn't always O(n): look at how the index moves

The number of iterations depends entirely on how the loop variable changes. When it grows multiplicatively, you get a logarithm:

pythoni = 1
while i < n:
    do_work()
    i *= 2          # 1, 2, 4, 8, ... doubles each time
# reaches n after log₂(n) steps  →  O(log n)

Why log? If you double from 1 and want to reach n, you need the number of doublings k such that 2ᵏ ≥ n, which is k ≥ log₂ n. Halving a search space each step (binary search) has the identical structure for the identical reason.

The base of the log is irrelevant in Big O

Whether you halve, third, or tenth the space each step, the result is O(log n) — different log bases differ only by a constant factor (log₂ n = log₁₀ n / log₁₀ 2), and constant factors are dropped. That's why nobody writes the base.

The subtle one — when nested loops are NOT n²

This is where careful counting pays off. Look closely at the inner bound:

pythonfor i in range(n):       # outer: n times
    for j in range(i):   # inner: depends on i, not n!
        do_work()

The inner loop runs 0 times, then 1, then 2, … up to n−1. The total is 0 + 1 + 2 + … + (n−1) = n(n−1)/2. That has an n² term, so it is O(n²) — but notice you only learned that by summing the series, not by assuming "two nested loops means n²". The habit of actually counting is what separates reliable analysis from guessing.

Check yourself

What is the time complexity of this function?

pythondef mystery(n):
    count = 0
    i = n
    while i > 1:
        for j in range(n):
            count += 1
        i = i // 2
    return count
▶ Reveal answer

O(n log n). The outer while halves i each pass, so it runs log n times. The inner for runs n times on every pass. Nested, so multiply: n × log n.

This is the structure of an O(n log n) algorithm in miniature — a linear amount of work repeated a logarithmic number of times. Recognising that shape will serve you well when we reach sorting.

IntermediateModule 04

The formal definitions

So far Big O has meant "the dominant term, constants dropped". That's the right working intuition, but the precise definition is what lets you reason rigorously and, crucially, understand why O, Ω, and Θ are three different things that people routinely confuse.

Big O — an upper bound

Definition · Big O

f(n) = O(g(n)) means there exist positive constants c and n₀ such that:

0 ≤ f(n) ≤ c · g(n) for all n ≥ n₀

In words: beyond some point n₀, f never exceeds g scaled by some constant. g is a ceiling on the growth rate.

The two escape hatches — "some constant c" and "beyond some point n₀" — are precisely what formalise our two rules. The constant c is why constant factors don't matter. The threshold n₀ is why lower-order terms don't matter: they only ever win for small n, and we're allowed to ignore small n.

Big Omega and Big Theta — the other two bounds

Big O alone only bounds from above. Two siblings complete the picture:

NotationBoundMeansLoose analogy
O(g)Uppergrows no faster than g≤
Ω(g)Lowergrows no slower than g≥
Θ(g)Tightgrows exactly like g (both bounds)=
o(g)Strict uppergrows strictly slower than g<
ω(g)Strict lowergrows strictly faster than g>

f(n) = Θ(g(n)) exactly when f is both O(g) and Ω(g). Theta is the strongest, most informative claim: it pins the growth rate from both sides. When you can prove Theta, say Theta — it tells the reader far more than Big O does.

The misuse everyone commits

In casual conversation people say "Big O" when they mean "Big Theta". Saying merge sort "is O(n log n)" is technically a weaker statement than the truth — it's also, trivially, O(n²) and O(2ⁿ), because Big O is only a ceiling. What people usually mean is Θ(n log n): that's its actual growth rate. The habit is so universal it's mostly harmless, but knowing the distinction is what makes you precise when precision matters.

The distinction that actually trips up engineers: cases vs bounds

Here is the single most important clarification in this module, and the one most people get subtly wrong. "Best / worst / average case" and "O / Ω / Θ" are completely independent ideas. They answer different questions:

  • Case selects which input scenario you're analysing — the luckiest input, the unluckiest, or the typical one. Each case is its own function of n.
  • O / Ω / Θ describe how you're bounding whichever function you've chosen.

So you can legitimately speak of the Big O of the best case, or the Big Theta of the worst case, and so on. They combine freely. Consider linear search for a target in an unsorted array:

CaseWhat happensTight bound
Besttarget is the first elementΘ(1)
Worsttarget is last or absent — scan allΘ(n)
Averagetarget equally likely anywhere → ~n/2 checksΘ(n)

When someone says linear search "is O(n)" with no qualifier, the convention is that they mean the worst case — because that's the guarantee that matters for safety. But notice nothing forced that: the worst case has a best-case bound too (it's also Ω(n)), and the best case is Θ(1). Keeping "which input" and "which bound" in separate mental boxes dissolves an enormous amount of confusion.

Check yourself

True or false: "Quicksort is O(n²)." And separately: "Quicksort is O(n log n)." Can both be defensible?

▶ Reveal answer

Both can be defended because they describe different cases. Quicksort's worst case is Θ(n²) (e.g. an already-sorted array with a naïve pivot), so "O(n²)" is a true statement about the worst case. Its average / expected case is Θ(n log n), so "O(n log n)" is a true statement about the typical case.

The lesson: an unqualified complexity claim is ambiguous until you state the case. Good engineers state it — "quicksort is O(n log n) on average but O(n²) in the worst case" — precisely because the bare number invites the confusion above.

IntermediateModule 05

Space complexity

Everything you've learned about counting operations applies identically to counting memory. Space complexity asks: as n grows, how does the amount of memory the algorithm needs grow? The same classes, the same rules, the same notation.

Auxiliary space vs total space

One distinction matters up front. Total space includes the input itself; auxiliary space is the extra memory the algorithm allocates beyond the input. Usually when people quote "space complexity" they mean auxiliary space, because the input is a fixed cost you're handed regardless of which algorithm you choose.

pythondef double_all(nums):
    result = []           # grows to size n
    for x in nums:
        result.append(x * 2)
    return result          # O(n) auxiliary space

def double_in_place(nums):
    for i in range(len(nums)):
        nums[i] *= 2       # reuses the input
    # O(1) auxiliary space — a few variables, regardless of n

Both are O(n) in time, but they differ in space: the first builds a new list of size n; the second mutates in place using only a counter. That gap is a textbook space–time tradeoff in reverse — here you save space at no time cost, but often you'll pay one resource to relieve the other.

The sneaky source of space: the call stack

Recursion costs memory even when you never explicitly allocate any. Every pending recursive call occupies a stack frame, and those frames pile up until the base case unwinds them. The depth of the recursion is your space cost.

pythondef sum_to(n):
    if n == 0: return 0
    return n + sum_to(n - 1)
# O(n) TIME and O(n) SPACE — n stacked frames before any return

This looks like it uses no memory, but it quietly holds n stack frames open simultaneously — which is why deep recursion can blow the stack on inputs an iterative version handles effortlessly. When analysing recursive code, always account for stack depth as space.

Contrast worth internalising

Merge sort is O(n log n) time and O(n) auxiliary space (it needs scratch arrays to merge). Heapsort is also O(n log n) time but O(1) auxiliary space (it sorts in place). Same time class, different space — and in a memory-constrained environment that difference decides which one you ship.

Check yourself

What is the auxiliary space complexity of a recursive function that computes Fibonacci numbers by branching into fib(n-1) + fib(n-2)? (Hint: think about the maximum depth of the call stack at any one moment, not the total number of calls.)

▶ Reveal answer

O(n) space, even though the time is O(2ⁿ). The key insight is that the call stack only holds the calls along one path from root to base case at any instant — the recursion explores the tree depth-first, so the deepest the stack ever gets is the height of the tree, which is n.

This is a beautiful example of time and space diverging: exponentially many calls happen over time, but they're never all alive at once. Counting peak simultaneous memory, not cumulative work, is the whole skill of space analysis.

AdvancedModule 06

Recursion & the Master Theorem

Recursive algorithms resist the loop-counting method, because the work is defined in terms of itself. The tool for them is a recurrence relation: an equation that expresses the cost on size n in terms of the cost on smaller sizes. Solving the recurrence gives the complexity.

Writing a recurrence

Take merge sort. On an input of size n it does three things: split into two halves, recursively sort each half, then merge the sorted halves in linear time. As a recurrence:

Merge sort's recurrence
T(n) = 2 · T(n/2) + Θ(n)

Two subproblems (2), each half the size (n/2), plus linear merge work (Θ(n)). Read every divide-and-conquer recurrence this way: how many subproblems, how much smaller, and how much work to combine.

The recursion-tree intuition

Before any theorem, you can solve recurrences by drawing the tree of calls and summing the work level by level. For merge sort, every level does a total of Θ(n) work (the subproblems shrink but there are proportionally more of them), and there are log₂ n levels until the pieces are size 1:

each level totals Θ(n) work · there are log₂ n levels · total = Θ(n log n)

Work per level (n) multiplied by number of levels (log n) gives Θ(n log n). The recursion tree is the most intuitive way to see why the good sorts land where they do, and it works for any recurrence you can draw.

The Master Theorem — a formula for the common shape

Most divide-and-conquer recurrences fit one mould, and the Master Theorem solves that mould directly, no tree-drawing required. It applies to:

Master Theorem form
T(n) = a · T(n/b) + f(n) with a ≥ 1, b > 1

Here a is the number of subproblems, b is the factor each subproblem shrinks by, and f(n) is the work done outside the recursion (dividing and combining). Everything hinges on comparing f(n) against the special quantity n^(log_b a) — the total cost of all the leaf-level work.

Compare f(n) with n^(log_b a) and one of three cases applies:

CaseConditionResultWho wins
1f(n) = O(n^(log_b a − ε))Θ(n^(log_b a))leaves dominate
2f(n) = Θ(n^(log_b a))Θ(n^(log_b a) · log n)balanced (tie)
3f(n) = Ω(n^(log_b a + ε))Θ(f(n))root work dominates

The idea behind the three cases is genuinely simple once named: either the bulk of the work is at the bottom of the tree (many tiny subproblems — case 1), spread evenly across all levels (case 2), or concentrated at the top in the divide/combine step (case 3). Case 3 carries a technical "regularity" side-condition (a·f(n/b) ≤ c·f(n) for some c < 1), which essentially every natural f satisfies.

Worked applications

AlgorithmRecurrencea, b, fCase → Result
Merge sort2T(n/2)+Θ(n)n^log₂2 = n; f=n2 → Θ(n log n)
Binary searchT(n/2)+Θ(1)n^log₂1 = 1; f=12 → Θ(log n)
Naïve traversal2T(n/2)+Θ(1)n^log₂2 = n; f=11 → Θ(n)
Karatsuba mult.3T(n/2)+Θ(n)n^log₂3 ≈ n^1.5851 → Θ(n^1.585)
Scope check

The Master Theorem only covers recurrences of this exact divide-by-a-constant-factor form with equal subproblem sizes. Recurrences like T(n) = T(n−1) + Θ(n) (which gives Θ(n²)) or unequal splits need the recursion-tree or substitution methods instead. Reach for the theorem when it fits, and fall back to drawing the tree when it doesn't.

Check yourself

Use the Master Theorem on T(n) = 4T(n/2) + Θ(n). (This is the recurrence for naïve matrix multiplication's structure.)

▶ Reveal answer

Here a = 4, b = 2, so the critical exponent is log₂ 4 = 2, giving n^(log_b a) = n². The combine work is f(n) = Θ(n).

Since n grows strictly slower than n² (specifically f(n) = O(n^(2−ε)) with ε = 1), we're in Case 1: the leaves dominate. The answer is Θ(n²). The recursion does so much branching that the work piles up at the bottom of the tree, and the linear combine step is negligible by comparison.

AdvancedModule 07

Amortized analysis

Some operations are usually cheap but occasionally expensive, and quoting the expensive case as their complexity badly overstates the real cost. Amortized analysis measures the average cost per operation across a worst-case sequence, capturing the truth that the rare expensive step is "paid for" by the many cheap ones around it.

Amortized is NOT average-case

This distinction is essential. Average-case averages over a probability distribution of inputs and can be unlucky. Amortized averages over a sequence of operations and is a worst-case guarantee — it makes no assumption about randomness. An amortized O(1) operation is genuinely cheap on average no matter what, not just on typical inputs.

The canonical example: a growing array

A dynamic array (Python's list, C++'s vector, Java's ArrayList) appends in O(1) — usually. But when it runs out of capacity, it must allocate a bigger block and copy every existing element, an O(n) operation. So how can we honestly call append "constant time"?

The trick is that implementations grow capacity multiplicatively — typically doubling. Because of that, the expensive copies happen exponentially rarely. Consider n appends starting from capacity 1:

  • Resizes happen at sizes 1, 2, 4, 8, …, up to n — copying that many elements each time.
  • Total copying across all resizes is 1 + 2 + 4 + … + n < 2n. (A doubling geometric series sums to less than twice its last term.)
  • So n appends cost under 2n + n = O(n) work in total — which is O(1) amortized per append.
The geometric series that makes it work
1 + 2 + 4 + 8 + ⋯ + n = 2n − 1 < 2n = O(n)

The total cost of every resize combined is only linear, so spread across n appends each one pays a constant share. Had the array grown by a fixed amount each time instead of doubling, resizes would be frequent and the same analysis would give O(n) amortized — disastrously worse. The multiplicative growth is the whole reason it works.

Three lenses on the same truth

There are three standard methods, and they always agree — they're different bookkeeping for the same fact:

  • Aggregate method. Bound the total cost of any sequence of n operations, then divide by n. (That's exactly what we did above.)
  • Accounting method. "Overcharge" cheap operations and store the surplus as credit, then spend that credit to pay for expensive ones. Charge each append 3 units: 1 to insert, 2 saved to later fund a future copy. The credit never goes negative, so the true average is at most 3 = O(1).
  • Potential method. Define a "potential" function on the data structure's state that rises during cheap operations and is released to pay for expensive ones. The most formal of the three, and the one used to analyse sophisticated structures like Fibonacci heaps and splay trees.
Where this shows up in real life

Amortized bounds are why everyday operations you treat as free actually are: appending to a list, inserting into a hash table (which periodically rehashes everything), and incrementing a binary counter are all amortized O(1) despite occasional expensive steps. When a data structure's docs promise "amortized constant time", this is the kind of reasoning behind it.

Check yourself

Suppose a dynamic array grew by adding a fixed 100 slots each time it filled, rather than doubling. What would the amortized cost of append become, and why is it so much worse?

▶ Reveal answer

It degrades to O(n) amortized. With fixed-increment growth, a resize happens every 100 appends, and the k-th resize copies about 100k elements. Over n appends there are n/100 resizes, and the copy costs form an arithmetic series: 100 + 200 + 300 + …, which sums to Θ(n²) total work. Divided across n appends, that's Θ(n) each.

The contrast is the entire point of amortized analysis: doubling produces a geometric series (total O(n)), fixed increments produce an arithmetic series (total O(n²)). The growth policy, invisible in any single append, determines the amortized cost.

AdvancedModule 08

Algorithms & data structures, by the numbers

With the analysis toolkit complete, the payoff is fluency with the complexities of the building blocks you use every day — and, more importantly, why they are what they are. Memorising the tables is useful; understanding them is mastery.

Common data structure operations

These are the bounds you reason with when choosing a structure. Note how often the right choice trades one operation's speed for another's:

Reading the table like an expert

Quicksort's worst case is Θ(n²), yet it's the default sort in many libraries — because its average case has small constant factors and excellent cache behaviour, and random pivots make the worst case astronomically unlikely. The complexity class is the start of the decision, not the end.

ExpertModule 09

Lower bounds, P vs NP & the gap with reality

Expert-level complexity is about three deeper questions: not just how fast a given algorithm is, but how fast any possible algorithm could be (lower bounds), which problems are fundamentally hard (complexity classes), and where the clean theory diverges from the messy machine (practical reality).

Lower bounds — proving you can't do better

Big O describes an algorithm. A lower bound describes a problem: the minimum work any correct algorithm must do, no matter how clever. The most famous result of this kind is that comparison-based sorting cannot beat Ω(n log n).

Why comparison sorting needs Ω(n log n)

Any sort that works only by comparing pairs of elements can be modelled as a decision tree: each internal node is a comparison, each leaf is one possible ordering of the input. To sort correctly, the tree must have a distinct leaf for every one of the n! possible permutations.

tree height ≥ log₂(n!) = Θ(n log n)

A binary tree with n! leaves must be at least log₂(n!) deep, and the height is the worst-case number of comparisons. Since log₂(n!) = Θ(n log n) (by Stirling's approximation), no comparison sort can do better. This is why merge sort and heapsort are, in a precise sense, optimal — and why counting sort can only beat the bound by abandoning comparisons entirely.

When an algorithm's upper bound matches the problem's lower bound, the algorithm is asymptotically optimal — you have provably hit the floor. Recognising that ceiling-meets-floor moment is a hallmark of mature analysis.

P, NP, and NP-completeness — done correctly

This is the most misunderstood topic in all of complexity, so precision matters. These classes concern decision problems (yes/no questions) and how they scale:

  • P — problems solvable in polynomial time (O(n^k) for some constant k). These are the tractable problems.
  • NP — problems whose proposed solutions can be verified in polynomial time. NP does not stand for "non-polynomial" — a near-universal error. It stands for nondeterministic polynomial: solvable in polynomial time by a hypothetical machine that can explore all options at once, which is equivalent to "a correct answer can be checked quickly".
  • NP-complete — the hardest problems in NP, with a remarkable property: if any single one had a polynomial-time algorithm, all of NP would, and P would equal NP. Examples include boolean satisfiability (SAT), the travelling salesman decision problem, and graph colouring.
  • NP-hard — at least as hard as the hardest NP problems, but not necessarily in NP themselves (they may not even be decision problems).
The million-dollar question

Every problem in P is in NP (if you can solve it quickly, you can certainly verify it quickly). Whether the reverse holds — does P = NP? — is the most important open problem in computer science and one of the Clay Millennium Prize Problems. Almost everyone believes P ≠ NP (that some problems are fundamentally harder to solve than to check), but no one has proved it. If it were false, modern cryptography, which relies on certain problems being hard to solve, would collapse.

Pseudo-polynomial time — a subtle trap

The dynamic-programming solution to the knapsack problem runs in O(n·W) time, where W is the capacity. That looks polynomial — but it's pseudo-polynomial, and the distinction is genuinely expert territory. Complexity is measured against the size of the input in bits, and a number W takes only log W bits to write down. So W is exponential in its own bit-length, making the algorithm exponential in the true input size. Knapsack is NP-complete, and this is exactly why O(n·W) doesn't contradict that.

The gap between Big O and the stopwatch

Asymptotic analysis is indispensable, but treating it as the whole story is itself a kind of mistake. Several realities live in the constants and lower-order terms that Big O discards on purpose:

  • Constants are real. An O(n) algorithm with a huge constant can lose to an O(n log n) one across the entire range of inputs you'll ever see. Big O tells you who wins eventually; "eventually" might be larger than any input you'll encounter.
  • Memory hierarchy matters enormously. Two O(n) algorithms can differ by 10× because one streams through memory cache-friendly and the other jumps around. This is why a contiguous array often crushes a linked list in practice despite identical Big O — the constant hidden in "one memory access" varies by orders of magnitude depending on caching.
  • The crossover point. A "worse" complexity can be the right choice below some input threshold — exactly why production sorts use insertion sort for small subarrays. Mastery means knowing where the asymptotically better algorithm actually overtakes the simpler one.
Holding both truths at once

Big O is the correct tool for the question "how will this scale?" and a misleading one for "which is faster on my actual data?" Experts use asymptotic analysis to rule out algorithms that won't scale, and benchmarking to choose among those that will. Neither replaces the other.

Putting it together

You now have every piece. This final module consolidates them into a repeatable method, names the misconceptions that survive even in experienced engineers, and leaves you with graduated problems to cement the skill.

A framework for analysing any code

  1. Identify n. What grows? If several things grow independently, name them all (n, m, V, E).
  2. Find the dominant structure. Locate the loops and recursion. Apply the rules: sequential adds, nested multiplies, independent inputs stay separate.
  3. Watch how indices move. Multiplicative steps give logs; additive steps give linear. Don't assume a loop runs n times — verify it.
  4. Look beneath each line. Account for hidden costs — a membership test on a list, a sort inside a loop, a string concatenation that copies.
  5. For recursion, write the recurrence and solve it (Master Theorem if it fits, recursion tree otherwise). Don't forget stack depth as space.
  6. State the case. Is this best, worst, or average? Default to worst case unless told otherwise, and say which you mean.
  7. Check for amortization. If a rare operation is expensive but cheap-on-average over a sequence, the amortized bound is the honest one.
  8. Sanity-check against reality. Does the Big O match what you'd observe? Remember constants and caching for the final practical call.

Misconceptions that survive to senior level

Big O measures speed
It measures growth rate, not absolute time. An O(n²) algorithm can be faster than an O(n) one on real inputs.
Lower-order terms never matter
They dominate at small n. Big O is an asymptotic claim about large inputs only.
Worst case = Big O
Case and bound are independent. You can take the Big O of any case; people just default to worst case.
Amortized means average-case
Amortized is a worst-case guarantee over a sequence, with no probability assumptions.
NP means non-polynomial
It means nondeterministic polynomial — verifiable quickly. P ⊆ NP.
O(2n), O(n+5), O(n/2)
All of these are just O(n). Constants and lower-order terms are always dropped.

Capstone problems

Work these without revealing the answer first. If you can reason through all four, you have the analysis skill this course set out to build.

Problem 1 · warm-up

What is the time complexity of checking whether two strings of length n are anagrams, if you do it by sorting both and comparing?

▶ Reveal answer

O(n log n) — dominated by sorting the two strings (each O(n log n)); the final comparison is only O(n), so the sorts win. As a bonus: a counting-based approach (tally character frequencies, then compare tallies) does it in O(n) time, beating the sort by avoiding comparisons — the same trick that lets counting sort beat the Ω(n log n) barrier.

Problem 2 · hidden cost

A developer builds a string by concatenating n single characters in a loop: s = s + c each iteration, in a language where strings are immutable. They expect O(n). What's the real complexity, and why?

▶ Reveal answer

O(n²). Because strings are immutable, each s = s + c builds a brand-new string by copying all characters accumulated so far. The copies cost 1 + 2 + 3 + … + n = n(n−1)/2 = Θ(n²). This is the Module 03 "look beneath each line" lesson and the Module 07 arithmetic-series pattern combined — the innocent-looking + hides a linear copy, turning a loop into quadratic work. The fix is a mutable builder or joining a list, restoring O(n).

Problem 3 · recurrence

An algorithm splits its input into three pieces of one-third size, recurses on all three, and does Θ(n²) work to combine. Give its complexity.

▶ Reveal answer

The recurrence is T(n) = 3T(n/3) + Θ(n²), so a = 3, b = 3, and n^(log_b a) = n^(log₃3) = n¹ = n. The combine work f(n) = Θ(n²) grows strictly faster than n (f(n) = Ω(n^(1+ε))), so this is Master Theorem Case 3: the root work dominates. The answer is Θ(n²). The expensive combine step at the top of the tree swamps everything below it.

Problem 4 · the full picture

You must repeatedly look up items by key and also frequently retrieve them in sorted order. A colleague proposes a hash table for its O(1) lookups. What's the catch, and what would you actually choose?

▶ Reveal answer

The catch is that a hash table has no order — retrieving items sorted requires extracting all n keys and sorting them, an O(n log n) operation every time you need sorted output. If sorted retrieval is frequent, that swamps the O(1) lookups.

A balanced binary search tree gives O(log n) lookups (slightly worse than the hash table) but maintains sorted order intrinsically, so in-order retrieval is O(n) with no sort needed. Which wins depends on the ratio of lookups to sorted retrievals — and reasoning about that tradeoff, rather than reflexively reaching for the lowest single-operation complexity, is exactly the judgement this whole course was building toward.
