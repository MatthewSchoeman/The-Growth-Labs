A Course · Advanced → Expert · The TypeScript type system

Types all the way down.
A language that runs on types.

Most people use TypeScript as annotations bolted onto JavaScript. But the type system is a full programming language in its own right — one that runs at compile time, computing new types from old ones. A type is a set of values; assignability is the subset relation; never is the empty set and unknown the universal one. From there it climbs: structural (not nominal) compatibility, control-flow narrowing of unions, generics with constraints, inference, and then genuine type-level programming — conditional types with infer, mapped types, template-literal types — that compute one type from another. And at the end, the twist that separates it from C#: none of it exists at runtime. tsc checks every type, then erases them all and emits plain JavaScript. AdvancedModule 00

Types as sets

The single mental model that unlocks TypeScript: a type is a set of values. string is the set of all strings; "click" is a set with one element; boolean is { true, false }. Once you see types as sets, the whole system follows. Assignability is the subset relation — a value of type A fits where B is expected exactly when A ⊆ B. A union A | B is set union; an intersection A & B is set intersection. never is the empty set ∅ (no values), and unknown is the universal set (every value). Every rule in this course is a fact about sets.

Two more foundations. TypeScript is structural, not nominal: compatibility is decided by an object's shape, not the name you declared it with — if it has the required members, it fits. And the type system is a compile-time layer that computes with types and then vanishes: tsc checks everything and emits JavaScript with the types erased. So the type system is doing real work — a whole language of its own — that leaves no trace at runtime. That combination, sets plus structural plus erased, is what makes TypeScript both extraordinarily expressive and occasionally surprising.

types-are-sets.ts 
type Kind = "click" | "scroll";   // the set { "click", "scroll" }

const k: Kind = "click";          // "click" ⊆ Kind — assignable
const s: string = k;              // Kind ⊆ string — assignable (widening)
const bad: Kind = someString;      // string ⊄ Kind — NOT assignable

type Never = "click" & "scroll";  // ∅ — no value is both → never

Every module builds on this. Narrowing shrinks a set as the compiler learns more; generics are functions over sets; conditional and mapped types compute new sets from old ones; and the checker's job is to verify subset relationships everywhere. Hold the model the whole way: a type is a set, assignability is ⊆, and the type system is a language that computes with those sets and then disappears.

↳ OOP & Coding Principles · Advanced C#
This is the opposite of C#'s nominal typing, where a class matches only its declared name and base types. TS's structural model is duck typing made static — and its erasure is the mirror image of C#'s reflection: C# keeps full type info at runtime, TypeScript keeps none.
Types are labels the compiler attaches to variables.
Types are sets of values, and the relationships between them (subset, union, intersection) are what the checker reasons about. Thinking in sets explains assignability, narrowing, and never far better than thinking in labels — it's the model the compiler actually uses.
TypeScript is just JavaScript with types that stick around.
The types are erased — the emitted JavaScript has none of them. TypeScript is a compile-time verification layer plus a genuinely powerful type-level language, sitting on top of JS. At runtime, there's only JavaScript; the entire type system has done its work and left.
StructuralAdvancedModule 01

Structural typing & assignability

TypeScript decides whether one type fits another by shape, never by name. An object is assignable to a target type if it has (at least) all the required members with compatible types — it doesn't matter what you called it or whether it declares anything extra. This is structural typing, and it's why two unrelated types with the same members are interchangeable. There's one wrinkle: excess-property checks. A fresh object literal assigned directly is checked more strictly — extra properties are flagged as likely typos — but the same object stored in a variable first passes freely, because structurally it still has everything required.

assignability.ts 
type Point = { x: number; y: number };

const a: Point = { x: 1, y: 2 };              // ✓ exact shape
const b: Point = { x: 1 };                    // ✗ Property 'y' is missing
const c: Point = { x: 1, y: 2, z: 3 };       // ✗ excess property 'z' (literal)
const raw = { x: 1, y: 2, z: 3 }; const d: Point = raw;  // ✓ via a variable — z ignored

Build an object by toggling which members it has, choose whether it's assigned as a literal or via a variable, and watch the checker decide assignability to Point:

Does the shape fit?
structural match · excess-property check
target · type Point = { x: number; y: number }
source object — toggle members
x: numbery: numberz: number (extra)
assigned as
const p: Point = { x, y }
✓ assignable
Missing a required member fails structurally. An extra member fails only on a fresh object literal (excess-property check) — route it through a variable and it's accepted, because the shape still contains everything Point needs.
↳ OOP & Coding Principles
Structural typing is the "program to an interface" principle without the ceremony — any object of the right shape satisfies the contract, no implements declaration needed. It's duck typing, but fully checked at compile time.
Two types must share a name or base to be compatible.
Compatibility is by shape. A { x; y } and a differently-named Vector2 with the same members are freely interchangeable. TypeScript never checks the declared name — only whether the required members are present and compatible.
Extra properties always cause an error.
Only fresh object literals are checked for excess properties, as a typo-catching heuristic. The identical object assigned through a variable passes, because structurally it has everything required. Same values, different check — and knowing which is which explains a lot of confusing errors.
UnionsAdvancedModule 02

Unions & narrowing

A union type A | B | C is a value that could be any one of several types — and the checker's superpower is narrowing: as control flow proves which one it is, the type shrinks. A typeof check, an in check, or best of all a discriminated union (a shared literal kind field) lets a switch narrow the value to exactly one member in each branch, unlocking that member's specific properties. And never makes it exhaustive: in the default branch, if you've handled every case, the value has narrowed to never — so a leftover case becomes a compile error. This is the same flow analysis as C#'s nullable narrowing, generalized to any union.

narrowing.ts 
type Ev =
  | { kind: "click";  url: string }
  | { kind: "scroll"; depth: number }
  | { kind: "error";  message: string };

switch (ev.kind) {
  case "click":  ev.url;      // narrowed to the click member
  case "scroll": ev.depth;    // narrowed to the scroll member
  default: const _: never = ev;  // exhaustive — leftover cases won't compile
}

Pick which event you're holding and watch the switch narrow the union to a single member — exposing only that member's property. Then drop the error case and see the exhaustiveness check fail:

discriminated union · exhaustive with never
ev : Ev  —  narrowed type in the matched branch
ev: Ev // pick an event to narrow
case "click" → ev.url{ kind: "click"; url: string }
case "scroll" → ev.depth{ kind: "scroll"; depth: number }
case "error" → ev.message{ kind: "error"; message: string }
✓ exhaustive — in default, ev narrows to never
pick an event to see the union narrow to a single member
The shared kind field discriminates the union: each case narrows ev to exactly one member, exposing its property. With every case handled, the default sees never — drop one and that assignment fails, so the compiler catches the missing case.
↳ Advanced C# · Vol. II (nullable) · Vol. I (pattern matching)
This is the C# null-flow lab generalized: there, the compiler narrowed string? from maybe-null to not-null through a guard; here it narrows any union through control flow. And the never-based exhaustiveness is exactly C#'s switch exhaustiveness (CS8509) — the empty set standing in for "no cases left."
To use a union you have to cast it to the type you want.
You narrow it instead — a typeof, in, or discriminant check proves which member it is, and the compiler updates the type automatically. Casting throws away safety; narrowing keeps it, and the checker rewards you with exactly the right member's properties.
A default case is just defensive boilerplate.
Assigning the value to never in the default turns it into an exhaustiveness check. If every case is handled, the value is never there and it compiles; add a new union member and forget a case, and the assignment fails — the compiler points you straight at the gap.
GenericsAdvancedModule 03

Generics & constraints

Generics are functions at the type level: a type parameter T is an input the compiler fills in. The magic is inference — you rarely write the type argument, because TypeScript deduces T from the value you pass. A constraint (T extends { id: number }) restricts the input to types that have a certain shape, and — crucially — preserves the specific type rather than collapsing it to the constraint. So a well-constrained generic accepts exactly the right values, infers their precise type, and rejects everything else, all before your code runs.

generics.ts 
function save<T extends { id: number }>(x: T): T { return x; }

save({ id: 1, name: "Snip" });   // T inferred as { id: number; name: string }
save({ name: "Snip" });          // ✗ Property 'id' is missing (constraint)
save(42);                     // ✗ number does not satisfy { id: number }

Call the constrained generic save<T extends { id: number }> with different arguments and watch T get inferred — or the constraint reject a value that doesn't fit:

T = // call save() to infer T
call the generic with an argument
A value with id infers T as its full, precise shape and passes. A value missing id fails the constraint; a number isn't even an object with id. The constraint gates the input while inference keeps the exact type.
↳ Data Structures · Advanced C# (generics)
Same idea as C# generics with a where constraint — but with a twist that matters at runtime: C# specializes generics into real machine code per type, while TypeScript generics are pure compile-time checks that erase entirely. One produces code; the other produces only guarantees.
You have to write the type argument, like save<User>(...).
Almost never — TypeScript infers it from the argument. Explicit type arguments are for the rare cases where inference can't (or shouldn't) figure it out. Good generic APIs are designed so the caller writes no type arguments at all.
A constraint T extends X makes T behave like X.
The constraint gates what's allowed in, but T stays the caller's specific type — so save returns the exact shape passed, not a widened { id: number }. That preservation of the precise type is what makes generics compose instead of losing information.
InferenceAdvancedModule 04

Type inference & widening

TypeScript infers a type for almost everything you don't annotate — and the rules around widening trip up even experienced developers. A let binding of "click" is inferred as string (widened, because a let can be reassigned), while a const of a primitive keeps the literal type "click". But const on an object does not deep-narrow its members — they still widen. To freeze everything to its narrowest literal, readonly form, you need as const. Knowing exactly what gets inferred is the difference between a type that's precise and one that's uselessly broad.

widening.ts 
let   a = "click";                 // a: string  (widened — let is mutable)
const b = "click";                 // b: "click" (literal kept)

const o = { kind: "click", depth: 3 };            // { kind: string; depth: number }
const p = { kind: "click", depth: 3 } as const;   // { readonly kind: "click"; readonly depth: 3 }

Take the object { kind: "click", depth: 3 } and see how it's inferred under each declaration. Note that const alone still widens the members — only as const pins them to literals:

What does TypeScript infer?
widening · literal types · as const
declaration of  { kind: "click", depth: 3 }
inferred type
switch the declaration to see how inference changes
let widens to the general type; const keeps a primitive's literal but still widens object members; as const freezes every member to its readonly literal type. The narrower the inference, the more the type system can prove.
const always gives me a literal type.
Only for primitives. const s = "click" is "click", but const o = { k: "click" } infers { k: string } — the members widen because the object is still mutable. Reach for as const when you need the members frozen to literals too.
Widening is a bug I should always fight.
It's usually what you want — a let counter should be number, not 0. Widening only bites when you need the literal (discriminants, tuple shapes, config objects), and as const is the precise tool for exactly those cases. Understand it and you stop over-annotating.
Type-levelExpertModule 05

Conditional types & infer

Here the type system becomes a real programming language. A conditional type — T extends U ? X : Y — branches on a type, computing one type or another. And infer is pattern-matching for types: inside the extends clause, infer E introduces a new type variable that TypeScript solves for by matching structure. That's how the built-in ReturnType<T> extracts a function's return type, and how you pull the element type out of an array or the value out of a Promise. Conditionals plus infer plus recursion make the type system Turing-complete — you're writing programs whose values are types.

conditional.ts 
type Unpack<T> =
    T extends Array<infer E>   ? E :   // if T is an array, extract its element type
    T extends Promise<infer E> ? E :   // else if a Promise, extract its value type
    T;                             // else pass T through unchanged

type A = Unpack<string[]>;      // string

Feed the conditional type Unpack<T> different inputs and watch infer match the structure and extract the inner type — or fall through when nothing matches:

Match the structure, extract the type

evaluation

choose an input and watch infer extract the inner type
string[] matches Array<infer E> with E = string. Promise<number> matches the second branch with E = number. boolean matches neither and falls through to T. infer solves the variable by structural matching.
↳ Advanced C# · Vol. I (expression trees)
Conditional types are metaprogramming, like C#'s expression trees — but they run in the type checker, computing types instead of values. infer is the type-level equivalent of destructuring a tree to pull out a node: pattern-matching over structure to extract a piece.
Conditional types are just ternaries for picking between two known types.
With infer and recursion they're a full computation — extracting, transforming, and building types. ReturnType, Awaited, and most of the utility types are conditional types doing real work, and you can write your own that compute arbitrarily complex results.
infer is an obscure keyword I'll never need.
It's how you reach inside a type to grab a part — the element of an array, the resolved value of a promise, the parameters of a function. Any time you've wished you could say "the type of the thing inside this type," infer in a conditional is the answer.
Type-levelExpertModule 06

Mapped & template types

This is the peak of type-level programming, and the program from the hero. A mapped type — { [K in keyof T]: … } — iterates over the keys of a type and builds a new object type, transforming each member. Combine it with key remapping (as) and template-literal types (types built from string interpolation, like `on${Capitalize<K>}`) and you can compute an entirely new shape from an old one. The classic example: derive a Handlers<T> type whose keys are onId, onName, … and whose values are typed callbacks — generated automatically from the original object's keys and value types.

mapped.ts 
type User = { id: number; name: string };

type Handlers<T> = {
  [K in keyof T as `on${Capitalize<string & K>}`]: (v: T[K]) => void;
};   // { onId: (v: number) => void; onName: (v: string) => void }

Build Handlers<User> one step at a time: extract the keys with keyof, transform them through the template literal, then map into the final object type with correctly-typed callbacks:

keyof gives the union "id" | "name"; the template literal rewrites each to "onId" | "onName"; the mapped type builds an object whose callbacks are typed by each original member — T[K]. A whole type, computed from another.
↳ Front-End Engineering · Mastering Angular
Mapped types power the typed APIs you relied on in the front-end courses — form values derived from a schema, event maps, prop types generated from a component's shape. When a library "just knows" the exact types of your keys and handlers, a mapped type is doing this computation behind the scenes.
Utility types like Partial and Readonly are compiler built-ins.
They're ordinary mapped types you could write yourself: Partial<T> is { [K in keyof T]?: T[K] }. The whole utility-type library is mapped and conditional types — which means you can build your own transformations exactly the same way.
Template-literal types are just for pretty string unions.
They're a computation over strings in the type system — combined with mapped types and infer, they parse and build type-safe strings: route params, CSS units, event names, SQL fragments. Real libraries use them to type `/users/${id}`-style APIs with full checking.
Type-levelExpertModule 07

keyof & indexed access

The operators that let types derive from other types and from values. keyof T produces the union of a type's keys. Indexed access T[K] looks up the type of a property — and because K can be a union, T[keyof T] gives the union of all value types. And typeof value crosses from the value world into the type world, capturing the inferred type of a runtime value. Together they're how you keep types in sync with data automatically: derive types from a config object, index into a shape, and never hand-write a type that could drift from its source.

keyof.ts 
const config = { host: "localhost", port: 8080, tls: true };

type Config = typeof config;          // { host: string; port: number; tls: boolean }
type Keys   = keyof Config;           // "host" | "port" | "tls"
type Port   = Config["port"];          // number

Start from a runtime config value and derive types from it: capture its type with typeof, list its keys with keyof, and index into it with [K]:

Derive types from values and keys
typeof · keyof · T[K]
value · const config = { host: "localhost", port: 8080, tls: true }
derived type
— // derive a type from the value
derive a type from the config value
typeof lifts the value into a type; keyof lists its keys as a union; indexed access looks up a member's type. Derive types from your data and they can never fall out of sync with it.
typeof in a type means the same as typeof at runtime.
They're different operators in different worlds. Runtime typeof returns a string like "object"; type-level typeof captures the full inferred type of a value. TypeScript overloads the keyword across the value and type spaces — same spelling, entirely different job.
You should declare a type, then build the object to match it.
Often it's better to write the object and derive the type with typeof and keyof — one source of truth, no drift. Especially for config, enums-as-objects, and lookup tables, deriving the type from the data beats maintaining two parallel declarations.
The checkerExpertModule 08

Variance & function types

Function-type compatibility is where variance gets subtle. Under strictFunctionTypes, function return types are covariant — a function returning Dog is assignable where one returning Animal is expected, because every Dog is an Animal. But function parameters are contravariant — a handler that accepts any Animal can stand in where a Dog handler is expected (it can certainly handle a Dog), but not the reverse: a Dog-only handler can't be used where arbitrary Animals will be passed. It feels backwards until you think about who calls whom — and it's the same variance rule as C#'s in/out, applied structurally to functions.

fn-variance.ts 
class Animal {} class Dog extends Animal {}

let getAnimal: () => Animal;
getAnimal = () => new Dog();      // ✓ return covariance (Dog ⊆ Animal)

let onDog: (d: Dog) => void;
onDog = (a: Animal) => {};        // ✓ param contravariance (handles any Animal)
onDog = (d: Dog) => {}; getDog = getAnimal;  // ✗ can't narrow the other way

Each row assigns one function type to another. Click to test it against the variance rules — and see why parameters flip the direction that return types keep:

click an assignment to test it against strictFunctionTypes
Returns keep the subtype direction (Dog fits where Animal is wanted). Parameters reverse it: a handler of any Animal fits where a Dog handler is needed, but a Dog-only handler can't take arbitrary Animals. Who calls whom sets the direction.
↳ Advanced C# · Vol. II (variance)
Identical rule to C#'s in/out from Vol. II — producers (returns) covary, consumers (parameters) contravary — but TypeScript infers it structurally for every function type rather than requiring you to annotate the variance. The theory is the same; the ergonomics differ.
A function taking a more specific parameter is a safe drop-in.
It's the opposite. A callback typed (d: Dog) => void can't stand in where the caller will pass any Animal — it might get a Cat. The safe substitution is a handler that accepts the wider type. Parameter contravariance is unintuitive precisely because it inverts the usual subtype direction.
Method parameters follow the same strict rule.
For historical, ergonomic reasons, method parameters (and event handlers via EventListener) are checked bivariantly — a deliberate unsoundness. strictFunctionTypes enforces contravariance for function-typed properties but exempts method syntax, which is why the same-looking code can check differently.
The checkerExpertModule 09

Type guards & assertions

Narrowing has built-in triggers (typeof, in, discriminants), but you can teach the checker your own. A user-defined type guard is a function whose return type is a type predicate — x is Link — and when it returns true, TypeScript narrows the argument to that type in the branch. An assertion function — asserts x is Link — narrows for the rest of the scope instead of a branch, throwing if the claim is false (the pattern behind runtime validators). These are the bridge between runtime checks and compile-time types: your validation logic teaches the type system what you've proven.

guards.ts 
function isLink(x: Link | ApiError): x is Link {
    return "code" in x;                 // runtime check → type predicate
}
if (isLink(x)) x.code;           // x narrowed to Link in this branch
else          x.message;        // x narrowed to ApiError in the other

function assertLink(x: unknown): asserts x is Link { /* throw if not */ }

You're holding x: Link | ApiError. Apply a custom guard and watch x narrow to a single type — exposing that type's members — in each branch, or for the rest of the scope via an assertion:

Teach the checker what you've proven
x is Link · asserts x is Link
x — narrowed type
x: Link | ApiError // apply a guard to narrow
accessible membersonly shared members
apply the guard to narrow the union
The type predicate narrows x to Link inside the if and to ApiError in the else, each exposing its own members. An assertion function narrows for the whole rest of the scope — the pattern every runtime validator uses to hand results back to the type system.
↳ Mastering ASP.NET Core · SDLC
Type guards are where runtime and compile time meet: at a trust boundary — a parsed request body, an API response — you validate at runtime and a guard hands the proof to the type checker. It's the typed edge of the input-validation discipline the API and lifecycle courses insisted on.
Returning a boolean from a check narrows the type.
Only if you annotate the return as a type predicate (x is Link). A plain : boolean return does the runtime check but tells the compiler nothing — x stays the union. The predicate is what connects your logic to the narrowing.
A guard makes x safe everywhere after you call it.
A type guard narrows only inside the branch that tested it; an assertion function (asserts x is …) is what narrows for the rest of the scope — and it must actually throw when false. Using the wrong one gives you either too little narrowing or a false guarantee.
RuntimeExpertModule 10

Types are erased

The fact that changes how you architect everything: types have no runtime existence. tsc type-checks your program and then erases every type — interfaces, annotations, generics, the lot — emitting plain JavaScript. There is no reflection, no typeof x === "User", no way to ask at runtime what type something was, because the answer was deleted. This is the exact opposite of C#, which keeps full type metadata and lets you inspect it at runtime. The consequence: any decision that must happen at runtime — validating an API payload, dispatching on a shape — needs a value-level discriminant (a kind field, a validator), because the types you'd want to consult are gone.

erasure.ts → erasure.js 
// TypeScript in:
interface User { id: number; name: string; }
function greet(u: User): string { return `hi ${u.name}`; }

// JavaScript out — every type erased:
function greet(u) { return `hi ${u.name}`; }   // no interface, no annotations

Compile the TypeScript above and watch tsc strip every type away, leaving runnable JavaScript. Then try to check a type at runtime and see that there's simply nothing left to check:

— press compile to run tsc —

runtime type info—
The interface and every annotation vanish; the JS is what actually runs. There's no way to ask an object "are you a User?" at runtime — the type is gone. Runtime decisions need a value you can inspect, not a type you can't.
↳ Advanced C# · Vol. I (generics · reflection) · Bits
The starkest contrast in the whole library: C# generics specialize and its types are reflectable at runtime; TypeScript's erase completely. What's actually in memory at runtime — the Bits question — is plain JavaScript objects with no type tags at all. Same word "type," opposite runtime reality.
I can check at runtime whether a value matches an interface.
You can't — the interface doesn't exist at runtime. You check values: a discriminant field, the presence of a property, or a validator library (Zod, io-ts) that carries a runtime schema alongside the type. The type itself is unavailable because it was erased.
Since types vanish, they don't affect the running program.
They shape it profoundly before it runs — every bug the checker caught is one that never shipped. Types are a compile-time proof system: enormous influence on correctness, zero runtime footprint. That's the trade TypeScript makes, and it's the opposite of C#'s reflective, type-carrying runtime.
ExpertModule 11

The whole type system

Step back and it's one coherent language. Types are sets; assignability is ⊆; the checker narrows sets as it learns; generics are functions over sets; conditional, mapped, and template types compute new sets from old ones; and the whole apparatus runs at compile time and then erases. Every feature in the course is a move in that single system:

the type system, in one view 
  a type            = a set of values                 // m00
  A ⊆ B             = assignable · structural          // m01
  A | B  + narrowing = union, shrunk by control flow    // m02
  <T extends C>      = a function over types             // m03–04
  T extends U ? … : … / infer / keyof / mapped          // m05–07 · compute types
  variance · guards  = the checker's finer rules         // m08–09
  tsc → .js          = checked, then erased              // m10

And here is Snip's client, fully typed — a discriminated event union, a derived handler map, a guarded API boundary, and the reminder that all of it compiles away:

snip-client.ts 
type Ev =                                          // m02 · discriminated union
  | { kind: "click"; url: string } | { kind: "scroll"; depth: number };

type Handlers = { [K in Ev["kind"] as `on${Capitalize<K>}`]: (e: Ev) => void };  // m06 · mapped + template

function handle<T extends Ev>(e: T): T["kind"] {   // m03/m07 · generic + indexed access
  switch (e.kind) {
    case "click":  fetch(e.url); return e.kind;   // narrowed to the click member
    case "scroll": log(e.depth); return e.kind;   // narrowed to scroll
  }
}
function isEv(x: unknown): x is Ev { return !!x && "kind" in (x as any); }  // m09 · guard at the boundary
// ...and tsc erases every type above, emitting plain JavaScript (m10)
The type system, entire

The event union narrows through the switch so each case sees exactly its member (m02). Handlers is computed from the union's kind values via a mapped type over a template literal (m06). handle is a generic constrained to Ev, returning an indexed-access type (m03, m07). isEv is a type guard turning a runtime check into a compile-time proof (m09). And every type here is a set, related by ⊆, that the checker verifies and then deletes (m00, m10). The annotations you write are the visible surface of a compile-time language — one that reasons about your whole program and then vanishes without a trace.

The type system, mastered.

You started with one idea — a type is a set of values — and watched the whole system unfold from it: structural assignability as subset, narrowing as the set shrinking under evidence, generics as functions over sets, and conditional, mapped, and template types as a genuine language that computes new types from old. You can read the type-level programs the libraries you use are written in, and write your own. Go make illegal states unrepresentable — and know exactly what survives to runtime and what doesn't.
