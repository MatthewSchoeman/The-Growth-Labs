BeginnerModule 00

Hello, C# & .NET

C# is a statically typed, object-oriented language created by Microsoft and now thoroughly cross-platform and open-source. .NET is the platform it runs on — the runtime that executes your code, plus an enormous standard library. The two are often spoken of together, but they're distinct: C# is the language you write, .NET is the engine and toolbox it runs against. The same .NET also runs F# and Visual Basic.

When you build, the C# compiler turns your source into a portable intermediate language (IL), which the runtime compiles to native machine code just in time as the program runs. Two ideas matter from day one. First, C# is compiled and type-checked before it ever executes — whole categories of mistakes are caught at build time, not in production. Second, it's managed: a garbage collector handles memory for you, so there's no manual free (though, as the series' bits course showed, knowing what's underneath still pays off).

Here's a complete modern C# program. Recent versions allow top-level statements, so the ceremony of a class and a Main method is optional for simple cases:

Program.csvar name = "world";
Console.WriteLine($"Hello, {name}!");   // $ = string interpolation

int a = 7, b = 5;
Console.WriteLine($"{a} + {b} = {a + b}");    // Hello, world!  /  7 + 5 = 12

That $"..." is an interpolated string — expressions inside {} are evaluated and spliced in, type-checked like everything else. Try misspelling a type or passing the wrong kind of value and the program won't compile; the error arrives at your desk, not your users'. That's the promise this whole course builds on: the compiler is on your side.

C# only runs on Windows.
Not since .NET Core. Modern C# runs on Windows, Linux, and macOS, in containers and the cloud, and compiles to WebAssembly for the browser. The Windows-only era ended years ago — it's now a mainstream cross-platform language.
C# and .NET are the same thing.
C# is the language; .NET is the runtime and library it executes on. You write C#, it compiles to IL, and .NET runs that IL. The distinction matters because the same .NET hosts other languages, and .NET versions evolve independently of C# versions.
TypesBeginnerModule 01

Types, variables & values

Every variable in C# has a type known at compile time, and that type is a contract the compiler enforces on every use. You can write the type explicitly or let the compiler infer it with var — but make no mistake, var is still fully static. var n = 5; makes n an int forever, not a dynamic free-for-all.

c#int count = 42;            // 32-bit integer
double price = 9.99;       // 64-bit float — fine for math
decimal total = 9.99m;     // exact base-10 — use this for money
bool ok = true;
string who = "Ada";
var list = new List<int>();  // inferred as List<int> — still static

The deepest divide in C#'s type system is value types versus reference types. A value type (the numbers, bool, char, and any struct) holds its data directly in the variable's box. A reference type (class objects, arrays, and string) holds a reference — an address pointing to the data, which lives on the heap. This single distinction decides what happens when you copy or pass a variable.

↳ Bits course
This is exactly the value-vs-reference and stack-vs-heap story from Bits, Pointers & Overflow, now wearing C# syntax. A struct is a value living in its own box; a class is a reference to a box on the heap. C# just makes the choice explicit in how you declare the type.

Assign or pass a value type and you get an independent copy; the original is untouched by later changes. Assign or pass a reference type and you copy the reference — both variables now point at the same object, so a change through one is visible through the other. Below, the same b = a; a.X = 99; behaves completely differently depending on whether Point is a struct or a class:

struct vs class
value types copy · reference types share
A struct copies its data; a class copies only the reference to shared data.
var makes a variable dynamic, like in JavaScript.
var is pure type inference — the compiler picks the exact static type from the right-hand side and locks it in. var x = 5 is identical to int x = 5. For genuinely dynamic typing you'd opt in with the separate dynamic keyword, which is rarely needed.
Strings are value types because they feel like simple values.
string is a reference type — but an immutable one. You can't change a string in place; every "modification" makes a new string. That immutability is why it behaves like a value even though it's a reference under the hood.
Control flowBeginnerModule 02

Control flow & methods

The basics are familiar: if/else, while, do/while, the C-style for, and foreach for iterating a collection without managing an index. What sets modern C# apart is the switch expression and pattern matching — a far more powerful relative of the old switch statement that returns a value and matches on shape, type, and condition, not just constants.

c#string Describe(int n) => n switch
{
    0            => "zero",
    < 0         => "negative",        // relational pattern
    < 10        => "small",
    >= 100      => "large",
    _            => "medium"          // _ is the catch-all (discard)
};

That single switch expression replaces a tangle of if/else if, reads top-to-bottom as a table of cases, and — because it's an expression — hands back a value the compiler guarantees every path produces. Pattern matching also works on types (obj is string s), on properties ({ Age: >= 18 }), and on tuples, which makes it a workhorse you'll meet everywhere in modern C#.

Pick a value and watch the switch expression test each arm in order, top to bottom, and stop at the first pattern that matches:

Pattern matching, arm by arm
the first matching pattern wins
Describe( )
Each arm is a pattern. Evaluation stops at the first one that matches the input.

Logic gets packaged into methods — a name, typed parameters, and a return type. C# supports overloading (same name, different parameter lists), default and named arguments, and expression-bodied members with => for one-liners (as in Describe above). A method's signature is itself a small contract: the types in and the type out are checked at every call site.

A switch can only compare against constant values.
The old statement mostly did, but C#'s modern switch expression matches relational patterns (< 10), type patterns, property patterns, and combinations with and/or. It's closer to a typed, exhaustive decision table than a C-style switch.
Expression-bodied methods (=>) are lambdas.
They look alike but differ. => on a member is just concise syntax for a method with one expression. A lambda is an anonymous function value you can store and pass around (Module 09). Same arrow, different roles.
ObjectsBeginnerModule 03

Classes & objects

A class is a blueprint that bundles state (data) with behavior (methods) and controls who may touch what. You create objects (instances) from it with new. The headline C# feature here is the property: it looks like a field from the outside but is really a pair of get/set methods, so reading or writing it can run code — validate, compute, log, or simply forbid.

c#class BankAccount
{
    public string Owner { get; init; }        // set once, at creation
    public decimal Balance { get; private set; } // readable; only this class writes it

    public void Deposit(decimal amount)
    {
        if (amount <= 0) throw new ArgumentException("Must be positive");
        Balance += amount;                       // the ONLY way balance grows
    }
}

var acct = new BankAccount { Owner = "Ada" };  // object initializer
acct.Deposit(100);
// acct.Balance = 999;  ← compile error: no public setter
↳ OOP course
This is encapsulation from OOP & Coding Principles, expressed in C#. The private set and the validating Deposit method are how C# guarantees an invariant — balance never goes negative, and state changes only through approved operations. Properties make "hide the data, expose operations" the path of least resistance.

Try to break the account below. The balance has no public setter, so the only way to change it is through methods that check the rules first — exactly the guarantee a raw public field could never give:

Deposit and Withdraw validate before touching Balance — the invariant can't be broken from outside.

When you want a simple, immutable data holder, C# offers the record — a class with value-based equality and concise syntax generated for you: record Person(string Name, int Age); declares a whole type in one line, with a constructor, read-only properties, and sensible Equals and ToString baked in.

A property is just a fancy public field.
A property is a pair of methods dressed up as a field. That's the whole point: it can validate, compute on the fly, raise events, or be read-only — none of which a bare field can do. Public fields give up all control; properties keep it while staying ergonomic.
Records are just classes with less typing.
They default to value-based equality (two records are equal if their contents match, not their references) and encourage immutability with with-expressions for copies. Use a record for data; a class for an entity with identity and mutable, behavior-rich state.
ObjectsIntermediateModule 04

Inheritance & interfaces

C# gives you two ways for types to relate. Inheritance (class Dog : Animal) is an is-a relationship where a derived class gains the base's members and can override marked-virtual behavior. An abstract class can't be instantiated and may leave methods unimplemented for subclasses to fill in. The companion idea is the interface — a pure contract of members with no implementation, a promise that "whoever implements me provides these."

c#abstract class Shape
{
    public abstract double Area();        // no body — subclasses must supply one
}

class Circle : Shape
{
    public double Radius { get; init; }
    public override double Area() => Math.PI * Radius * Radius;
}

Shape s = new Circle { Radius = 2 };  // hold a Circle as a Shape
double a = s.Area();                  // runs Circle.Area — chosen at runtime

That last line is polymorphism: the variable's type is Shape, but the object is a Circle, so Area() dispatches to Circle's override at runtime. Code written against the base type or interface works with any implementation — present or future — without change.

↳ OOP course
This is polymorphic dispatch from OOP & Coding Principles. One Area() call, many implementations, each object answering for itself. It's also the engine behind the Open/Closed Principle: add a new Shape and existing code that loops over shapes never changes.

Send the same Area() message to a list of different shapes and watch each one run its own override:

One call, many implementations
foreach (Shape s in shapes) s.Area();
Press run — each shape computes its own area through the same call.
The variable is a Shape; the object decides which Area() actually runs.

When should you reach for which? Use an interface to describe a capability many unrelated types can have (IComparable, IDisposable) — a class can implement many. Use an abstract class to share common implementation among closely related types. A type inherits from one class but can implement any number of interfaces.

Inheritance is the main tool for reusing code.
Often composition (holding an object) reuses better than inheritance (being a subtype), because deep hierarchies are rigid and fragile. Reach for inheritance when there's a genuine, stable is-a; reach for interfaces and composition far more often.
Interfaces and abstract classes are interchangeable.
An interface is a contract a type can adopt alongside many others; an abstract class is a partial base you can only have one of. Interfaces model "can do"; abstract classes model "is a kind of, with shared guts."
GenericsIntermediateModule 05

Collections & generics

Real programs juggle many values, and C#'s collections — backed by generics — are how. A generic type is parameterized by another type: List<T> is "a list of some type T," and List<int> fixes T to int. The payoff is type safety with zero casting: the compiler knows a List<int> yields ints, so you can't accidentally put a string in or pull the wrong type out.

c#List<int> nums = [ 3, 1, 4, 1, 5 ];     // dynamic array, type-safe
nums.Add(9);
int first = nums[0];                       // indexing is O(1)

Dictionary<string, int> ages = new()
{
    ["Ada"] = 36, ["Alan"] = 41      // key → value lookup
};
int a = ages["Ada"];                     // hashed lookup — O(1) average

Each collection makes different performance trade-offs, and choosing well is the difference between a snappy program and a slow one. A List<T> is a dynamic array: instant by index, but searching its contents is linear. A Dictionary<K,V> and a HashSet<T> are hash-based: near-instant lookup by key, at the cost of order.

↳ Data Structures + Big O
The collection you pick is a data structure decision from earlier in the series. List<T> is a dynamic array; Dictionary and HashSet are hash tables; there's a LinkedList<T> too. The lab below shows the Big O of each operation — the same complexity analysis, now as a day-to-day C# choice.

structure decides the cost
A List scans to find a value; a Dictionary and HashSet jump straight to it by hashing.
List<T> is a linked list.
It's a dynamic array — contiguous storage that grows as needed. That's why indexing is O(1) and why it's the right default. C# does have a real LinkedList<T>, but you reach for it rarely, only when you need cheap insertion in the middle.
Generics are just to save you from writing casts.
They also deliver real performance: a List<int> stores ints directly, with no boxing into objects and no runtime type checks. Generics give you one implementation that's both type-safe and efficient for every T — the old object-based collections were neither.
DataIntermediateModule 06

LINQ — querying in the language

LINQ (Language Integrated Query) is one of C#'s defining features: a single, typed, composable way to query data — whether it's an in-memory list, an XML document, or a database table. Instead of hand-writing loops, you describe what you want as a pipeline of operations, and LINQ produces it. The same query you'll write over a List below is, with Entity Framework, translated into SQL and run on a database.

c#var result = products
    .Where(p => p.Price < 50)          // filter — keep cheap items
    .OrderBy(p => p.Price)             // sort ascending by price
    .Select(p => p.Name);             // project — just the names

// p => ... is a lambda: an inline function. result is IEnumerable<string>.

Each step takes a sequence and returns a new sequence, so they chain. The p => p.Price < 50 bits are lambdas — small anonymous functions passed as arguments (much more on those in Module 09). And LINQ is lazily evaluated: building the query does no work; the pipeline runs only when you actually iterate it (with foreach, .ToList(), etc.). That deferral is what lets the database provider rewrite the whole chain into one efficient query.

↳ SQL course → EF
If Where/OrderBy/Select/GroupBy feel like WHERE/ORDER BY/SELECT/GROUP BY from SQL & PostgreSQL — that's the point. LINQ is set-based querying brought into C# with full type-checking. Entity Framework turns this exact syntax into SQL, which is where this track is headed next.

Build a query by toggling clauses. Watch the C# update as you go, and the results recompute live against a small product catalog:

results
Each clause is one operator in the pipeline. Nothing runs until the results are read.
A LINQ query runs the moment you write it.
Most LINQ is deferred — defining the query builds a recipe; it executes only when enumerated. That's why you can keep composing, and why a database provider can fold the whole chain into a single round-trip. Methods like .ToList() or .Count() force it to run.
LINQ is just for databases.
LINQ works on any IEnumerable<T> — lists, arrays, dictionaries, files, even your own types. The same operators query in-memory objects (LINQ to Objects) and remote data (LINQ to Entities). One query language, many sources.
RobustnessIntermediateModule 07

Exceptions & error handling

When a method hits a situation it can't handle — a missing file, a bad argument, a failed network call — it throws an exception: an object describing what went wrong that travels up the call stack until something catches it. This separates the error path from the happy path, so normal code stays readable and failures can't be silently ignored.

c#try
{
    var text = File.ReadAllText(path);   // might throw
    Console.WriteLine(text);
}
catch (FileNotFoundException ex)         // catch a SPECIFIC type
{
    Console.WriteLine($"Missing: {ex.FileName}");
}
catch (IOException ex) when (ex.HResult != 0)  // filtered catch
{
    Console.WriteLine("I/O problem");
}
finally
{
    // always runs — cleanup, whether or not we threw
}
✓ try succeeds
run the try block
→ skip every catch
→ run finally
→ carry on
✕ try throws
stop the try block
→ find the first matching catch
→ run finally (always)
→ if none match, propagate up

Catch the most specific exception types you can actually do something about, and let the rest propagate to a place that can. For deterministic cleanup of resources — files, connections, locks — C# pairs exceptions with IDisposable and the using statement, which guarantees disposal even if an exception unwinds the stack:

c#using var conn = new SqlConnection(cs);  // disposed automatically at scope end
conn.Open();
// ... conn.Dispose() runs even if the next line throws
A good program catches every exception so it never crashes.
Catching what you can't handle just hides bugs. Catch only what you can meaningfully recover from, close to where it happens; let truly unexpected failures surface (and get logged) rather than swallowing them into a corrupt, silent state.
Exceptions are a fine way to handle ordinary control flow.
They're for the exceptional — they're relatively costly and obscure intent when used as goto. For expected outcomes (a lookup that may miss), prefer return values like TryParse's bool, nullable results, or a result type.
RobustnessIntermediateModule 08

Null safety

A null reference — a variable that points at no object — is the source of the dreaded NullReferenceException, so common its inventor called it his "billion-dollar mistake." Modern C# fights back with nullable reference types: with the feature on, reference types are non-null by default, and you must explicitly opt into null with a ?. The compiler then warns you wherever a possibly-null value is used unsafely.

c#string  name  = null;   // warning: name is non-nullable
string? maybe = null;   // fine: explicitly nullable
int?    count = null;   // nullable value type too

A family of null operators makes working with maybe-null values safe and concise. The null-conditional ?. stops a chain the moment it hits null (returning null instead of throwing), and the null-coalescing ?? supplies a fallback. Together they turn a minefield into a one-liner:

c#string city = person?.Address?.City ?? "unknown";
//            ^ if person or Address is null, the whole chain is null,
//              and ?? supplies "unknown" — no exception, ever.

Toggle parts of the chain to null and watch how ?. short-circuits and ?? catches the result. Click a box to flip it between a value and null:

The null-safe chain
person?.Address?.City ?? "unknown"
Click person or Address to toggle null. ?. stops the chain safely; plain . would throw.
Nullable reference types make null errors impossible at runtime.
They're a compile-time aid — warnings, not airtight guarantees. Old libraries, reflection, and the null-forgiving ! operator can still smuggle nulls in. They catch the vast majority of cases early, which is huge, but you still write defensively at the edges.
?. and ?? do the same thing.
Different jobs. ?. guards a member access — "only call .City if the left side isn't null." ?? supplies a fallback — "use this value if the left side is null." They team up constantly: safely navigate with ?., then default with ??.
FunctionalIntermediateModule 09

Delegates, lambdas & events

In C#, functions are values you can store, pass, and return. A delegate is the type of such a value — a type-safe reference to a method with a particular signature. You rarely declare delegate types by hand anymore, because the library provides two generic families: Func<...,TResult> for functions that return a value, and Action<...> for ones that return void.

c#Func<int, int> square = x => x * x;        // a function value
Action<string>  log    = msg => Console.WriteLine(msg);

int nine = square(3);                       // call it like any method → 9

// pass behavior as an argument:
var evens = numbers.Where(n => n % 2 == 0);   // the lambda IS a delegate

The x => x * x is a lambda — an anonymous function written inline. Every lambda you passed to LINQ was really a delegate instance. Lambdas can capture variables from their surrounding scope (a closure), remembering them even after that scope returns — which is what makes them so expressive for callbacks and pipelines.

↳ Design Patterns
Passing a function as a parameter is the Strategy pattern from Design Patterns, made lightweight. Where classic OOP would define a strategy interface and a class per behavior, a C# lambda often expresses the same idea in a single line — the pattern dissolves into a language feature.

Built on delegates are events — the publisher/subscriber mechanism where an object notifies interested listeners that something happened (button.Click += OnClick;). The publisher exposes an event; subscribers attach handlers with +=; when the event is raised, every handler runs. It's the backbone of UI frameworks and message-driven code.

Lambdas replaced delegates; delegates are legacy.
A lambda is a delegate — the concise way to create one. The delegate type (Func, Action, or a custom one) is still what the lambda's value is. They're two halves of the same feature, not old versus new.
A closure copies the variables it captures.
It captures them by reference — the lambda sees later changes to the captured variable, and keeps it alive. Powerful, but a classic trap when capturing a loop variable: be deliberate about what a closure closes over.
AsyncIntermediateModule 10

Async & await

Most real work waits — on a database, a web request, a disk. Blocking a thread while it waits is wasteful, and in a server it caps how many requests you can serve. C#'s async/await lets you write code that looks sequential but releases the thread while waiting, picking up exactly where it left off when the result arrives. An async method returns a Task (a promise of future work); await unwraps it without blocking.

c#async Task<string> FetchUserAsync(int id)
{
    var response = await http.GetAsync($"/users/{id}");  // thread is free while waiting
    return await response.Content.ReadAsStringAsync();
}

The real payoff appears when work can overlap. Run three independent I/O calls sequentially and you wait for the sum of their times; start them together and await them as a group, and you wait only for the longest. Watch the difference:

Sequential vs overlapping
three I/O calls — one after another, or together
Sequential waits for the sum of the calls; awaiting them together waits for the longest one.
Async means multithreading — it runs my code in parallel.
Async is about not blocking while waiting, not about running your code on many cores. A single thread can juggle thousands of in-flight awaits. That's concurrency (managing many things at once), distinct from parallelism (doing many things at once on multiple cores).
It's fine to call .Result or .Wait() to get an async value.
Blocking on async code can deadlock and squanders the benefit. Go async all the way: await up the call chain. This matters enormously in ASP.NET, where blocking a request thread throttles the whole server.
IntermediateModule 11

Putting it together

You now have the working vocabulary of C#. The pieces aren't separate tricks — they compose into a single idiom. A typical feature defines a few types (often records for data, classes for behavior), stores them in generic collections, queries them with LINQ, guards the edges with null operators and exceptions, passes behavior around as lambdas, and does its I/O with async/await — all under a compiler that checks every join.

Code is organized into namespaces (logical groupings) across files, compiled into projects and assemblies. Two habits carry the most weight as programs grow: program to interfaces, so pieces can be swapped and tested in isolation, and let the types document intent, so a signature tells the reader (and the compiler) exactly what's expected. Both are the through-line of this whole course — say what you mean, and let the compiler hold you to it.

Where this leads — ASP.NET & Entity Framework

Everything here is the foundation for the rest of the track. ASP.NET builds web apps and APIs out of plain C# classes, wires them together by interface through dependency injection, and is async from top to bottom. Entity Framework maps your classes to database tables and lets you query them with the very same LINQ you just learned — your C# becomes SQL without you writing it. The fundamentals you built here are exactly what those frameworks expect you to bring.

The fastest way to grow from here is to build. Write small programs, lean on the compiler's errors as a teacher rather than an obstacle, and reach for the feature that says your intent most directly. When something won't compile, the type system isn't in your way — it's catching a mistake you'd otherwise ship.

The compiler has your back

From your first Console.WriteLine to LINQ pipelines and async I/O, one idea ran through all of it: C# lets you state your intent precisely, and checks it before your code ever runs. Types, generics, null-awareness, and the rest aren't ceremony — they're the language catching mistakes on your behalf.

You're now ready for the frameworks that made C# a mainstay of the backend. The same fluency you built here — types, objects, generics, LINQ, async — is what ASP.NET and Entity Framework are written in and built around. Next stop: the web.
