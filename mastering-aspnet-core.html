BeginnerModule 00

What is ASP.NET Core

ASP.NET Core is the modern, cross-platform framework for building web applications and HTTP APIs on .NET. It's a complete rewrite of the old Windows-only ASP.NET — fast, open-source, and equally at home on Linux, macOS, and Windows, in containers and the cloud. Whatever you build with it — a JSON API, a server-rendered site, a real-time hub — rests on one simple cycle: a client sends an HTTP request, your app produces an HTTP response.

A web app is, quite literally, a function from request to response. The client (a browser, a mobile app, another service) sends a request naming a method and a URL; your server runs some code and sends back a status code and a body. Everything else in this course — routing, middleware, dependency injection — is machinery that helps you turn requests into responses cleanly.

Here's a complete ASP.NET Core API. The builder wires up the app and its services; MapGet registers an endpoint; Run starts listening:

Program.csvar builder = WebApplication.CreateBuilder(args);
var app = builder.Build();

app.MapGet("/", () => "Hello, web!");              // GET /  → "Hello, web!"
app.MapGet("/products/{id}", (int id) =>          // id is bound from the URL
    Results.Ok(new { Id = id, Name = "Desk" }));   // returns 200 + JSON

app.Run();                                          // start the server (Kestrel)

That's a real, running web service — no configuration files, no IIS, no ceremony. The built-in Kestrel server listens for requests and hands each one to your app. From here, the course is about everything that happens between "a request arrives" and "a response leaves."

ASP.NET Core is just a new version of the old ASP.NET.
It's a ground-up rewrite with a different runtime, a new request pipeline, and built-in dependency injection. It dropped the Windows/IIS dependency entirely. The name carries over; the framework underneath is new.
You need IIS and a Windows server to run it.
ASP.NET Core ships its own cross-platform server, Kestrel, and runs anywhere .NET runs — Linux containers most commonly. IIS, Nginx, or a cloud load balancer can sit in front as a reverse proxy, but none is required.
HTTPBeginnerModule 01

HTTP & the request/response model

To build for the web you have to speak HTTP. A request has four parts: a method (the verb — what you want to do), a URL (which resource), headers (metadata like content type and auth), and an optional body (data you're sending). The response mirrors it: a status code (how it went), headers, and a body (the data coming back).

The methods map cleanly onto operations on resources — the heart of REST-style APIs. GET reads, POST creates, PUT replaces, DELETE removes. A well-designed API names resources with nouns (/products, /products/42) and uses the method to say what to do with them.

The status code is the response's headline, grouped into families by its first digit: 2xx success, 3xx redirection, 4xx "you (the client) made a mistake," and 5xx "I (the server) made a mistake." Picking the right one is part of an API's contract. Tap a scenario to see the code it should return:

Status codes
the response's one-glance verdict
Each scenario has a conventional status code — the first digit tells you the family at a glance.
GET vs POST is about how much data you're sending.
It's about semantics, not size. GET reads and should be safe and repeatable (no side effects); POST creates or causes a change. Caches, browsers, and proxies treat them very differently based on that meaning.
Status codes are cosmetic — returning 200 with an error message is fine.
The status code is a contract machines rely on. Clients, caches, and monitoring branch on it. Returning 200 OK for a failure (or 404 for a server bug) misleads every automated consumer of your API.
PipelineIntermediateModule 02

The middleware pipeline

Every request flows through a pipeline of middleware — small components chained in order, each one wrapping the next. A piece of middleware gets the request, can do work, and then either calls next() to pass control inward or short-circuits by producing a response itself. This is the central mechanism of ASP.NET Core: routing, authentication, logging, error handling — all are middleware.

The shape to hold in your head is an onion. The request travels inward through each layer to the endpoint at the center; the response then travels back outward through the same layers in reverse. So each middleware runs twice around a single await next() — once on the way in, once on the way out — which is how things like timing and exception handling wrap the entire downstream request.

Program.csapp.UseExceptionHandler(...);   // outermost — catches everything below it
app.UseAuthentication();        // who are you?
app.UseAuthorization();         // are you allowed?  (may short-circuit → 401/403)
app.MapControllers();           // innermost — the endpoint that makes the response

Because each layer wraps the ones after it, order is everything. Put authentication after your endpoints and it never gets to protect them. Send a request below and watch it travel down the pipeline to the endpoint and back up — then send one without an auth token and watch authorization short-circuit it before it ever reaches the endpoint:

The middleware pipeline
request flows in, response flows out — unless something short-circuits
▼ request in
▲ response out
Each layer runs on the way in and again on the way out. Authorization can stop a request early.
The order you register middleware doesn't really matter.
Order is critical. Middleware wraps everything registered after it, so authentication must come before authorization, both before endpoints, and exception handling near the top. A misordered pipeline silently fails to protect or wrap what it should.
Middleware only runs as the request comes in.
Each component runs code before await next() (inbound) and after it (outbound). That's how one middleware can start a timer on the way in and log the elapsed time on the way out, wrapping the entire rest of the pipeline.
RoutingBeginnerModule 03

Routing — URLs to endpoints

Routing is the matchmaker between an incoming URL and the code that should handle it. You register endpoints with route templates — patterns made of literal text and route parameters in braces. /products matches exactly; /products/{id} matches /products/42 and captures id = "42" for your handler to use.

c#app.MapGet("/products", ...);                 // literal
app.MapGet("/products/{id:int}", ...);          // {id} must be an int
app.MapGet("/products/{id:int}/reviews", ...);  // nested resource
app.MapGet("/categories/{name}", ...);         // string segment

A route constraint like {id:int} restricts what matches — /products/42 matches the int route, but /products/abc does not, so it falls through to be matched by something else or returns 404. Note that the query string (?q=desk) is not part of the route — it's bound separately as a parameter. Pick a URL and watch which registered route claims it, and what values it pulls out:

The route matcher
which template wins, and what it captures
The most specific matching template wins; route parameters are pulled out by name.
The query string is part of the route.
It isn't. Routing matches the path (/products/42); the query string (?sort=price) is bound separately as a parameter. Two requests with the same path but different queries hit the same route.
Routing is just plain string matching.
It's a small matching engine with constraints (:int, :guid), defaults, optional segments, and precedence — more specific templates beat catch-alls. It does real parsing and type-checking, not just string.StartsWith.
EndpointsIntermediateModule 04

Minimal APIs

Minimal APIs are the lightest way to define endpoints: a method, a route, and a lambda handler. They keep small services and microservices concise, with no controller classes or boilerplate, while still supporting routing, model binding, dependency injection, and everything else you'll meet in this course.

c#app.MapGet("/todos", () => store.All());            // list
app.MapGet("/todos/{id:int}", (int id) =>
    store.Find(id) is {} t ? Results.Ok(t)        // 200 + the todo
                       : Results.NotFound());   // 404 if missing
app.MapPost("/todos", (Todo todo) => {
    store.Add(todo);
    return Results.Created($"/todos/{todo.Id}", todo);  // 201 + Location
});

The handler's return value becomes the response. Results (and the strongly-typed TypedResults) is your toolbox for producing the right status code and body — Ok, NotFound, Created, BadRequest, NoContent, and more. Call the endpoints below against a live in-memory store and watch the data change and the responses come back:

list all todos
fetch one (404 if missing)
toggle done (uses id above)
remove (uses id above)
in-memory store
last response
Each call returns a status code and a body. POST adds, PUT toggles, DELETE removes — just like a real API.
Minimal APIs are toys; real apps need controllers.
They're fully featured — DI, model binding, validation, filters, auth, and OpenAPI all work. They're a great fit for focused services. Controllers add structure that helps in large apps, but "minimal" isn't "limited."
You can't organize minimal APIs, so they sprawl.
Route groups (app.MapGroup("/todos")) and putting handlers in separate methods/classes keep them tidy. You get organization without the ceremony of a controller per resource.
EndpointsIntermediateModule 05

Controllers & MVC

The other way to define endpoints is with controllers — classes that group related actions, the backbone of the MVC (Model-View-Controller) pattern. For APIs, you mark a class [ApiController], derive from ControllerBase, and write action methods that return results. Attribute routing on the class and methods builds the URLs.

ProductsController.cs[ApiController]
[Route("api/[controller]")]                 // → /api/products
public class ProductsController : ControllerBase
{
    [HttpGet("{id:int}")]                    // GET /api/products/42
    public ActionResult<Product> Get(int id) =>
        _store.Find(id) is {} p ? Ok(p) : NotFound();
}

The MVC pattern separates the Model (data), the View (presentation — for server-rendered HTML apps), and the Controller (request handling and coordination). For JSON APIs there's no view; the controller returns data that's serialized to the response. The [ApiController] attribute adds API-friendly behavior automatically — like returning 400 for invalid models before your code even runs.

↳ OOP & Design Patterns
A controller is just a class with methods, and MVC is a separation-of-concerns pattern from OOP & Coding Principles and Design Patterns — each piece has one responsibility. ASP.NET creates and wires controllers for you through dependency injection (Module 07), so they stay focused and testable.

Which to choose? Both are first-class and can coexist in one app. Reach for minimal APIs for small, focused services where brevity wins. Reach for controllers in larger applications that benefit from convention, grouping, filters, and a familiar structure across a big team. The request/response model underneath is identical.

MVC is outdated — everyone uses minimal APIs now.
Controllers and MVC are actively supported and widespread, especially in large codebases and server-rendered apps. Minimal APIs are newer and leaner, not a replacement. Many teams use both — controllers for the big surface, minimal for small endpoints.
The View in MVC means every app needs HTML templates.
Only server-rendered apps use Views. A JSON API skips the View entirely — the controller returns objects that get serialized. MVC's value for APIs is the Model/Controller split, not the templating.
InputIntermediateModule 06

Model binding & validation

Model binding is the bridge from the raw HTTP request to typed C# values. ASP.NET inspects your handler's parameters and pulls each one from the right place: route parameters from the URL, simple values from the query string, and complex objects from the JSON request body. You write a method that takes a Product; the framework deserializes the body into one for you.

c#record Product(
    [Required] string Name,                 // must be present & non-empty
    [Range(0.01, 10000)] decimal Price,    // must be in range
    [Range(0, 99999)] int Stock);

app.MapPost("/products", (Product p) => ...);  // p is bound from the JSON body

Binding pairs naturally with validation. Data-annotation attributes like [Required] and [Range] declare the rules; with [ApiController] (or an explicit check), ASP.NET validates the bound model and returns 400 Bad Request with the errors before your handler runs. Edit the incoming request below and watch it bind to the object — and watch validation react:

Request → object → validation

POST /products · JSON body
→
Product (bound)
Empty the name or set a negative price to see validation fail and return 400.
Model binding is just JSON deserialization.
JSON-to-object is only the body case. Binding also pulls from the route, query string, headers, and form data, choosing a source per parameter (overridable with [FromQuery], [FromBody], etc.). It assembles your parameters from the whole request.
Validation runs automatically on every endpoint.
You opt in. With [ApiController], controllers auto-return 400 on invalid models; minimal APIs need validation wired up explicitly. And validation only checks the rules you declare with attributes — no attributes, no checks.
DIIntermediateModule 07

Dependency injection

Dependency injection (DI) is built into the heart of ASP.NET Core. Instead of your classes creating the things they depend on, they declare them as constructor parameters, and the framework's container supplies them. You register services once at startup; everything that needs them just asks, and the container wires the graph together.

Program.csbuilder.Services.AddSingleton<IClock, SystemClock>();   // one for the whole app
builder.Services.AddScoped<IRepository, SqlRepository>();  // one per request
builder.Services.AddTransient<IEmailSender, Smtp>();        // new every time

// the container injects IRepository wherever it's needed:
app.MapGet("/products", (IRepository repo) => repo.All());
↳ OOP & Design Patterns
This is the Dependency Inversion Principle from OOP & Coding Principles, made the default. Endpoints depend on an IRepository interface, not a concrete database class — so you can swap implementations or inject a fake in tests. ASP.NET's container is the practical machine behind "depend on abstractions, not concretions."

The key decision is the lifetime: how long an instance lives and who shares it. Singleton — one instance for the whole application. Scoped — one per HTTP request (shared within that request, fresh for the next). Transient — a brand-new instance every time it's requested. Pick a lifetime and handle two requests to see exactly which instances get shared:

Service lifetimes
who shares an instance, and for how long
Each request resolves the service twice. Watch how many distinct instances appear.
Dependency injection is a third-party library you install.
In ASP.NET Core it's built in — the container ships with the framework and the whole stack uses it. DI is fundamentally a pattern (pass dependencies in), with the container as the tool that automates wiring. You can swap containers, but you rarely need to.
Just make everything a singleton — it's the most efficient.
A captive dependency bug lurks there: inject a scoped service (like a DB context) into a singleton and it gets frozen for the app's lifetime, shared unsafely across requests. Match lifetimes to purpose — per-request state is scoped, not singleton.
OutputIntermediateModule 08

Returning data & status codes

The final step is the response. An endpoint returns a result, and ASP.NET turns it into an HTTP response: a status code, headers, and a body. Return a plain object and it's serialized to JSON with a 200; return a Results value and you control the code, headers, and body precisely. The strongly-typed TypedResults variants also document exactly what an endpoint can return.

c#return Results.Ok(product);                       // 200 + JSON body
return Results.Created($"/products/{id}", product);   // 201 + Location header
return Results.NotFound();                        // 404, no body
return Results.BadRequest(new { error = "Invalid" }); // 400 + JSON
return Results.NoContent();                       // 204, nothing to return

Pick a result and see the exact HTTP response it produces — status line, headers, and body:

The response builder
a Results value → an HTTP response
Each Results helper sets the status code, the right headers, and serializes the body for you.

For errors, ASP.NET favors a standard shape called Problem Details (Results.Problem(...)) — a consistent JSON structure with a status, title, and detail that clients can parse uniformly, rather than every endpoint inventing its own error format. JSON serialization itself is handled by System.Text.Json and is fully configurable (naming, casing, converters).

Just return 200 for everything and put the real status in the body.
The HTTP status code is the machine-readable result. A created resource is 201, a missing one 404, a bad input 400. Clients, caches, and tooling depend on it — burying the truth in a 200 body breaks all of them.
Returning an object and returning Results.Ok(object) are different things.
For the success case they're equivalent — both yield 200 + JSON. Results/TypedResults matter when you need a different status, headers, or to express multiple possible outcomes from one endpoint clearly.
ProductionIntermediateModule 09

Configuration & options

Real apps need settings that differ between your laptop and production — connection strings, API keys, feature flags. ASP.NET Core's configuration system reads from many layered providers and merges them, with later sources overriding earlier ones: appsettings.json, then appsettings.{Environment}.json, then environment variables, then command-line args. Your code reads the merged result through IConfiguration without caring where each value came from.

appsettings.json{
  "ConnectionStrings": { "Db": "Server=...;Database=shop" },
  "Cache": { "DurationSeconds": 60 }
}

Rather than reading loose strings everywhere, the options pattern binds a configuration section to a strongly-typed class you can inject. It's configuration as a real C# object, validated and discoverable:

c#public class CacheOptions { public int DurationSeconds { get; set; } }

builder.Services.Configure<CacheOptions>(
    builder.Configuration.GetSection("Cache"));        // bind the section

// inject it anywhere:
app.MapGet("/x", (IOptions<CacheOptions> opt) => opt.Value.DurationSeconds);

The active environment (Development, Staging, Production) is itself a setting that switches behavior — detailed error pages in Development, terse ones in Production. ASP.NET reads it from the ASPNETCORE_ENVIRONMENT variable, and you branch on app.Environment.IsDevelopment().

Configuration means the appsettings.json file.
That file is one provider among many. Environment variables, command-line args, user secrets, and cloud key vaults all feed the same IConfiguration, layered so production can override file defaults without changing code.
Secrets like passwords belong in appsettings.json.
Never commit secrets to source. Use user secrets in development and environment variables or a key vault in production. The layering exists precisely so secrets stay out of your repo while still reaching IConfiguration.
ProductionIntermediateModule 10

Errors, logging & custom middleware

Things go wrong, and a production API must fail gracefully. Rather than wrapping every endpoint in try/catch, you handle exceptions once, at the top of the pipeline, with exception-handling middleware. It catches anything thrown downstream and turns it into a clean response — typically Problem Details — while hiding stack traces from clients in production.

Program.csif (app.Environment.IsDevelopment())
    app.UseDeveloperExceptionPage();   // rich diagnostics — dev only
else
    app.UseExceptionHandler();        // safe Problem Details — production

For visibility, ASP.NET has built-in logging. Inject ILogger<T> and write structured log entries at appropriate levels (Trace, Debug, Information, Warning, Error, Critical). Logs flow to configurable providers — console, files, or cloud platforms — and structured logging keeps the data queryable, not just human-readable.

c#app.MapGet("/orders/{id}", (int id, ILogger<Program> log) => {
    log.LogInformation("Fetching order {OrderId}", id);  // structured
    ...
});

When you need your own cross-cutting behavior — request timing, custom headers, rate limiting — you write custom middleware: a component that receives the request and the next delegate, does work around the call, and slots into the pipeline. It's the same onion layer you visualized in Module 02, now authored by you.

c#app.Use(async (context, next) => {
    var sw = Stopwatch.StartNew();
    await next(context);                 // run the rest of the pipeline
    log.LogInformation("Took {Ms}ms", sw.ElapsedMilliseconds);  // runs on the way out
});
Wrap every endpoint in try/catch to handle errors.
Handle the unexpected once, with exception middleware at the top of the pipeline — it catches everything below. Reserve local try/catch for errors you can specifically recover from right there, not as a blanket net.
Console.WriteLine is fine for logging.
Use ILogger. It gives you levels, structured data, and pluggable destinations (files, cloud, dashboards), and it can be filtered and turned down in production. Console.WriteLine has none of that and vanishes outside a console.
IntermediateModule 11

Putting it together

Step back and trace one request through everything you've learned. A client sends GET /api/products/42, and it flows through the whole machine you've now seen piece by piece:

1. Kestrel accepts the request and hands it to the pipeline
2. Middleware runs inbound: exception handler, then auth — which may short-circuit
3. Routing matches /api/products/{id:int} and captures id = 42
4. Model binding turns "42" into an int parameter
5. The endpoint runs, with an IRepository injected by the container
6. It returns Results.Ok(product) → a 200 response with a JSON body
7. The response flows back out through the middleware and to the client

Every feature in this course is one station on that journey, and they compose into a clean architecture: thin endpoints that bind and validate input, services (registered with DI) that hold the real logic, models that carry data, and middleware for the concerns that cut across everything. Keep endpoints small, push logic into injected services, and let the pipeline handle the cross-cutting work.

Where Entity Framework fits

The one big piece still abstracted away above is IRepository — the data layer. That's exactly where Entity Framework Core comes in: it maps your C# models to database tables and lets you query them with the same LINQ from the C# course. You register a DbContext as a scoped service (one per request — now you know why), inject it into your endpoints, and your .Where(...).Select(...) queries become SQL. ASP.NET handles the request; EF handles the data.

You came in knowing C#; you can now turn that into a real, running web API — routed, validated, dependency-injected, and returning the right responses. The framework's job, and yours, was always the same simple contract: a request goes in, a response comes out.

Request in, response out

From your first MapGet to a full pipeline with routing, binding, dependency injection, and error handling, one idea carried all of it: ASP.NET Core is a machine for turning HTTP requests into HTTP responses, and every feature is a station along that path. Master the flow and the framework stops being a mystery box.

The last piece is data. Next in the track, Entity Framework Core connects these APIs to a real database — your models become tables, your LINQ becomes SQL, and the C# → ASP.NET → EF journey is complete.
