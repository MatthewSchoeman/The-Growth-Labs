BeginnerModule 00

Thinking in questions

There's a moment in every developer's life when SQL stops being "the thing that fetches rows for the app" and becomes an instrument. It happens the first time someone asks a real question — which links are actually winning? did last week's campaign work? — and you realize the answer is already sitting in your tables, waiting for a query precise enough to extract it. This course is about that shift: from fetching to asking.

Our dataset is one you built. Snip, the URL shortener from the capstone, has grown up: it now records every click as an event. Three tables carry everything this course does:

schema.sql 
-- Snip's analytics schema (the capstone app, grown up)
CREATE TABLE links (
  id         bigserial PRIMARY KEY,
  code       text UNIQUE NOT NULL,      -- '4C92', 'launch', …
  url        text NOT NULL,
  user_id    bigint REFERENCES users(id),
  meta       jsonb DEFAULT '{}',          -- Module 07
  created_at timestamptz DEFAULT now()
);

CREATE TABLE clicks (
  id         bigserial PRIMARY KEY,
  link_id    bigint REFERENCES links(id),
  clicked_at timestamptz NOT NULL,
  country    text, device text, referrer text
);   -- 48,112 links · 1,204,392 clicks and counting

Every module poses a question a product person would genuinely ask of this data, then builds the query that answers it. The techniques stack: by Module 11 you'll write one query that joins, filters, pivots, and ranks in a single pass — the analytics dashboard for your own app.

↳ SQL & PostgreSQL · The Capstone
The prequel taught the mechanics — SELECT, WHERE, keys, the B-tree index — and the capstone gave Snip its links table. This course assumes both and builds upward: same database, harder questions.
Advanced SQL means memorizing obscure syntax.
It means composing simple pieces — join, filter, group, window — into precise questions. Every query in this course is built from parts you can already read; the skill is arranging them to say exactly what you mean.
Pull the rows into application code and compute there.
The database is a query engine sitting next to the data, with indexes, statistics, and forty years of set-processing optimizations. Shipping a million rows to C# to count them in a loop is paying network and memory for work Postgres does in one indexed pass. Push computation down.
JoinsBeginnerModule 01

Joins as questions

You've written JOIN before. The upgrade is realizing that each join type answers a different question. INNER asks "which clicks have a link?" — only matches survive. LEFT asks "tell me about every link, clicked or not" — unmatched links survive with NULLs on the click side. RIGHT flips it, and FULL keeps the unmatched from both sides — the reconciliation join.

Those NULLs aren't noise; they're often the entire point. The classic anti-join — "which links have never been clicked?" — is a LEFT JOIN that keeps only the rows where the match failed:

unclicked.sql 
-- every link with its click count — even the unclicked ones
SELECT l.code, COUNT(c.id) AS clicks
FROM links l
LEFT JOIN clicks c ON c.link_id = l.id
GROUP BY l.code;

-- the anti-join: links NEVER clicked
SELECT l.code FROM links l
LEFT JOIN clicks c ON c.link_id = l.id
WHERE c.id IS NULL;

Try each join on a miniature of Snip's data — four links (one never clicked) and five clicks (one pointing at a deleted link):

result
INNER keeps only matches. The unmatched rows the other joins preserve — with NULLs — are where the interesting questions live.
Prefer INNER JOIN — the outer joins are slower.
Choose the join by the question, not superstition. The planner executes them with the same machinery, and an outer join it can prove unnecessary gets simplified away. A fast query that drops the rows you needed is just a wrong answer, quickly.
NULLs in a join result mean something went wrong.
They mean "no match" — which is frequently the answer itself. Unclicked links, users without orders, orphaned events: the anti-join pattern (WHERE … IS NULL) turns the gap into the result set.
AggregationBeginnerModule 02

Aggregate & group

Analytics is mostly counting — but counting into buckets. GROUP BY folds rows that share a value into one group per value; aggregates like COUNT, SUM, AVG, MIN, MAX then compute one number per group. The subtlety everyone trips on is when filters apply: WHERE prunes rows before grouping; HAVING prunes whole groups after the aggregates exist.

by_country.sql 
SELECT country, COUNT(*) AS clicks
FROM clicks
WHERE clicked_at >= now() - interval '7 days'   -- rows, before grouping
GROUP BY country
HAVING COUNT(*) >= 100                            -- groups, after counting
ORDER BY clicks DESC;

Group the same twelve clicks by different dimensions, then raise the HAVING bar and watch small buckets get cut:

Buckets from rows
GROUP BY a dimension · HAVING a threshold
country device link
HAVING COUNT(*) ≥1(12 clicks total)
GROUP BY folds 12 rows into a handful of buckets; HAVING keeps only the buckets that clear the bar.
↳ Big O
Under the hood this is a hash aggregate: one pass over the rows, bucketing by hash key — O(n), just like the frequency-counting patterns from Big O and Data Structures. The database is running the same algorithms you learned, at scale.
HAVING is just another way to spell WHERE.
They run at different stages. WHERE filters rows before grouping (and can use indexes); HAVING filters after aggregation, so it's the only place a condition like COUNT(*) >= 100 can live. Put row conditions in WHERE — filtering early is cheaper.
COUNT(*), COUNT(col), COUNT(DISTINCT col) — same thing.
COUNT(*) counts rows; COUNT(col) skips NULLs in that column; COUNT(DISTINCT col) counts unique values. In Module 01's LEFT JOIN, COUNT(c.id) correctly gives unclicked links a 0 — COUNT(*) would have said 1.
SubqueriesIntermediateModule 03

Subqueries

Some questions have a question inside them. "Which links beat the average?" first needs the average. A scalar subquery computes that single value in place. "Which users were active this week?" is a membership test — IN or, better for pure existence, EXISTS, which probes for a match per outer row and stops at the first hit. When the inner query references the outer row, it's correlated — the inner question is asked about each candidate.

nested.sql 
-- scalar: links beating the overall average
SELECT code, clicks FROM link_stats
WHERE clicks > (SELECT avg(clicks) FROM link_stats);

-- EXISTS: users with a click this week (a semi-join)
SELECT u.name FROM users u
WHERE EXISTS (
  SELECT 1 FROM clicks c
  JOIN links l ON l.id = c.link_id
  WHERE l.user_id = u.id                -- ← correlated
    AND c.clicked_at >= now() - interval '7 days');

That link_stats relation isn't a table — it's a view we'll define in Module 11; subqueries compose with anything that produces rows. And a nested question that gets deep or repeated is your cue for the next module's tool: name the step with a CTE.

Correlated subqueries rerun once per outer row — always slow.
That's the mental model, not necessarily the execution. The planner routinely rewrites EXISTS/IN into hash semi-joins that scan each side once. Write the clearest form, then check what actually runs with EXPLAIN (Module 09).
EXISTS should select real columns to be meaningful.
EXISTS tests presence, nothing else — the select list is never evaluated, which is why SELECT 1 is idiomatic. It also short-circuits on the first match, making it the natural semi-join: "is there at least one?"
CTEsIntermediateModule 04

CTEs & recursion

A common table expression — the WITH clause — gives a subquery a name, turning a nested one-liner into a readable pipeline: stage one produces rows, stage two refines them, the final SELECT reads like a sentence. But CTEs hide a superpower: WITH RECURSIVE lets a query reference itself, iterating until no new rows appear — the fixpoint. That's how SQL walks chains, trees, and graphs.

Snip has a chain to walk: some short links point at other short links. Resolving promo means following hops until the URL leaves Snip:

resolve_chain.sql 
WITH RECURSIVE chain AS (
  SELECT code, url, 0 AS depth
  FROM links WHERE code = 'promo'          -- base case
  UNION ALL
  SELECT l.code, l.url, chain.depth + 1     -- recursive step
  FROM chain
  JOIN links l ON chain.url = 'snip.ly/' || l.code
)
SELECT * FROM chain;

Run the recursion one iteration at a time and watch the working set grow until it can't:

Recursion to a fixpoint
base case · UNION ALL · repeat until empty
Each iteration joins the newest rows against links again. When a pass adds nothing, the recursion is complete.
↳ Data Structures
This is graph traversal — the breadth-first walk from Data Structures, expressed declaratively. Each iteration is one BFS level; depth is the distance from the start node. Org charts, category trees, dependency graphs: same shape, same query.
CTEs are just formatting — a style choice.
Readable staging is reason enough, but CTEs also enable recursion and let one result feed several later stages by name. They're how a 50-line query stays maintainable — each stage testable by selecting from it alone.
A CTE is an optimization fence — Postgres always materializes it.
That was true historically. Since PostgreSQL 12, single-use CTEs are inlined into the outer query unless you write MATERIALIZED to force the fence. Write CTEs for clarity; the planner sees through them.
WindowsIntermediateModule 05

Window functions

Here is the single biggest level-up in analytical SQL. GROUP BY answers questions by collapsing rows — you get the total, but lose the rows. Window functions keep every row and add context alongside it: this row's rank, its running total, the value from the row before. The magic phrase is OVER (PARTITION BY … ORDER BY …) — "compute across this window of related rows, but don't collapse them."

leaderboard.sql 
-- a leaderboard with honest ties…
SELECT code, clicks,
       RANK() OVER (ORDER BY clicks DESC) AS place
FROM link_stats;

-- …and a running total per link, rows intact
SELECT day, code, clicks,
       SUM(clicks) OVER (PARTITION BY code ORDER BY day) AS so_far
FROM daily_clicks;

Below are two links' daily clicks. Apply each function and read the highlighted column — notice when partitions matter, how ties behave, and where NULL appears:

One table, four windows
rows stay · context appears
ROW_NUMBER RANK running SUM LAG
ROW_NUMBER restarts at 1 inside each partition — a per-link sequence, with every original row still present.
↳ EF Core
This is where LINQ runs out of road — EF Core has no natural expression for OVER (PARTITION BY …). The professional move is FromSql: let the database compute ranks and running totals in one pass instead of dragging rows into memory to loop over them.
To aggregate you must GROUP BY — and lose the rows.
Window aggregates like SUM(...) OVER (...) give you the aggregate and the rows. "Each day's clicks next to the running total" is one query, no self-join, no app-side loop.
ROW_NUMBER, RANK, DENSE_RANK — interchangeable.
Ties expose the difference: RANK gives equals the same place then skips (1,2,2,4); DENSE_RANK doesn't skip (1,2,2,3); ROW_NUMBER breaks ties arbitrarily unless you add a tiebreaker to ORDER BY — fine for pagination, wrong for a leaderboard.
Set opsIntermediateModule 06

Set ops & pivots

Sometimes the question compares two result sets: this week's active links vs last week's. SQL treats results as sets: UNION combines them (deduplicating — UNION ALL keeps duplicates), INTERSECT keeps what's in both, EXCEPT keeps what's only in the first. "Links clicked both weeks" is retention; "clicked this week but not last" is your new arrivals — each one operator:

Two weeks, three questions
UNION · INTERSECT · EXCEPT
UNION INTERSECT EXCEPT
clicked this week
clicked last week
result
Amber chips appear in both weeks. Each operator asks the two sets a different question.

The other classic reshaping is the pivot — turning values into columns. Postgres makes conditional aggregation elegant with the FILTER clause:

pivot.sql 
SELECT l.code,
  COUNT(*) FILTER (WHERE c.device = 'mobile')  AS mobile,
  COUNT(*) FILTER (WHERE c.device = 'desktop') AS desktop,
  COUNT(*) FILTER (WHERE c.device = 'tablet')  AS tablet
FROM links l JOIN clicks c ON c.link_id = l.id
GROUP BY l.code;   -- one row per link, one column per device
UNION and UNION ALL are interchangeable.
UNION deduplicates — which costs a sort or hash over the whole result and silently merges rows you may have wanted. Default to UNION ALL; reach for UNION only when distinctness is the actual requirement.
Cross-tabs need a special pivot extension.
COUNT(*) FILTER (WHERE …) — or the portable CASE WHEN inside any aggregate — pivots with plain SQL. One pass over the data, one column per condition, no extension required.
JSONBAdvancedModule 07

JSONB & arrays

Some data refuses a fixed shape. Snip's meta column holds per-link extras — UTM parameters, tags, feature toggles — that vary link to link. Postgres handles this natively with jsonb: a binary, indexable JSON type with its own operator family. -> extracts JSON, ->> extracts text, #>> follows a path, and @> asks "does this document contain that fragment?" — the operator a GIN index accelerates.

One document, four operators
extract · path · contain
-> ->> #>> @>
-> returns jsonb (still queryable); ->> returns text (comparable, indexable with B-tree expression indexes).
meta.sql 
-- paid-tagged links and where their traffic came from
SELECT code, meta#>>'{utm,source}' AS source
FROM links
WHERE meta @> '{"tags": ["paid"]}';

CREATE INDEX links_meta_gin ON links USING gin (meta);
-- arrays get the same treatment: unnest(), ANY(), && overlap
↳ SQL & PostgreSQL
The prequel taught you to normalize — and that's still the default. jsonb is the pressure valve for the genuinely variable edge of your data, not a replacement for columns you filter, join, or constrain every day. Core relational, edges flexible.
With JSONB you can skip schema design.
A jsonb column is a schema decision — one that trades constraints, foreign keys, and column statistics for flexibility. Use it where shape truly varies (metadata, settings, payloads); keep identity and relationships in real columns.
Queries inside JSON can't use indexes.
A GIN index makes containment (@>) and key-existence fast across millions of documents, and a B-tree expression index on (meta->>'campaign') serves equality like any column. Document data, indexed access.
SearchAdvancedModule 08

Searching text

"Find the link about the launch" is a text problem, and LIKE '%launch%' is the trap answer: a leading wildcard can't use a B-tree, so it scans everything. Postgres offers two real tools. For fuzzy, typo-tolerant matching on short strings — codes, names — the pg_trgm extension indexes trigrams (three-character shingles) with GIN, making similarity and even %launch% fast. For documents, full-text search parses text into a tsvector of stemmed lexemes ("launching" → "launch"), matches it against a tsquery, and ranks results.

search.sql 
-- fuzzy: typo-tolerant match on codes
CREATE EXTENSION pg_trgm;
SELECT code FROM links WHERE code % 'lanch';      -- similar to 'launch'

-- full-text: stemmed, ranked search over page titles
SELECT url, ts_rank(tsv, q) AS rank
FROM links, websearch_to_tsquery('launch week') q
WHERE tsv @@ q
ORDER BY rank DESC;   -- tsv: a generated tsvector column, GIN-indexed

The pattern to remember: store a generated tsvector column, GIN-index it, query with websearch_to_tsquery (it understands quoted phrases and -exclusions the way users type them), and order by ts_rank. Stemming, stop-words, ranking, and highlighting — inside the database you already run, transactionally consistent with the data it searches.

LIKE '%term%' is fine — it's the simple option.
Simple to write, brutal to run: the leading wildcard forces a full scan of every row. At Snip's 48k links it's sloppy; at millions it's an outage. A trigram GIN index makes the same pattern an index lookup — same query, different plan.
Real search needs Elasticsearch from day one.
Postgres FTS covers an enormous middle: stemming, multi-language configs, ranking, phrase search, highlighting — with no sync pipeline and no second system to operate. Reach for a dedicated engine when you need its extras, not by default.
PerformanceAdvancedModule 09

EXPLAIN & plans

Every query you write is a description of a result, not a procedure — the planner decides how to actually compute it, and EXPLAIN shows you its plan. Add ANALYZE and Postgres runs the query, reporting actual times and row counts next to its estimates. Reading plans is the skill that separates "the query is slow" from "the query is scanning 1.2 million rows to return 52 — it needs an index."

explain.sql 
EXPLAIN ANALYZE
SELECT * FROM clicks WHERE link_id = 42;

-- the fix, when the plan says Seq Scan:
CREATE INDEX idx_clicks_link_id ON clicks (link_id);

Here's that exact query against the 1.2-million-row clicks table. Create the index and watch the plan — and the work — change shape:

Seq Scan vs Index Scan, measured
rows touched to answer
Without an index the only strategy is reading everything. The B-tree turns "scan 1.2M rows" into "walk to the 52 that match."
↳ Big O · SQL & PostgreSQL
This is Big O made visible: the Seq Scan is O(n), the Index Scan is O(log n) + matches — the same B-tree you met in the prequel's index module. EXPLAIN is how you catch the O(n) hiding in production before your users do.
Slow query? Add indexes until it's fast.
Every index taxes every write and competes for cache. Index the access paths your queries actually take — EXPLAIN tells you which — and remember the planner will rightly ignore an index when scanning is cheaper (tiny tables, low-selectivity filters).
The cost numbers are milliseconds.
Cost is in arbitrary planner units — useful for comparing plans, meaningless as wall-clock. For truth, run EXPLAIN (ANALYZE, BUFFERS) and read actual times, actual rows, and pages hit. Estimates wildly off from actuals usually mean stale statistics: run ANALYZE.
ConcurrencyExpertModule 10

Transactions & isolation

Production databases are concurrent: dozens of sessions reading and writing the same rows at once. Transactions make a group of statements atomic — all or nothing — and MVCC (multi-version concurrency control) lets readers and writers avoid blocking each other by giving each transaction a consistent snapshot. But the default READ COMMITTED isolation still permits a classic bug: the lost update. Two sessions read the same value, both compute a new one, and the second write silently erases the first.

Race two sessions incrementing the same click counter — then fix it with a row lock:

Two sessions, one row
the lost update, and the lock that prevents it
Session A
Session B
Both sessions read 10, both write 11 — one increment vanishes. Concurrency bugs don't throw exceptions; they lose data quietly.
safe_increment.sql 
-- ① lock the row you're about to change
BEGIN;
SELECT clicks FROM link_stats WHERE id = 42 FOR UPDATE;
UPDATE link_stats SET clicks = 11 WHERE id = 42;
COMMIT;

-- ② better: make the write itself atomic — no read step at all
UPDATE link_stats SET clicks = clicks + 1 WHERE id = 42;

Stronger isolation levels are the third tool: REPEATABLE READ freezes your snapshot for the whole transaction, and SERIALIZABLE guarantees the outcome equals some serial order — aborting one transaction with a serialization error when it can't, which your code retries. Locks, atomic statements, or serializable-plus-retry: pick per operation, and keep transactions short either way.

↳ ASP.NET Core
Where does concurrency come from? Every API request is a session. Two users tapping the same Snip link hit this exact race on the click counter. The scoped DbContext from ASP.NET Core gives each request one unit of work — these tools decide what happens when those units collide.
Transactions are only for multi-table writes.
They're about invariants under concurrency. Even a single-row read-modify-write races with itself across sessions, as the lab shows. If correctness depends on what you just read still being true when you write — that's a transaction (or an atomic statement).
SERIALIZABLE is academically nice but too slow for real apps.
Postgres implements it optimistically (SSI): no extra blocking, just aborts on genuine conflict, which you retry. For workflows where a lost update means lost money, a rare retry is vastly cheaper than the bug — measure before assuming you can't afford it.
ExpertModule 11

Views & the dashboard

Good queries deserve names. A view stores a query as a virtual table — always fresh, computed on read — which is how link_stats has been following us around since Module 03. When a view is expensive and its consumers can tolerate slight staleness, a materialized view caches the results on disk, refreshed on your schedule — analytics dashboards' best friend:

views.sql 
CREATE VIEW link_stats AS
  SELECT l.id, l.code, COUNT(c.id) AS clicks
  FROM links l LEFT JOIN clicks c ON c.link_id = l.id
  GROUP BY l.id, l.code;              -- always current

CREATE MATERIALIZED VIEW weekly_top AS -- …expensive query…;
REFRESH MATERIALIZED VIEW CONCURRENTLY weekly_top;  -- cron, no read downtime

And now the payoff. Everything this course taught, in one query — the dashboard for your own app. A CTE stages the week's numbers, a join connects links to their clicks, FILTER pivots devices into columns, an aggregate counts countries, and a window function hands out places:

dashboard.sql 
WITH weekly AS (
  SELECT l.code,
         COUNT(*)                                    AS clicks,
         COUNT(*) FILTER (WHERE c.device = 'mobile') AS mobile,
         COUNT(DISTINCT c.country)                   AS countries
  FROM links l
  JOIN clicks c ON c.link_id = l.id
  WHERE c.clicked_at >= now() - interval '7 days'
  GROUP BY l.code
)
SELECT code, clicks, mobile, countries,
       RANK() OVER (ORDER BY clicks DESC) AS place
FROM weekly
ORDER BY place
LIMIT 10;
The query is the question

Ten modules ago that query would have been a page of application code: fetch everything, loop, count into dictionaries, sort, slice. Now it's one declarative statement the planner executes in a single indexed pass — joins for connection, WHERE for scope, GROUP BY and FILTER for shape, a window for context, a view to give it a name. Learning advanced SQL isn't learning more syntax; it's learning to say precisely what you want to know and let the database do what it was built for.
