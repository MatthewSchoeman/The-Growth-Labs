AdvancedModule 00

The second layer

Volume I went underneath the syntax to the runtime: where data lives, how await releases a thread, how the JIT specializes generics, how the GC bets on youth. Volume II sits one layer up — the expert language features — and its unifying secret is this: most of them are the compiler rewriting your code. You write a convenient surface; Roslyn emits the machinery that runs on the runtime you now understand. Learn to see the rewrite and these features stop being magic and start being predictable.

The pattern repeats across the whole volume. yield return is rewritten into a state machine implementing IEnumerator — the same trick as async, but for sequences. A lambda that captures a variable is rewritten into a heap-allocated display class holding that variable. A record is rewritten with a full value-equality implementation and a with clone. A source generator runs during compilation and writes entire files. Even nullable reference types are a compile-time flow analysis with no runtime footprint at all. Seeing the emitted machine is the through-line.

Ref semanticsAdvancedModule 01

ref returns, ref locals & in

C# has safe references without pointers. A ref local is an alias to an existing storage location — an array element, a field — and writing through it mutates the original, not a copy. A ref return hands a caller a writable alias into your data. And in passes an argument by readonly reference, avoiding the copy that a large struct would otherwise incur on every call. Together they give you pointer-like efficiency with full memory safety — no dangling, no arithmetic, checked by the compiler.

ref-aliasing.cs 
int[] hits = { 10, 20, 30 };

ref int slot = ref hits[1];   // alias to the element, not a copy
slot = 99;                    // hits[1] is now 99 — mutated in place

int copy = hits[1];         // a value copy
copy = 7;                     // hits[1] unchanged — copy is independent

Process(in bigStruct);        // readonly reference — no copy of the fields

The array below is Snip's per-region hit counts. Take a ref local to hits[1] and write through it — the array element changes. Take a value copy and write to that — the array is untouched:

An alias, not a copy
ref local · value copy · in-parameter
int[] hits · region counters
hits[1] (the aliased slot)20
value copies made0
Writing through the ref local mutates hits[1] in place; writing to a value copy leaves the array alone. The in-parameter passes a big struct by readonly reference — zero field copies.
↳ Advanced C# · Vol. I (Structs) · Bits
This is the value-vs-reference split from Vol. I, made explicit and controllable: a ref local gives you the aliasing of a class with the inline storage of a struct. It's the pointer-plus-length idea from Bits — an alias to memory — but bounds-checked and lifetime-verified by the compiler.
C# can't do references without unsafe pointers.
ref locals, ref returns, and in are safe references — the compiler enforces that they never outlive what they point to. You get in-place mutation and copy-free passing with none of the danger of raw pointers.
in just documents that a parameter won't change.
It changes the calling convention — the argument is passed by readonly reference instead of copied. For a large readonly struct on a hot path, that eliminates a full field-by-field copy per call. It's a performance tool, not just an annotation.
Ref semanticsAdvancedModule 02

ref struct & the stack-only rule

How does Span<T> guarantee it never dangles? It's a ref struct — a type the compiler forces to live only on the stack. Because a ref struct may hold a reference into stack memory or another object's interior, letting it escape to the heap would create a dangling reference. So the compiler forbids the escapes: you can't box it, can't make it a field of a class, can't capture it in a lambda or an async/iterator local, can't put it in an array. What's left — locals, parameters, returns — is exactly the set of uses that stay on the stack. The safety of Span is these rules.

stack-only.cs 
ref struct Cursor { public ReadOnlySpan<char> Rest; }

Cursor c = default;        // ✓ local — stays on the stack
object boxed = c;          // ✗ CS0029/CS8345 — cannot box a ref struct
class Holder { Cursor f; }  // ✗ CS8345 — cannot be a field of a class
async Task Read(Cursor c){}   // ✗ CS4012 — cannot live across an await

Locals, parameters, and ref-safe returns keep the value on the stack — allowed. Boxing, class fields, and async/iterator locals would let it reach the heap — rejected. Those rules are exactly why Span can't dangle.
↳ Advanced C# · Vol. I (Span<T>)
The zero-allocation Span you sliced in Vol. I is a ref struct — and now you know why it can't be stored in a field or awaited across. The constraint that felt arbitrary is the mechanism: stack-only lifetime is what makes a window into borrowed memory safe.
The restrictions on Span are annoying compiler pedantry.
They're the safety proof. A ref struct can reference stack memory or an object's interior; if it escaped to the heap it could outlive that memory and dangle. Forbidding the escapes is how you get a borrowed-memory view with zero risk — the rules are the feature.
If I need a Span in an async method, I'm just stuck.
You restructure: slice the span in a synchronous helper, or carry a heap-safe Memory<T> across the await and convert to a span only where you use it. The constraint pushes you toward code that's correct about lifetimes rather than accidentally wrong.
State machinesAdvancedModule 03

Iterators & yield

Here's the insight that unifies half of C#: yield return compiles to a state machine — the exact same mechanism as async, but producing a sequence instead of a result. Calling an iterator method runs none of its body; it returns an object implementing IEnumerator<T>. Each call to MoveNext() runs the body until the next yield return, hands back that value, and suspends, remembering exactly where it paused. This is why iterators are lazy, why LINQ is lazy (it's built on them), and why you can iterate an infinite sequence — nothing is computed until it's pulled.

iterator.cs 
IEnumerable<int> Numbers() {
    yield return 1;   // state 1 — produce, then suspend here
    yield return 2;   // state 2
    yield return 3;   // state 3
}   // calling Numbers() runs NOTHING — it returns an enumerator; MoveNext() drives it

Call the iterator (watch zero body code run), then step MoveNext() and see it advance to each yield, produce a value, and suspend — one state at a time:

produced: —
Calling the method executes zero lines — it just builds the enumerator. Each MoveNext runs to the next yield and suspends. That laziness is the same state-machine trick as async, applied to sequences.
↳ Advanced C# · Vol. I (async · LINQ)
Same machine, two costumes: the async state machine suspended at await; this one suspends at yield. And the LINQ deferred pipeline you counted in Vol. I is built on iterators — Where and Select are themselves yield methods, which is exactly why pulling one element runs one pass.
An iterator method runs and collects its results when you call it.
It runs nothing on the call — it returns an enumerator. The body executes incrementally, one yield per MoveNext, and only while something enumerates. That's why an iterator can be infinite and why side effects in it are deferred until iteration.
yield is just a shortcut for building and returning a list.
It's the opposite — a lazy, streaming producer that never materializes the whole sequence. It holds one element's worth of state and computes the next only on demand, which is what makes streaming huge or unbounded sequences possible without memory blowup.
ClosuresAdvancedModule 04

Closures & captured variables

When a lambda uses a local variable from its enclosing scope, it captures that variable — and the crucial word is variable, not value. The compiler hoists the captured variable into a heap-allocated display class shared between the outer method and the lambda; both see the same cell. This produces the most famous gotcha in C#: capture a loop variable and every closure shares one cell, so they all observe its final value. Capture a fresh per-iteration copy and each closure gets its own. The behavior falls straight out of what the compiler hoisted.

capture.cs 
var fns = new List<Func<int>>();
for (int i = 0; i < 3; i++)
    fns.Add(() => i);      // captures the variable i — one shared cell
// fns[0]() == fns[1]() == fns[2]() == 3   (all see the final i)

for (int i = 0; i < 3; i++)
    { int copy = i; fns.Add(() => copy); }  // fresh cell each iteration → 0,1,2

Build three closures in a loop under each strategy, then invoke them. Capturing the shared loop variable makes all three return the final value; capturing a fresh copy per iteration gives each its own:

It captures the variable, not the value
shared cell · fresh cell
capture strategy
display class: not built yet
closure outputs
build, then invoke
Shared capture: one display cell, so all three closures read the final i = 3. Fresh capture: a new cell per iteration, so each closure keeps 0, 1, 2. The hoisted variable decides the result.
A closure captures a snapshot of the variable's value.
It captures the variable — a live shared cell. Change the variable after creating the lambda and the lambda sees the new value. This is why the loop-variable capture surprises people: the cell keeps mutating after the closures are made.
Lambdas are free, lightweight function pointers.
A lambda that captures state compiles to a heap-allocated display object (and a delegate). In a tight loop that's real gen-0 pressure. A lambda that captures nothing can be cached as a static singleton — but the moment it closes over a variable, there's an allocation behind it.
TypesAdvancedModule 05

Records & value equality

A record is a class the compiler fills in for you with value semantics: a structural Equals and GetHashCode that compare every field, a readable ToString, and — most usefully — a with expression for non-destructive updates that copies the object with a few fields changed. Two records with equal fields are equal even though they're different instances; two plain classes never are, because a class compares by reference. Records are the idiomatic way to model immutable data, and they make "same value" mean what you'd expect.

records.cs 
record Link(string Code, int Hits);

var a = new Link("4C92", 10);
var b = new Link("4C92", 10);
bool eq  = a == b;                 // True — value equality (all fields match)
bool rEq = ReferenceEquals(a, b);   // False — still two instances
var c = a with { Hits = 99 };       // a copy; a is unchanged

Two objects a and b hold identical field values. Compare them as a record and as a class, and use with to produce a modified copy — watching the original stay put:

Record: value equality → equal fields make a == b True (though they're distinct instances). Class: reference equality → False. with copies a and changes Hits, leaving a untouched.
↳ Advanced C# · Vol. I (Structs) · OOP
Records give the value semantics of a struct without copying on every assignment, and finally deliver the equality-by-value the OOP course wanted. Combined with init-only setters, they're immutable data done right — the safe default the "prefer immutability" principle kept pointing at.
record is just shorthand for a class with a constructor.
It generates a whole value-equality contract — Equals, GetHashCode, ==, ToString, and a with-clone. Two records with matching fields are equal and hash the same, so they behave correctly as dictionary keys and in sets — a class does none of that for free.
with mutates the object in place cheaply.
It's non-destructive — it allocates a new object copying all fields, then applies your changes. The original is untouched (that's the point for immutable data), but it is a real allocation, so with in a hot loop has a cost like any other object creation.
NullabilityAdvancedModule 06

Nullable reference types

Nullable reference types are a pure compile-time flow analysis — zero runtime cost, no bit changes, nothing but the compiler tracking, at every point in your code, whether a reference could be null. A string? starts maybe-null; dereferencing it there earns a CS8602 warning. But add a guard — if (name is not null) — and inside that branch the compiler narrows the state to not-null, and the dereference becomes safe. The whole feature is the compiler reasoning about null-state as it flows, turning a class of NullReferenceExceptions into build-time warnings.

null-flow.cs 
string? name = GetName();   // state: maybe-null

int n = name.Length;         // CS8602 — dereference of a possibly-null reference

if (name is not null)          // guard narrows the flow-state...
    n = name.Length;         // ...to not-null here — safe, no warning

Step through a method that gets a maybe-null value. Try dereferencing it (the compiler warns), then add a guard and watch the flow-state narrow to not-null, making the same dereference safe:

The compiler tracks null-state
maybe-null → narrowed by a guard
string? name  ·  flow-state: maybe-null
name.Length → —
get a maybe-null value, then try to dereference it
Maybe-null → dereference warns (CS8602). A guard narrows the state to not-null inside the branch, so the dereference is proven safe. The null-forgiving ! overrides the compiler — on your word.
Nullable reference types add runtime null checks.
They add nothing at runtime — it's static analysis only. A string? and a string are the identical type to the CLR. The ? and the warnings live entirely in the compiler's flow analysis; they change what it tells you, not what runs.
The null-forgiving ! operator fixes a nullability problem.
It silences the warning by asserting you know better — it performs no check. If you're wrong, you get the exact NullReferenceException the analysis was warning about. Use it only where you can prove non-null by logic the compiler can't see; reach for a guard first.
Type systemExpertModule 07

Generic variance

Why can you assign an IEnumerable<string> to an IEnumerable<object>, but not a List<string> to a List<object>? Variance. An interface marked out T is covariant — safe because it only ever produces T, so a producer of strings is a valid producer of objects. An interface marked in T is contravariant — safe because it only consumes T, so a consumer of objects can stand in for a consumer of strings. A type that both produces and consumes (like List<T>) must be invariant, or you could break type safety. The in/out annotations encode exactly which substitutions are sound.

variance.cs 
IEnumerable<out T>   // covariant — produces T
Action<in T>         // contravariant — consumes T

IEnumerable<object> a = strings;   // ✓ covariance (string → object)
Action<string> b = objAction;      // ✓ contravariance (object → string)
List<object> c = stringList;      // ✗ invariant — would break type safety

covariance (out) · contravariance (in)
IEnumerable<out T>   IEnumerable<string> → IEnumerable<object>test →
IEnumerable<out T>   IEnumerable<object> → IEnumerable<string>test →
Action<in T>   Action<object> → Action<string>test →
Action<in T>   Action<string> → Action<object>test →
List<T>   List<string> → List<object>test →
click an assignment to test it against the variance rules
Covariant out allows string → object (producers). Contravariant in allows object → string (consumers). The reverse of each is unsound, and an invariant type like List permits neither.
↳ OOP & Coding Principles · Data Structures
This is the Liskov Substitution Principle expressed in the type system: covariance is exactly when a Sequence<Derived> can safely stand in for a Sequence<Base>. The in/out keywords are the compiler encoding which direction substitution stays safe.
If string is an object, then List<string> is a List<object>.
No — that would let you Add an object into a list of strings and blow up. List<T> is invariant precisely because it both reads and writes T. Variance is only safe for one-directional interfaces: produce-only (out) or consume-only (in).
Variance is an obscure feature I'll never touch.
You use it constantly without naming it — passing an IEnumerable<Derived> where IEnumerable<Base> is expected, or a general IComparer<object> where a specific one is needed. The out/in on the framework interfaces is what makes those everyday assignments compile.
ResourcesExpertModule 08

IDisposable & finalization

The GC reclaims memory, but it doesn't know or care about resources — file handles, sockets, database connections. Those need deterministic release, and that's what IDisposable plus using provide: Dispose() is called the instant control leaves the scope, in reverse (LIFO) order, even if an exception is thrown. Relying on a finalizer instead is a last-resort safety net — it runs on the GC's schedule, possibly much later, and stalls object cleanup. The rule is simple: own a resource, wrap it in using.

disposal.cs 
using var conn = Open();      // Dispose() called at scope exit...
using var cmd  = conn.Command();  // ...in reverse order: cmd, then conn
DoWork();                        // even if this throws, both are disposed on the way out
// no using → the handle leaks until a finalizer eventually runs (non-deterministic)

Acquire resources inside a using scope and exit it (or throw) — watch Dispose fire deterministically in LIFO order. Then leak one without using and see it linger until a garbage collection finalizes it:

Deterministic, or eventually
using · LIFO · finalizer fallback
open resources
open handles0
disposed0
acquire resources in a using scope, then exit it
using disposes deterministically at scope exit, LIFO, exception or not. A resource acquired without using leaks until a finalizer runs on the GC's schedule — deterministic release is the whole reason using exists.
↳ Advanced C# · Vol. I (the GC)
The GC from Vol. I frees memory but runs finalizers non-deterministically — which is exactly why resources need using. Deterministic disposal is the complement to generational collection: one handles managed memory automatically, the other releases unmanaged resources on time.
The garbage collector will clean up my file handles and connections.
The GC frees memory, not OS resources, and it runs whenever it decides to. A handle held until finalization can starve a connection pool or lock a file for far too long. Deterministic release via using is the only reliable way — the finalizer is a backstop, not a plan.
If a Dispose throws or I forget one, the others are skipped.
using compiles to try/finally, so each disposal runs on the way out regardless of exceptions in the body — and nested usings unwind in reverse order. That LIFO, exception-safe guarantee is precisely why you prefer it over calling Dispose by hand.
Async perfExpertModule 09

ValueTask & async allocation

Every Task<T> you return is a heap object. For a method that usually completes synchronously — a cache lookup that mostly hits — that allocation is pure waste, once per call, on your hottest path. ValueTask<T> fixes it: it's a struct that can wrap an already-available result with zero allocation, falling back to a real Task only when the operation genuinely runs asynchronously. Return ValueTask from hot, usually-synchronous methods and you delete an allocation from the common case while keeping full await support for the slow one.

value-task.cs 
async ValueTask<Link> GetAsync(string code) {
    if (_cache.TryGet(code, out var hit))
        return hit;                 // synchronous — wrapped in a struct · 0 allocation
    return await _db.LoadAsync(code);   // async miss — falls back to a Task
}   // Task<Link> here would allocate on EVERY call, hit or miss

Call a cache lookup under each return type. On a hit (synchronous completion), Task still allocates while ValueTask allocates nothing; only a real async miss allocates for either:

Zero allocation on the synchronous path
Task always allocates · ValueTask can skip it
return type
call the method — a hit completes synchronously
heap allocations0
ValueTask on a cache hit wraps the result in a struct — 0 allocations no matter how often you call. Task allocates every time. Only a genuine async miss forces an allocation for either.
↳ Advanced C# · Vol. I (async · GC)
This is the async state machine from Vol. I meeting the allocation discipline of the Span module: when the machine completes without ever suspending, ValueTask lets it return without touching the heap — less gen-0 pressure on the exact paths that run most often.
Always return ValueTask instead of Task — it's strictly faster.
It's faster for hot, usually-synchronous methods and worse as a default: a ValueTask must not be awaited twice or stored, and misusing it is a real bug. Use it deliberately where you've measured a synchronous-completion hot path — not everywhere.
await always involves an allocation and a thread hop.
When the awaited operation is already complete, await continues synchronously, no hop — and with ValueTask, no allocation either. The machinery only pays its full cost when the operation actually suspends. Fast paths stay fast.
MetaprogrammingExpertModule 10

Source generators

The final rewrite is one you author yourself. A source generator is code that runs inside the compiler: it inspects your program as Roslyn compiles it and emits additional C# source into the same compilation. Mark a class with an attribute and a generator can produce its ToString, a JSON serializer, or strongly-typed config — all as real code, checked and compiled alongside yours, with zero runtime cost. It's the compile-time answer to reflection: instead of reading metadata on every call at runtime, you generate the exact code once at build time. Modern .NET uses this for high-performance JSON, logging, and regex.

source-gen.cs 
[GenerateToString]
partial class Link { public string Code; public int Hits; }

// at compile time, the generator reads Link and emits:
// partial class Link { public override string ToString()
//   => $"Link {{ Code = {Code}, Hits = {Hits} }}"; }
// no reflection, no runtime scan — just code, compiled with yours

The generator runs once at compile time and emits real C# — zero runtime cost, no reflection. Reflection does the same job by reading metadata on every call: slower and allocating. Same result, opposite trade-off.
↳ Advanced C# · Vol. I (Expression trees)
Two flavors of metaprogramming, mirror images in time: expression trees are built and compiled at runtime for flexibility; source generators emit code at compile time for speed. Together they bracket the trade space the Vol. I module opened — one buys adaptability, the other buys zero-overhead.
Source generators run when my program starts.
They run during compilation, as part of the build — their output is ordinary source compiled into your assembly. By the time the program starts, the generated code is just… code. There's no startup step and nothing to initialize.
Reflection and source generation are interchangeable ways to do the same thing.
They trade runtime cost for build-time work. Reflection reads metadata every call and allocates; a generator emits the specialized code once at build. That's why .NET moved hot paths — JSON, logging, regex — onto generators: same capability, no per-call reflection tax.
ExpertModule 11

The whole language

Step back and the volume has one shape. Nearly every feature you met is the compiler lowering a convenient surface into machinery that runs on the Volume I runtime. Iterators and async become state machines; closures become display classes; records become value-equality contracts; source generators emit whole files. Even the features with no emitted code — ref safety, ref-struct rules, nullable flow — are the compiler reasoning about your program so the runtime doesn't have to. The map:

what you write → what the compiler makes of it 
  yield / async  ──▸  state machine (IEnumerator / MoveNext)     // m03
  lambda + capture ─▸  display class on the heap                 // m04
  record / with   ──▸  Equals · GetHashCode · clone              // m05
  [Generator]     ──▸  emitted .g.cs at compile time             // m10
  ref · ref struct ─▸  lifetime rules, checked (no emitted code) // m01–02
  string? · in/out ─▸  flow & variance analysis (compile-time)   // m06–07

And here is Snip's resolver once more — every line now a feature whose rewrite you can see, composed with Volume I's runtime tricks:

snip-resolver.cs 
record Link(string Code, int Hits, bool Active);   // m05 · immutable value

async ValueTask<Link?> ResolveAsync(string url) {
  using var conn = _pool.Rent();                    // m08 · deterministic release
  ReadOnlySpan<char> code = url.AsSpan(5, 4);       // ref struct (m02) · 0 alloc
  if (_cache.TryGet(code, out var hit)) {
    ref int n = ref hit.HitCount;                    // m01 · alias, mutate in place
    n++;
    return hit;                                       // m09 · sync path — ValueTask, 0 alloc
  }
  var link = await conn.LoadAsync(code.ToString());
  return link is { Active: true } ? link : null;      // m06 · pattern + nullable
}   // streamed to analytics by an iterator; serialized by a source generator
The whole language, read as machine

The record is a generated value-equality type (m05). The using is exception-safe LIFO disposal (m08). AsSpan is a stack-only ref struct slicing with zero allocation (m02). The ref local aliases the counter and mutates it in place (m01). The ValueTask returns the cache hit without allocating (m09). The pattern and the nullable return are compile-time-checked (m06). Behind each keyword is a rewrite you've now watched happen. Two volumes in, C# has no unlit corners: the runtime beneath, and the compiler that lowers every feature onto it, are both things you can now read.
