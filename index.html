TypeScript · The Type System

The type system as a language in its own right — one that computes at compile time, then erases. Start from types-as-sets (assignability is ⊆, never is ∅, unknown is the universal set), then work through structural typing and excess-property checks, control-flow narrowing of discriminated unions with never-based exhaustiveness, generics with constraints and inference, widening and as const, then genuine type-level programming: conditional types with infer extracting inner types, mapped types over template literals building a Handlers type key by key, keyof and indexed access deriving types from values. Finish with function variance, user-defined type guards and assertion functions, and the twist that separates TS from C# — tsc checks every type and emits plain JavaScript with all of them erased.

Distributed Systems

The frontier the Capstone leads to: what happens when one machine isn't enough, or when that one machine fails? A course built on two unforgiving facts — some part is always failing, and there is no global clock — as Snip scales from one box to a global fleet. Partition data with consistent hashing so adding a node barely reshuffles the ring; replicate with quorums and watch W + R > N flip reads between strong and stale; elect a leader and commit by majority with Raft; order events with Lamport clocks; walk the consistency spectrum and the CAP theorem live by splitting the network; run two-phase commit until the coordinator dies and the participants block; make message delivery idempotent; trip a circuit breaker open and watch it self-heal; and merge divergent replicas with a CRDT that can't lose a write. Failure isn't an exception here — it's the steady state you design for.
