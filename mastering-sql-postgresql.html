BeginnerModule 00

The relational model

Before any syntax, one idea. A relational database stores data as tables — grids of rows and columns. Each row is one record (one employee, one order, one event). Each column is one attribute of that record, with a fixed type (an integer, some text, a date). That's it. The power doesn't come from the tables themselves; it comes from the relationships between them and a language for asking questions across those relationships.

Keys: how rows find each other

Look at the colored column. Each table has a primary key — the id column — a value that uniquely identifies one row. No two departments share an id; given id = 2 there is exactly one department, Sales.

Now look at employees.dept_id. It isn't an arbitrary number — it's a foreign key that points at departments.id. Bob's dept_id is 1, which means Bob is in Engineering. The relationship between an employee and a department lives entirely in that one number. This is the relational model's central trick: store each fact once, and link to it by key. Department names aren't copied onto every employee row; they sit in one place and employees reference them.

Two rows that don't line up — on purpose

Ivan (employee 9) has a NULL dept_id: a new hire not yet assigned to a department. And Legal (department 5) has no employees pointing at it. These two deliberate gaps will matter enormously when we reach joins — they're exactly where INNER and LEFT joins behave differently.

Declarative: the mental shift

If you've written code in Python, Java, or JavaScript, you're used to telling the computer how to do something step by step: loop over this list, check each item, accumulate a result. SQL asks you to think differently. You describe what you want — "the names and salaries of employees earning over 90,000, highest first" — and the database's query planner decides how to get it: which table to scan, whether to use an index, how to sort. You write the goal; it writes the algorithm.

Holding onto this "describe the result" mindset is what separates people who fight SQL from people who fly with it.

A database is just a place to store data.
Storage is the least of it. The value is the query engine — the ability to ask complex questions across related tables and have them answered efficiently and correctly, with guarantees about consistency.

BeginnerModule 01

SELECT — asking for data

Every question to a database is a query, and almost every query starts with SELECT. The minimal shape is two clauses: SELECT which columns you want, and FROM which table. The result is itself a table — rows and columns — which is why SQL composes so cleanly: the output of a query looks just like the input.

sqlSELECT name, salary      -- the columns you want
FROM employees;          -- the table to read from

To get every column, use *. It's convenient when exploring, but in real queries you should name the columns you need — it's clearer, survives schema changes, and moves less data.

sqlSELECT *  FROM departments;

Aliases, expressions, and DISTINCT

Columns can be renamed in the output with AS (an alias), and you can compute new columns with expressions — arithmetic, function calls — right in the SELECT list. The result is computed per row.

sqlSELECT name,
       salary / 12 AS monthly_pay,        -- computed column
       upper(name) AS shout
FROM employees;

And when you want the distinct values a column takes — no duplicates — reach for DISTINCT. "Which locations do we have offices in?" is really "the distinct values of location":

sqlSELECT DISTINCT location  FROM departments;

BeginnerModule 02

Filtering with WHERE

A query without a filter returns the whole table. The WHERE clause keeps only the rows where a condition is true. Conceptually the database tests every row against the condition and discards the ones that fail.

sqlSELECT name, salary
FROM employees
WHERE salary >= 100000;

Conditions combine with AND, OR, and NOT. AND binds tighter than OR, so use parentheses when mixing them — the meaning changes completely otherwise.

sqlSELECT name, dept_id, salary
FROM employees
WHERE dept_id = 1 AND salary > 110000;   -- both must hold

The convenient operators

Three operators save you from clumsy chains of AND/OR:

  • IN (…) — matches any value in a list. dept_id IN (1, 2) beats dept_id = 1 OR dept_id = 2.
  • BETWEEN a AND b — an inclusive range. salary BETWEEN 80000 AND 100000 includes both ends.
  • LIKE — pattern matching on text, where % means "any run of characters" and _ means "one character". name LIKE 'A%' finds names starting with A. (PostgreSQL's ILIKE does the same, case-insensitively.)
sqlSELECT name, salary
FROM employees
WHERE dept_id IN (1, 2)
  AND salary BETWEEN 85000 AND 125000;

NULL: the value that isn't

A NULL means "unknown" or "absent" — not zero, not an empty string. This has a startling consequence: comparisons with NULL are never true. NULL = NULL is not true; it's unknown. So WHERE dept_id = NULL matches nothing, ever. To test for null you must use IS NULL or IS NOT NULL.

sql-- find the unassigned employee (Ivan)
SELECT name  FROM employees  WHERE dept_id IS NULL;

-- this returns NOTHING — a classic beginner trap:
SELECT name  FROM employees  WHERE dept_id = NULL;

This is three-valued logic: every condition is true, false, or unknown, and only true rows survive. It trips up nearly everyone at first, so we'll make it tangible. Type a condition below and watch which rows it keeps:

BeginnerModule 03

Sorting & limiting

Here is a fact that surprises people: a table has no inherent order. Rows come back in whatever order the database finds convenient, and that can change between runs. If you want a guaranteed order, you must ask for it with ORDER BY.

sqlSELECT name, salary
FROM employees
ORDER BY salary DESC;     -- DESC = high → low; ASC (default) = low → high

You can sort by several columns — the first is the primary sort, the next breaks ties, and so on. Sort by department, then by salary within each department:

sqlSELECT name, dept_id, salary
FROM employees
ORDER BY dept_id ASC, salary DESC;

Where do NULLs sort? PostgreSQL places them last for ascending order and first for descending, and you can override with NULLS FIRST / NULLS LAST.

LIMIT and OFFSET

LIMIT caps how many rows you get back — perfect for "top N" questions and for not accidentally pulling a million rows. OFFSET skips rows first, the naive basis of pagination.

sql-- the three highest-paid employees
SELECT name, salary
FROM employees
ORDER BY salary DESC
LIMIT 3;
LIMIT without ORDER BY is a coin flip

LIMIT 3 on its own returns some three rows — not the "first" three in any meaningful sense, because there is no first without an order. Always pair LIMIT with ORDER BY when the identity of the rows matters. (And for real pagination, large OFFSET values get slow; keyset pagination — "where id > last_seen" — scales far better.)

Try the top-N pattern in the playground above: change the ORDER BY column, flip ASC/DESC, and adjust the LIMIT.

AggregationIntermediateModule 04

Aggregation & GROUP BY

So far every query returned individual rows. Often you want a summary instead: how many employees, the average salary, the biggest budget. That's aggregation — collapsing many rows into a single computed value.

sqlSELECT COUNT(*)        AS headcount,
       AVG(salary)    AS avg_salary,
       MAX(salary)    AS top_salary
FROM employees;

The five core aggregates are COUNT, SUM, AVG, MIN, and MAX. One subtlety worth burning in: COUNT(*) counts rows, while COUNT(column) counts non-NULL values in that column. COUNT(dept_id) over our employees returns 8, not 9 — Ivan's NULL isn't counted.

GROUP BY: aggregate per group

The real power arrives when you aggregate per group rather than over the whole table. GROUP BY partitions rows into groups that share a value, then computes the aggregate once per group, returning one row each. "Average salary by department" is the canonical example:

sqlSELECT dept_id,
       COUNT(*)     AS headcount,
       AVG(salary) AS avg_salary
FROM employees
GROUP BY dept_id;
The golden rule of GROUP BY

Every column in the SELECT list must either be inside an aggregate or listed in GROUP BY. Why? Because the group collapses to one row — if you ask for a bare salary alongside grouped data, the database has many salaries and one slot, and no way to choose. PostgreSQL will reject it with "must appear in the GROUP BY clause or be used in an aggregate function."

Watch grouping happen. Pick an aggregate, then collapse the nine employee rows into one row per department:

GROUP BY aggregation
many rows in, one row per group out
GROUP BY dept_id, computing
Rows sharing a dept_id form a group; the aggregate is computed once per group. Note the NULL group — Ivan, with no department.

HAVING: filtering groups

WHERE filters rows before grouping. To filter the groups themselves — "only departments whose average salary exceeds 100k" — you need HAVING, which runs after aggregation and can reference aggregate values.

sqlSELECT dept_id, AVG(salary) AS avg_salary
FROM employees
GROUP BY dept_id
HAVING AVG(salary) > 100000;   -- filter the groups, not the rows
Use HAVING to filter rows by a condition.
Use WHERE for rows; HAVING is specifically for filtering groups by aggregate results. WHERE salary > 90000 filters people; HAVING AVG(salary) > 90000 filters departments.
COUNT(*) and COUNT(column) are interchangeable.
Only when the column has no NULLs. COUNT(*) counts rows; COUNT(col) skips NULLs. The gap between them is literally a count of missing values.
JoinsIntermediateModule 05

Joining tables

This is the payoff of the relational model. Our employees table stores a dept_id, but humans want the department name. A JOIN follows the foreign key, stitching rows from two tables together wherever a condition matches.

sqlSELECT e.name, d.name AS department
FROM employees e
JOIN departments d ON e.dept_id = d.id;

Read it as: for each employee row e, find the department row d where e.dept_id = d.id, and combine them. The short letters (e, d) are table aliases, and once tables share column names like name, qualifying with the alias (e.name) stops things being ambiguous.

The four joins — it's all about the non-matches

A plain JOIN is an INNER join: it returns only rows that match on both sides. The other join types differ purely in how they treat rows with no match:

  • INNER JOIN — only matched pairs. Unmatched rows on either side vanish.
  • LEFT JOIN — every left row, matched or not; unmatched ones get NULLs for the right columns.
  • RIGHT JOIN — the mirror image: every right row is kept.
  • FULL JOIN — every row from both sides; unmatched on either side gets NULLs.

This is exactly where Ivan (an employee with no department) and Legal (a department with no employees) earn their keep. Switch join types below and watch which rows survive — and where NULLs appear:

JOIN visualizer
employees.dept_id = departments.id
An INNER join keeps only rows that match on both sides. Switch the join type to see unmatched rows reappear with NULLs.

Self-joins: a table joined to itself

Nothing stops you joining a table to itself — you just alias it twice. Our manager_id points at another employee's id, so to list each person beside their manager's name, join employees to employees:

sqlSELECT e.name AS employee, m.name AS manager
FROM employees e
LEFT JOIN employees m ON e.manager_id = m.id;

(A LEFT join so that people with no manager — Alice, Carla, Erin — still appear, with NULL for the manager.) Try all of these in the playground; joins are fully supported there.

A JOIN combines tables by stacking their rows together.
That's UNION (rows on top of rows). A JOIN combines them side by side, widening rows by matching on a condition. Different operations entirely.
LEFT and INNER joins return the same rows, just ordered differently.
They differ whenever the left side has unmatched rows. INNER drops Ivan (no department); LEFT keeps him with NULL. Choosing the wrong one silently loses or invents rows.
IntermediateModule 06

How a query really runs

Here is the single most clarifying idea in all of SQL. You write a query in one order, but the database executes it in a different one. Almost every confusing error and "why can't I do that?" dissolves once you know the real sequence — the logical query processing order.

1FROM / JOINassemble the rows from the source tableswritten 2nd
2WHEREkeep only rows matching the conditionwritten 3rd
3GROUP BYcollapse rows into groupswritten 4th
4HAVINGkeep only groups matching the conditionwritten 5th
5SELECTcompute the output columns & aliaseswritten 1st
6ORDER BYsort the result rowswritten 6th
7LIMITcut to the first N rowswritten 7th

Notice SELECT — written first — runs fifth, after the rows have been chosen and grouped. This one fact explains a whole family of behaviors:

  • Why WHERE can't see a SELECT alias. You can't write WHERE monthly > 5000 for an alias defined in SELECT — WHERE (step 2) runs long before SELECT (step 5) has created monthly.
  • Why ORDER BY can. Sorting (step 6) happens after SELECT, so ORDER BY monthly works fine — the alias exists by then.
  • Why WHERE can't use aggregates. WHERE COUNT(*) > 2 is illegal because grouping (step 3) hasn't happened yet at step 2. That's precisely the job HAVING exists to do.

Let's watch it. Below is a query that uses every stage. Run the pipeline and see the rows flow through, shrinking and transforming at each step:

The query pipeline
watch row counts change at each logical stage
SELECT dept_id, COUNT(*) AS n, AVG(salary) AS avg_pay
FROM employees
WHERE salary > 70000
GROUP BY dept_id
HAVING COUNT(*) >= 2
ORDER BY avg_pay DESC
LIMIT 2;
Nine rows enter at FROM. Each stage transforms them in logical order — ending with the two rows the query actually returns.
The database reads your query top to bottom, like a script.
It reads it as a declaration and executes in logical order (FROM → WHERE → GROUP → HAVING → SELECT → ORDER → LIMIT), then the planner reorders the physical work however is fastest.
Putting a condition in WHERE vs HAVING is a style choice.
It's a correctness choice tied to the order. WHERE filters rows pre-grouping; HAVING filters groups post-aggregation. They run at different stages and can give different results.
AdvancedModule 07

Subqueries & CTEs

A query's result is a table — so a query can be used inside another query, anywhere a table or a value is expected. These nested queries, subqueries, let you build answers in layers.

Subqueries by where they sit

A scalar subquery returns a single value and can stand in for one. "Who earns more than the company average?" — the average is itself a one-row query:

sqlSELECT name, salary
FROM employees
WHERE salary > (SELECT AVG(salary) FROM employees);   -- inner query yields one number

A subquery in FROM acts as a derived table you can query further. And a subquery in WHERE with IN or EXISTS tests membership — "departments that actually have employees":

sqlSELECT name
FROM departments d
WHERE EXISTS (SELECT 1 FROM employees e WHERE e.dept_id = d.id);

That inner query references d.id from the outer one — a correlated subquery, conceptually re-evaluated for each outer row. (EXISTS is usually the right tool over IN when you only care whether a match exists, and it handles NULLs more predictably.)

CTEs: subqueries with a name

Deeply nested subqueries become unreadable. A Common Table Expression (WITH) lets you name a query and reference it below, reading top-to-bottom like steps in a recipe:

sqlWITH dept_avg AS (
  SELECT dept_id, AVG(salary) AS avg_pay
  FROM employees
  GROUP BY dept_id
)
SELECT e.name, e.salary, da.avg_pay
FROM employees e
JOIN dept_avg da ON e.dept_id = da.dept_id
WHERE e.salary > da.avg_pay;   -- everyone paid above their department's average
Recursive CTEs walk hierarchies

A WITH RECURSIVE CTE can reference itself, which is how you traverse tree-shaped data — an org chart through manager_id, a category tree, a graph of dependencies. It starts from a base case and repeatedly joins the table to the rows found so far until none are left. It's the one place SQL expresses genuine iteration, and it's remarkably powerful for reporting lines and bill-of-materials problems.

A note on the in-page engine

The playground's mini-engine covers the core: SELECT, WHERE, JOIN, GROUP BY/HAVING, ORDER BY, and LIMIT. Subqueries, CTEs, and window functions (next module) are beyond it — running those is exactly when you'd reach for a real PostgreSQL instance. The engine will tell you politely if you try.

AdvancedModule 08

Window functions

Window functions are the feature that turns SQL from "fetch and summarize" into a genuine analytics language — and the one that most often makes people say "I didn't know SQL could do that." The core idea: compute across a set of related rows while keeping every individual row.

Contrast with GROUP BY, which collapses rows. A window function computes over a "window" of rows defined by OVER (…) but returns a value for every row — nothing is collapsed. Ranking each employee's salary within their department, while still listing every employee:

sqlSELECT name, dept_id, salary,
       RANK() OVER (PARTITION BY dept_id ORDER BY salary DESC) AS dept_rank
FROM employees;

Read OVER (...) as "for each row, look at this window of rows." PARTITION BY dept_id restarts the calculation per department; the inner ORDER BY sets the ranking order. Other essentials: ROW_NUMBER() (a unique sequential number), DENSE_RANK() (ranks with no gaps after ties), and LAG()/LEAD() (peek at the previous/next row — perfect for "change since last period"). A running total is just SUM with an ordered window:

sqlSELECT name, hire_date, salary,
       SUM(salary) OVER (ORDER BY hire_date) AS running_total
FROM employees;   -- cumulative payroll as people were hired

See the difference directly. The same nine employees, viewed three ways — collapsed by group, ranked within group, and accumulated:

GROUP BY vs window functions
collapse the rows, or keep them and compute across them
GROUP BY returns one row per department. A window function returns one row per employee, with the computed value alongside.
Window functions are just a fancier GROUP BY.
They're complementary. GROUP BY collapses rows to summaries; a window function keeps every row and adds a value computed over related rows. You often want both detail and summary at once — only windows give you that.
You can filter on a window function in WHERE.
No — windows are computed at the SELECT stage, after WHERE. To filter by dept_rank = 1, wrap the query in a CTE or subquery and filter in the outer query.
AdvancedModule 09

Designing the schema

Good queries start with good tables. The schema — the definitions of your tables, their columns and types, and the rules tying them together — is where you encode what's possible. A well-designed schema makes bad data hard to insert; a sloppy one lets contradictions creep in that no query can fully undo.

sqlCREATE TABLE departments (
  id      serial PRIMARY KEY,            -- unique, auto-incrementing identity
  name    text   NOT NULL UNIQUE,         -- required, and no two alike
  budget  numeric CHECK (budget >= 0)        -- a rule the data must obey
);

CREATE TABLE employees (
  id         serial PRIMARY KEY,
  name       text   NOT NULL,
  salary     numeric NOT NULL DEFAULT 0,
  dept_id    int    REFERENCES departments(id)  -- foreign key: must point at a real dept
);

Constraints: rules the database enforces

Constraints are guardrails checked on every insert and update. The workhorses: PRIMARY KEY (unique row identity), NOT NULL (value required), UNIQUE (no duplicates), CHECK (a custom boolean rule), DEFAULT (a fallback value), and the crucial FOREIGN KEY / REFERENCES, which guarantees referential integrity — you cannot give an employee a dept_id that doesn't exist, and (depending on the rule) you can't delete a department while employees still point at it. The database refuses to let your data become inconsistent.

Normalization: store each fact once

Why are department names in their own table instead of repeated on every employee row? Because duplicated data drifts. If "Engineering" were copied onto 200 employee rows and you renamed it, you'd have to update all 200 perfectly or end up with both "Engineering" and "Enginering" in your data. Normalization is the discipline of organizing tables so each fact lives in exactly one place, reached by key. The practical heuristics:

  • Each column holds a single value (no comma-separated lists stuffed in a text field).
  • Each non-key column depends on the whole key, and on nothing but the key.
  • Repeating groups get their own table, linked by a foreign key — exactly our employees/departments split.

This prevents update, insertion, and deletion anomalies — the trio of ways duplicated data betrays you. The counterweight is that heavily normalized data needs more joins to reassemble; high-read analytical systems sometimes denormalize deliberately, trading storage and write-complexity for read speed. Normalize by default; denormalize as a conscious, measured exception.

Indexes, briefly

An index is a separate, sorted data structure that lets the database find rows by a column's value without scanning the whole table — like the index at the back of a book versus reading every page. It makes matching reads dramatically faster, at the cost of extra storage and slightly slower writes (the index must be kept up to date). Primary keys and unique constraints are indexed automatically; foreign-key columns you join or filter on are usually worth indexing too. We'll see exactly how PostgreSQL uses them next.

Add an index to every column to be safe.
Indexes cost write time and space, and the planner may ignore ones that don't help. Index the columns you actually filter, join, or sort on — and verify with EXPLAIN that they're used.
Foreign keys are just documentation.
In PostgreSQL they're enforced. An insert referencing a non-existent parent row is rejected outright. They actively prevent the orphaned, inconsistent data that plagues schemas without them.
ExpertModule 10

PostgreSQL in depth

Everything so far is standard SQL that runs on any relational database. PostgreSQL takes that foundation and adds a famously rich, extensible feature set — it behaves less like a passive store and more like a programmable data platform. Here's what sets it apart.

A type system far beyond rows and columns

Alongside the usual numbers, text, dates, and booleans, PostgreSQL has first-class types most databases lack:

  • JSONB — JSON stored in a decomposed binary form that's indexable and queryable. You get schema flexibility without giving up SQL, querying nested fields with operators like -> and ->> and the containment operator @>.
  • Arrays — a column can hold int[] or text[], with operators to test membership and overlap.
  • Ranges (int4range, tstzrange) — a value representing a span, with built-in overlap logic, ideal for booking and scheduling.
  • UUID, ENUM, full-text search (tsvector), geometric types, network address types, and more.
sql-- query inside a JSONB column
SELECT data ->> 'email' AS email
FROM accounts
WHERE data @> '{"plan": "pro"}';   -- rows whose JSON contains this

EXPLAIN: see the plan the planner chose

Remember that SQL is declarative — you never told PostgreSQL how to run your query. EXPLAIN shows you what it decided, and EXPLAIN ANALYZE actually runs it and reports real timings. This is the single most important tool for performance work.

query planEXPLAIN SELECT * FROM employees WHERE dept_id = 1;

Seq Scan on employees  (cost=0.00..1.11 rows=3 width=64)
  Filter: (dept_id = 1)        -- reads every row; fine for 9 rows, slow for 9 million

On a tiny table a sequential scan (read everything, filter) is optimal. Add an index on dept_id and grow the table, and the planner switches to an index scan — jumping straight to matching rows. The plan is cost-based: PostgreSQL estimates the cheapest path using table statistics, which is why keeping statistics fresh (autovacuum/analyze) matters.

How an index finds a row so fast

The default index is a B-tree: a balanced, sorted tree. Instead of scanning N rows, a lookup walks from the root down a handful of levels — O(log n) — comparing at each step, exactly like binary search. That's the difference between reading nine million rows and reading roughly twenty-three:

PostgreSQL offers several index types for different jobs: B-tree for equality, ranges, and sorting; GIN for "does this contain that" over JSONB, arrays, and full-text; GiST for geometric and range overlap; BRIN for huge, naturally-ordered tables; plus partial indexes (index only some rows) and expression indexes (index lower(email) rather than email).

MVCC: concurrency without blocking reads

PostgreSQL's transactions are fully ACID — atomic, consistent, isolated, durable — and its concurrency model, MVCC (Multi-Version Concurrency Control), is a defining strength. Instead of locking rows for reading, it keeps multiple versions of a row, so each transaction sees a consistent snapshot and readers never block writers, and writers never block readers. The trade is that superseded row versions accumulate as "dead tuples," which VACUUM (usually automatic) reclaims. Isolation levels (Read Committed, Repeatable Read, Serializable) let you tune exactly how transactions perceive each other.

Extensible by design

Much of PostgreSQL's power is that you can extend it: PostGIS turns it into a world-class geospatial database, pg_trgm enables fuzzy text matching, pg_stat_statements profiles your queries, and you can add custom types, operators, functions, and even index methods. The same engine scales from a laptop side-project to enormous production systems.

ExpertModule 11

Writing fast, correct SQL

You know the language now. Expert-level SQL is mostly judgment: writing queries the planner can run efficiently, and sidestepping the traps that produce wrong answers or slow ones. A field guide to the habits that matter most.

Make the planner's life easy

  • Keep predicates sargable. An index on email can't help WHERE lower(email) = '…' — the function on the column hides the indexed value. Index the expression, or store it normalized. The same defeats indexes when you wrap a column in arithmetic or a cast.
  • Select only the columns you need. SELECT * moves data you'll discard and can prevent index-only scans. Name your columns.
  • Join instead of looping. Fetching a list and then querying once per item (the "N+1" pattern, common via ORMs) turns one query into hundreds. A single join does it in one round trip.
  • Filter early. Push conditions into WHERE so rows are eliminated before joins and grouping do expensive work over them.
  • Prefer EXISTS to IN for "is there a match" tests on subqueries — it can stop at the first hit and behaves better with NULLs.

And the meta-skill: measure, don't guess. When a query is slow, run EXPLAIN ANALYZE and read what the planner actually did — a sequential scan where you expected an index scan, a row-count estimate wildly off from reality (stale statistics), a join strategy you didn't anticipate. The plan tells you the truth; intuition often lies.

Capstone · solve these in the playground

1. List every department whose average salary is above 90,000, highest average first.

▸ Show answer

SELECT dept_id, AVG(salary) AS avg_pay FROM employees GROUP BY dept_id HAVING AVG(salary) > 90000 ORDER BY avg_pay DESC;

2. Show each employee's name with their department name and location, excluding the unassigned, sorted by salary (highest first).

▸ Show answer

SELECT e.name, d.name AS dept, d.location, e.salary FROM employees e JOIN departments d ON e.dept_id = d.id ORDER BY e.salary DESC; — the inner join naturally drops Ivan.

3. Find departments with no employees. (Hint: LEFT JOIN, then keep rows where the employee side is NULL.)

▸ Show answer

SELECT d.name FROM departments d LEFT JOIN employees e ON e.dept_id = d.id WHERE e.id IS NULL; — returns Legal. This "anti-join" is a hugely useful pattern.

4. (stretch) The highest-paid employee in each department, with their name. This one needs a window function or correlated subquery — run it in a real PostgreSQL.

▸ Show answer

SELECT dept_id, name, salary FROM (SELECT *, RANK() OVER (PARTITION BY dept_id ORDER BY salary DESC) AS r FROM employees) t WHERE r = 1; — rank within each department, then keep the top of each. The classic "top-N-per-group" pattern.
