AdvancedModule 00

Beneath the syntax

C# reads like a high-level language — garbage-collected, memory-safe, no pointers in sight. But that friendliness is a surface. Underneath sits a genuine systems platform: a stack and a heap you can reason about precisely, value types that live inline and reference types that live behind a pointer, a JIT that turns your generics into specialized machine code, and a garbage collector making generational bets about object lifetimes. The intro course taught you to write C#. This one shows you the machine it runs on — because at the senior level, performance and correctness both live in the details the syntax hides.

Three layers sit under every line you write. Your C# is compiled to IL (a portable stack-based bytecode) by Roslyn. At runtime the JIT compiles that IL to machine code for your CPU, specializing and inlining as it goes. And the whole thing executes on the CLR — the runtime that manages memory, threads, and the type system. Most of the time you can ignore all three. The moments you can't — a hot path allocating on every request, an async deadlock, a data race — are exactly what separates fluent from expert.

the surface, and under it 
int x = 5;                    // value — lives inline, on the stack
var link = new Link();          // reference — a pointer to a heap object

ReadOnlySpan<char> code = url.AsSpan(5, 4);  // a window into memory — 0 allocation
await db.SaveAsync(link);       // compiles to a state machine — blocks no thread

Every module opens one of these boxes and instruments it, so you can watch the mechanism move rather than take it on faith. We'll optimize Snip along the way: parse short codes without allocating, run the request pipeline without blocking threads, and count hits without losing a single one. Hold one idea for the whole course: the abstraction is a convenience, not a mystery — and you can always look underneath.

↳ Bits, Pointers & Overflow · OOP
The Bits course showed how numbers sit in memory; this one shows how everything else does — objects, references, spans, stack frames. And the value-vs-reference split you'll instrument here is the concrete, in-memory reason the OOP course's "identity vs equality" distinction matters.
C# is a high-level language, so memory isn't my concern.
C# is high-level and gives you precise control when you want it — spans, stackalloc, struct layout, ref semantics. Ignoring memory is fine until a hot path allocates millions of times a second. The GC is a safety net, not a reason to be careless.
The JIT just translates IL one-to-one to machine code.
The JIT specializes and optimizes — inlining calls, eliminating bounds checks it can prove safe, generating distinct code per value-type generic. Your List<int> and List<string> run genuinely different machine code, and knowing that explains real performance differences.
MemoryAdvancedModule 01

Structs, classes & the copy

The single most consequential choice in C# is struct versus class, and it's a choice about memory and identity. A class is a reference type: the variable holds a pointer to an object on the heap, and assigning it copies the pointer — two variables now alias one object. A struct is a value type: the variable is the data, laid out inline, and assigning it copies the whole value — two independent copies. Mutating one never affects the other. Get this wrong and you get either surprising aliasing bugs or accidental allocation storms.

value-vs-reference.cs 
struct PointV { public int X; }    // value type
class  PointR { public int X; }    // reference type

var a = new PointV { X = 1 };
var b = a;        // struct: COPY — b is a separate value
b.X = 9;          // a.X is still 1 — they don't share

object boxed = a; // BOXING — copies the struct onto the heap (allocation!)

Pick a type, then assign b = a, mutate b.X, and watch what happens to a — and to the heap. Struct copies leave a untouched and allocate nothing; class assignment aliases, so mutating b changes a too:

Struct: b = a copies; mutating b leaves a at 1; zero heap allocations. Class: b = a aliases the same object; mutating b changes a too. Boxing a struct always allocates.
↳ OOP & Coding Principles
"Prefer immutability" has teeth here: a mutable struct copied by accident is a classic bug source, which is why modern C# leans on readonly struct and record struct. Value semantics done right gives you the equality-by-value the OOP course wanted, with no heap cost.
Structs are just lightweight classes.
They have opposite semantics: copied on assignment, compared by value, stored inline, no heap allocation, no null by default. That makes them great for small immutable data and terrible for large mutable state passed around — every copy duplicates every field.
Boxing is a rare, obscure thing.
It happens whenever a value type flows into an object or a non-generic interface slot — and each box is a heap allocation plus a copy. Generics and Span<T> exist largely to avoid it. Spotting hidden boxing in a hot loop is a core optimization skill.
MemoryAdvancedModule 02

Span<T> & zero allocation

Here's the flagship idea of high-performance C#: you can work with slices of memory without copying them. A Span<T> (and its read-only sibling) is a window — a pointer plus a length — into contiguous memory that already exists: an array, a string, or a stackalloc'd buffer. Slicing a span produces another window into the same memory, allocating nothing. Contrast Substring, which allocates a brand-new string every time. For Snip, parsing a short code out of a URL a million times a second is the difference between zero allocations and a million.

parse-code.cs 
string url = "SNIP-4C92-LAUNCH";

ReadOnlySpan<char> code = url.AsSpan(5, 4);  // window into the SAME chars — 0 alloc
string copy = url.Substring(5, 4);       // a NEW 4-char string — 1 allocation

Span<byte> scratch = stackalloc byte[16];    // 16 bytes on the stack — 0 heap

The buffer below is Snip's URL in memory. Slice out the code as a span and watch the window land on the existing cells with the allocation counter frozen at zero — then do the same with Substring and watch the heap counter climb with every call:

The span reads the same "4C92" straight out of the buffer — the counter never moves. Every Substring materializes a fresh string. On a hot path, that gap is the whole game.
↳ Bits, Pointers & Overflow · Data Structures
A span is the safe, bounds-checked version of the pointer-plus-length the Bits course described — the same idea C used for decades, minus the foot-guns. And it's why array-backed structures from Data Structures can be sliced and passed around without ever copying their contents.
Slicing a string or array copies the data out.
A Span slice copies nothing — it's a new (pointer, length) pair over the same memory. That's why it's a ref struct that can't escape to the heap: its whole purpose is to reference memory it doesn't own. Zero-copy is the default, not an optimization.
Avoiding allocations is premature optimization.
On a request hot path, allocation is the cost — it's what feeds GC pressure and pauses. Spans, stackalloc, and pooling let hot code run allocation-free while the rest of the app stays comfortably high-level. Measure first, then span the paths that matter.
AsyncAdvancedModule 03

The async state machine

The biggest misconception in C# is that async means "runs on another thread." It doesn't. The compiler rewrites an async method into a state machine — a struct with an integer state field and a MoveNext() method. Each await is a state. When you hit an await whose operation isn't finished, MoveNext returns, handing the thread back to do other work, and schedules a continuation to call MoveNext again when the operation completes. No thread is blocked waiting — that's the entire point of async I/O.

what the compiler generates (sketch) 
async Task HandleAsync() {
  await db.SaveAsync(link);      // state 0 — suspend here if not done
  await cache.SetAsync(code);    // state 1 — suspend here if not done
}
// ≈ struct with: int _state; MoveNext(){ switch(_state){ ... } }
// incomplete awaiter → return, release the thread, resume via continuation

Call the method and step it. Watch MoveNext run to the first await and suspend — releasing the thread; complete the awaited operation to resume into the next state, and again, until the machine finishes:

state field

Each await is a suspension point: MoveNext returns and the thread goes back to the pool. When the I/O completes, a continuation calls MoveNext again to pick up at the next state. Nothing blocked.
↳ Mastering ASP.NET Core
This is why an async web server handles thousands of concurrent requests on a handful of threads: each awaited database or network call releases its thread instead of parking it. The intro course's async controllers were this state machine, once per request.
async/await spins up a background thread for the work.
By default it spins up nothing. Async I/O uses the OS's completion mechanism; the thread is returned to the pool at each await and reused. Task.Run is how you'd move CPU work to a thread — awaiting I/O is the opposite of that.
Blocking on async (.Result / .Wait()) is basically the same as awaiting.
It defeats the machine — the thread is held instead of released, and can deadlock when the continuation needs that very thread back. The state machine only helps if you let it suspend. "async all the way" isn't a style preference; it's how the mechanism stays correct.
ConcurrencyAdvancedModule 04

Channels & backpressure

Once work is async, you need to coordinate producers and consumers — and the modern tool is Channel<T>, an async-native queue. The subtle, essential feature is backpressure: a bounded channel has a capacity, and when it's full, WriteAsync doesn't fail or buffer unboundedly — it awaits until a consumer makes room. That single behavior is what keeps a fast producer from exhausting memory when the consumer can't keep up. For Snip, it's how you'd feed a burst of click events into an analytics writer without falling over.

click-pipeline.cs 
var ch = Channel.CreateBounded<Click>(capacity: 3);   // bounded → backpressure

await ch.Writer.WriteAsync(click);   // awaits here when the buffer is full
await foreach (var c in ch.Reader.ReadAllAsync())   // consumer drains at its own pace
    await analytics.RecordAsync(c);

Fill the bounded channel by writing, and watch the writer block on backpressure once capacity is reached — then read to free a slot and let a pending write through:

Full means wait
bounded capacity · WriteAsync awaits
capacity 3 →
WriteAsync awaiting…
buffered0
pending writes0
total written0
total read0
write items until the buffer fills — the next write will have to wait
Three writes fill the buffer; a fourth awaits. A read frees exactly one slot and lets one pending write complete. That coupling is backpressure — the producer can never outrun memory.
A queue between producer and consumer smooths everything out.
An unbounded queue just relocates the failure — a fast producer fills it until you run out of memory. A bounded channel converts overload into waiting, which is recoverable. Backpressure is the difference between slowing down and falling over.
Channels are just a fancy ConcurrentQueue.
They're an async coordination primitive: writers and readers await instead of spin, completion is signalled, and bounded mode applies backpressure. ConcurrentQueue gives you thread-safe storage; a channel gives you the whole producer/consumer protocol.
GenericsAdvancedModule 05

Generics & JIT specialization

C# generics aren't C++ templates (compile-time code duplication) or Java generics (type erasure). They're a runtime mechanism with a clever split. For each value-type argument, the JIT generates dedicated machine code with the type laid out inline — no boxing, real specialization. For all reference-type arguments, it shares one canonical implementation (called __Canon) because every reference is the same size — a pointer. So List<int> and List<double> run distinct code, while List<string> and List<Link> share it. This is why generics are both fast and compact.

specialization.cs 
var a = new List<int>();      // JIT: dedicated code, int inline, no boxing
var b = new List<double>();   // JIT: a second dedicated specialization
var c = new List<string>();   // JIT: shared __Canon (reference)
var d = new List<Link>();     // JIT: same __Canon — no new code

Instantiate the generic with different type arguments and watch which ones trigger a fresh JIT specialization and which reuse the shared canonical code:

One generic, many machine-code bodies
value → dedicated · reference → __Canon
JIT-compiled specializations
none yet — instantiate the generic
distinct native code bodies0
int and double each get their own specialized body; string and Link both resolve to the single shared __Canon. Four instantiations, three native bodies.
↳ Data Structures · Big O
Every generic collection from Data Structures pays this off: List<int> stores ints inline with no per-element boxing, so the constant factor behind its O(1) access is genuinely small. Generics are how you get type safety and the tight memory layout the complexity assumed.
Generics are erased at runtime like in some other languages.
C# keeps full type information at runtime — you can ask a List<int> what its element type is, and the JIT produces specialized code for value types. No erasure, no hidden boxing for value-type generics. That reified model is a real performance and expressiveness advantage.
Using many generic type arguments bloats the binary.
Reference-type instantiations share one canonical body, so a hundred List<SomeClass> types cost one implementation. Only distinct value types add code — and that's the code you want, because it's the fast, allocation-free path.
Type systemAdvancedModule 06

Pattern matching

Modern C# turned switch from a jump table into a pattern-matching expression. You can match on type, on property values, on relational conditions (> 0), on tuples, and on list shapes — and the whole thing is an expression that returns a value. The feature that makes it safe is exhaustiveness checking: for a closed set of inputs, the compiler warns (CS8509) if you haven't handled every case, including null. That turns "did I forget a case?" from a runtime surprise into a compile-time warning.

classify.cs 
string label = n switch {
    < 0    => "negative",       // relational pattern
    0      => "zero",
    > 0    => "positive",
    null   => "none",           // handle null → exhaustive
};   // omit the null arm and the compiler warns: not all inputs handled

Pick an input and watch the matching arm light up. Then toggle the null arm off and see the switch become non-exhaustive — the compiler's warning, live:

relational patterns · exhaustiveness
< 0⇒ "negative"
0⇒ "zero"
> 0⇒ "positive"
null⇒ "none"
✓ exhaustive — every input is handled
pick an input to see which arm the switch selects
Relational patterns partition the number line; the null arm closes the last gap. Drop it and the compiler warns that null falls through — exhaustiveness caught at build time.
↳ OOP & Coding Principles · Design Patterns
Pattern matching is often the clean alternative to a Visitor or a type-switch tower from the Patterns course — especially over record hierarchies, where property patterns read like the shape of the data. Exhaustiveness gives you the "handled every subtype" guarantee OOP polymorphism promised.
switch expressions are just cleaner switch statements.
They're expressions that return values and support rich patterns — type, property, relational, positional, list — plus exhaustiveness analysis. That combination replaces whole families of if-else towers and Visitor boilerplate with one checked, value-producing construct.
A default (_) arm is always good defensive practice.
Over a closed set (an enum, a sealed hierarchy), an explicit _ can hide the exhaustiveness warning that would otherwise tell you a new case is unhandled. Sometimes the safer choice is to omit it and let the compiler flag the gap when the type grows.
LINQAdvancedModule 07

LINQ & deferred execution

Here's a truth that will feel familiar if you took Angular Internals: LINQ is lazy, pull-based enumeration. Building a query with Where and Select executes nothing — it composes an iterator. Elements flow through the pipeline one at a time, and only when a terminal operation (ToList, Count, First) pulls them. This is exactly the signals model — writes mark, reads compute — applied to sequences. It's why First() can short-circuit after one match, and why a query can stream a huge source without materializing it.

deferred.cs 
var query = numbers.Where(n => n % 2 == 0)   // deferred — nothing runs
                   .Select(n => n * n);       // deferred — still nothing

var list = query.ToList();   // NOW it enumerates — pulls every element through
var one  = query.First();    // pulls only until the first match, then stops

Build the query and see the per-stage counters stay at zero. Then ToList() to pull everything through, or First() to watch it short-circuit after the first element that passes:

ToList pulls all six through Where (×6) and the three survivors through Select (×3). First pulls only until 2 passes — Where ×2, Select ×1 — then stops. Laziness, counted.
↳ Angular Internals · Advanced SQL
The deferred pipeline is the signals graph in another costume — lazy, pull-driven, computed on demand. And it's why LINQ-to-SQL can defer: the query is a tree until you enumerate, so the provider can translate the whole thing into one SQL statement instead of fetching and filtering in memory.
Each LINQ operator loops the collection once, so chaining is slow.
Operators don't loop on their own — they compose one iterator that pulls each element through the whole chain a single time. Where().Select() is one pass, not two, and nothing runs until a terminal operator asks for elements.
A LINQ query is a value you can reuse freely.
It's a recipe, re-executed on every enumeration — iterate it twice and the source is walked twice (and any side effects happen twice). If you need the results as data, call ToList once. This is the same cold-vs-materialized distinction as observables.
MetaprogrammingExpertModule 08

Expression trees

A lambda can be two completely different things depending on its target type. Assigned to a Func<>, it's compiled code — a delegate you call. Assigned to an Expression<Func<>>, it's data — a tree describing the code, which you can inspect, rewrite, and Compile() into a delegate at runtime. This "code as data" is the engine behind ORMs (EF turns your Where(x => ...) into SQL by reading the tree), mappers, and validation libraries. It's the most powerful metaprogramming tool in the language.

expression-tree.cs 
Func<int,int> compiled = x => x * 2 + 1;          // executable delegate

Expression<Func<int,int>> tree = x => x * 2 + 1;   // DATA — an inspectable tree
Func<int,int> f = tree.Compile();                    // tree → real delegate at runtime
int y = f(5);   // 11

Below is the tree for x => x * 2 + 1. Compile() it to a delegate and invoke it; then rewrite the multiply operand and watch the tree change and demand a recompile — proof you're manipulating code as data:

Compile turns the tree into a callable delegate; f(5) = 5×2+1 = 11. Rewriting the operand mutates the data and invalidates the delegate — recompile to get 5×3+1 = 16.
↳ Mastering Entity Framework Core
This is exactly how EF Core works: your Where(u => u.Active) is an expression tree, and the provider walks it to emit SQL instead of running it in memory. The intro course's LINQ-to-SQL translation was this mechanism — a tree read by a query provider.
A lambda is always just an anonymous function.
Its target type decides everything. As a delegate it's compiled code; as an Expression<> it's a data structure the runtime hands you to inspect and transform. Same syntax, two entirely different runtime objects — and the second is what makes ORMs possible.
Source generators and expression trees are the same idea.
They run at different times. Source generators run at compile time, emitting extra C# with zero runtime cost. Expression trees are built and compiled at runtime. Both are metaprogramming, but one trades startup work for flexibility and the other trades flexibility for speed.
RuntimeExpertModule 09

The garbage collector

The GC is why you never free() in C# — and understanding its bet is how you write allocation-friendly code. The .NET collector is generational, built on a well-supported hypothesis: most objects die young. New allocations go into gen 0, which is small and collected often and cheaply. Objects that survive a collection get promoted to gen 1, and survivors of that to gen 2 — collected rarely, because long-lived objects tend to keep living. The practical upshot: short-lived garbage is nearly free, but allocating heavily still drives collection frequency, and objects that leak into gen 2 are expensive to reclaim.

the generational bet 
var temp = new byte[64];   // gen 0 — cheap, collected soon
// survives a gen-0 collection → promoted to gen 1 → then gen 2
// gen 0: collected constantly · gen 2: collected rarely (full GC)
GC.Collect(0);   // force a gen-0 collection (survivors promote, garbage is freed)

Dropped objects are reclaimed on the next collection; survivors move gen 0 → gen 1 → gen 2. Keep allocations short-lived and they stay in the cheap generation; leak them and they cost a full GC.
↳ Data Structures · Span<T> (m02)
This is why the allocation-free tricks matter: every object you don't allocate is gen-0 pressure you avoid. Spans, stackalloc, pooling, and struct locals keep hot paths out of the GC's way — the runtime payoff of the memory discipline from module 02.
The GC makes memory management something I never think about.
It removes manual free, not the cost of allocation. Allocation rate drives collection frequency, and collections pause your app. Senior .NET performance work is largely reducing allocations so the GC has less to do — exactly what modules 01 and 02 set up.
Calling GC.Collect() regularly keeps memory tidy.
It usually hurts — forcing full collections throws away the generational schedule the runtime tunes for you, and can promote objects that would have died in gen 0. The collector is smarter than a manual timer; the real lever is allocating less, not collecting more.
Memory modelExpertModule 10

The memory model

Concurrency's sharpest edge: count++ is not one operation. It's read, add one, write back — three steps — and when two threads run it on a shared field without synchronization, their reads and writes interleave and updates get lost. Each thread reads the same old value, both add one, both write back the same new value: two increments, one result. The fix is to make the update atomic: Interlocked.Increment performs read-modify-write as a single indivisible CPU instruction, or a lock serializes the section. For Snip's click counter under load, this is the difference between an accurate number and a silently wrong one.

counting-clicks.cs 
int hits = 0;

// two threads, 50 increments each:
hits++;                          // read-modify-write — RACY, loses updates
Interlocked.Increment(ref hits);   // atomic — never loses one

Run 100 increments across two threads. In unsafe mode the interleaving drops updates and the total comes up short; switch to Interlocked and every increment survives:

↳ Bits, Pointers & Overflow · SDLC
The race is a bit-level truth — the CPU's load/store of a word interleaved across cores — and its cost is a Lifecycle-grade production bug: intermittent, load-dependent, invisible in a single-threaded test. Knowing the memory model is how you prevent the heisenbug rather than chase it.
Incrementing an int is atomic — the CPU does it in one step.
The read-modify-write is three steps and can interleave across threads, losing updates. Even a single aligned read or write being atomic doesn't make ++ atomic. You need Interlocked or a lock — the hardware guarantee is narrower than the operation.
If it passed my tests, the concurrent code is correct.
Races are non-deterministic — a passing run proves nothing, because the losing interleaving simply didn't happen this time. Correctness here comes from the memory model (atomics, locks, immutability), not from green tests. This is the one place "it works on my machine" is actively dangerous.
ExpertModule 11

C# to machine code

Zoom all the way out and the pipeline is four stages. Your C# is compiled by Roslyn to IL — a portable, stack-based bytecode in the assembly. At runtime the JIT compiles each method's IL to machine code for your exact CPU, specializing generics, inlining, and eliding checks it can prove safe. And it all runs on the CLR, which manages the heap, the GC, and the type system. Every module in this course opened one stage of that stack:

And here is Snip's request hot path, rebuilt with the whole course — allocation-free parsing, a specialized cache, a non-blocking pipeline, and a correct counter:

snip-hotpath.cs 
async Task<Link?> ResolveAsync(string url) {
  ReadOnlySpan<char> code = url.AsSpan(5, 4);    // m02 · parse, 0 alloc
  if (_cache.TryGet(code, out var hit)) {              // m05 · List<Link> shares __Canon
    Interlocked.Increment(ref hit.Hits);            // m10 · atomic — never loses a click
    return hit;                                       // gen-0 friendly: nothing allocated
  }
  var link = await _db.FindAsync(code.ToString());     // m03 · state machine, no thread held
  await _clicks.Writer.WriteAsync(new Click(link));   // m04 · bounded — backpressure
  return link is { Active: true } ? link : null;      // m06 · property pattern
}
The machine, in your hands

Every line now traces to a mechanism you instrumented. AsSpan is a window you watched slice with zero allocation (m02). The cache is a value-type-specialized generic (m05). Interlocked is the atomic that never drops a count (m10). await is a state machine that releases its thread (m03), feeding a bounded channel that applies backpressure (m04). The is { Active: true } is an exhaustively-checkable pattern (m06). The syntax was never the language — the machine underneath was, and now you can see it, measure it, and make it fast.
