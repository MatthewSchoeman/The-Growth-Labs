BeginnerModule 00

Objects: state and behavior, together

Before any principle, one idea. Object-oriented programming organizes a program around objects: self-contained bundles that hold some data and the operations that act on that data. The data is the object's state (its fields); the operations are its behavior (its methods). That pairing — state and the behavior that guards it living in one place — is the atom everything else is built from.

A class is the blueprint; an object is one thing built from it. class Player describes what every player is and can do; new Player("Aria") creates one specific player with its own state. You can stamp out a thousand players from one class, each carrying its own name, health, and level.

jsclass Player {
  constructor(name) {
    this.name  = name;   // state: this player's own data
    this.hp    = 100;
    this.level = 1;
  }
  takeDamage(amount) {        // behavior: an operation on that state
    this.hp = Math.max(0, this.hp - amount);
  }
  heal(amount) { this.hp = Math.min(100, this.hp + amount); }
}

const aria = new Player("Aria");   // an object — one instance
aria.takeDamage(30);                // send it a message

The crucial mental model is the capsule. Picture the object as a container with an inside and an outside. The public interface — its methods — is the outside: the set of messages other code is allowed to send. The private state is the inside, and ideally nobody reaches in directly; they ask the object to do something, and the object changes its own state. Hold that picture — every pillar in this course is a different way of drawing or respecting that boundary.

Here's a live one. This Player object exposes a few methods on its surface and keeps its state inside. Press a method and watch the object act on itself — the state never changes except through the behavior the class defined:

Classes and objects are the same thing.

Pillar 1BeginnerModule 01

Encapsulation — guarding the inside

The first pillar, encapsulation, is the discipline of hiding an object's state behind a controlled interface, so the object alone decides how its data changes. The point isn't secrecy for its own sake — it's protecting invariants: rules that must always be true. A bank balance must never go negative past an overdraft limit; a thermostat must never be set beyond its safe range. If outside code can write the field directly, nothing can guarantee those rules.

The mechanism is access control: fields are made private, and the only way to read or change them is through public methods that enforce the rules. deposit() and withdraw() become the gatekeepers; they can reject a bad operation before it corrupts the object.

jsclass BankAccount {
  #balance = 0;                 // # = private: unreachable from outside

  deposit(amount) {
    if (amount <= 0) throw new Error("must be positive");
    this.#balance += amount;       // the object guards its own state
  }
  withdraw(amount) {
    if (amount > this.#balance) throw new Error("insufficient funds");
    this.#balance -= amount;
  }
  get balance() { return this.#balance; }  // read-only window
}
Invariant

An invariant is a condition an object promises to keep true for its whole life — here, "the balance is never negative." Encapsulation is what lets the object keep that promise: because every path that changes the balance runs through a method, every change can be checked. Expose the field and the promise is void — any line anywhere can break it.

This is easiest to feel by trying to break it. Below are two versions of the same account. On the right, it's encapsulated — the same operations must go through deposit/withdraw, which refuse anything illegal. Try to drive each one negative:

Encapsulation means adding a getter and setter for every field.
Thoughtless get/set pairs just re-expose the field with extra typing. Real encapsulation means exposing operations (withdraw), not raw data — and often not offering a setter at all.
Private fields are about hiding secrets from other programmers.
They're about protecting invariants and shrinking the surface that can break them. Fewer ways to change state means fewer places a bug can come from. It's a correctness tool, not a privacy one.
Pillar 2BeginnerModule 02

Abstraction — programming to a contract

If encapsulation hides an object's data, abstraction hides its implementation. It lets you work with what something does while ignoring how it does it. You use a List without knowing it's an array or a linked structure; you call save() without knowing whether it writes a file or a database. The "what" is a contract — usually expressed as an interface — and the "how" is some concrete class that fulfills it.

The payoff is substitutability. When your code depends only on the contract, any implementation that honors it can be dropped in without the calling code changing a line. This single idea — program to an interface, not an implementation — is the seam along which flexible systems are built.

js// the contract: what a payment method must be able to do
class PaymentMethod {
  pay(amount) { throw new Error("not implemented"); }
}

// interchangeable implementations — each its own "how"
class CreditCard extends PaymentMethod { pay(a){ /* call card network */ } }
class PayPal     extends PaymentMethod { pay(a){ /* call PayPal API */ } }

// the caller depends on the contract, not the concrete type
function checkout(method, total) { method.pay(total); }

Notice that checkout never mentions CreditCard or PayPal. It speaks only the contract — "something I can call pay on." Add ApplePay tomorrow and checkout doesn't change. The interface is a stable boundary that lets the two sides evolve independently.

Watch the swap happen. The caller below is fixed; it only knows the PaymentMethod contract. Choose which concrete implementation sits behind the interface and call it — the same line of calling code drives completely different machinery:

The calling code speaks only the contract. Any implementation that honors it slots in unchanged.
Abstraction means making things abstract and complicated.
The opposite — it removes detail so you can reason at a higher level. A good abstraction is simpler to use than the thing it hides, exposing intent and concealing mechanism.
An interface is just an extra file to maintain.
It's the seam that makes parts swappable and testable. Depend on a concrete class and you weld your code to one implementation; depend on a contract and you can replace, mock, or extend it freely.
Pillar 3IntermediateModule 03

Inheritance — the is-a relationship

Inheritance lets one class build on another. A subclass (or child) extends a superclass (parent), automatically receiving its state and behavior, and can add new members or override existing ones. The relationship it models is is-a: a Dog is an Animal, so it inherits everything an animal can do and specializes the parts that differ.

When you call a method on an object, the runtime performs method resolution: it looks for that method on the object's own class first, and if it isn't there, walks up the inheritance chain until it finds one. This is how an overridden speak() on Puppy wins over the one on Dog, while an un-overridden breathe() falls through to Animal.

jsclass Animal {
  breathe()  { return "…breathing"; }
  describe() { return "an animal"; }
}
class Dog extends Animal {
  speak()    { return "Woof"; }
  describe() { return "a dog"; }   // override
}
class Puppy extends Dog {
  speak() { return "Yip!"; }     // override again
}

Trace it yourself. Call a method on a Puppy below and watch resolution start at Puppy and climb the chain until it lands on the class that actually defines the method:

puppy.
Resolution starts at the object's own class and climbs until it finds the method. The first match wins — that's how overriding works.
Inheritance is the tightest coupling there is

A subclass depends on its parent's internals, not just its interface. Change the base class and every descendant can break in surprising ways — the fragile base class problem. Inheritance also locks a type into one axis of variation forever. These costs are exactly why the next two modules push you toward interfaces and composition for most reuse, reserving inheritance for genuine, stable is-a hierarchies.

Inheritance is the main way to reuse code in OOP.
It's one way, and often the worst. It couples child to parent permanently. Most reuse is better served by composition (Module 05) — assembling behavior from parts you can swap.
Deep inheritance hierarchies are a sign of good design.
Usually the reverse. Each level adds coupling and cognitive load; behavior gets smeared across many files. Favor shallow hierarchies — most well-designed ones are one or two levels deep.
Pillar 4IntermediateModule 04

Polymorphism — one message, many forms

Polymorphism — literally "many forms" — is the ability to send the same message to different objects and have each respond in its own way. You call .area() on a shape; a circle computes πr² and a square computes s², but the calling code issues one identical call. The runtime picks the right method based on the object's actual type. This is dynamic dispatch, and it's the feature that makes abstraction pay off.

The power is what it lets you delete: the giant switch. Without polymorphism, code that handles many types is riddled with if (shape.type === "circle") … else if (…) …, and every new type means editing every such block. With polymorphism, each type carries its own behavior, and the caller just says shape.area(). Adding a new shape touches nothing in the existing code.

js// WITHOUT polymorphism — the caller must know every type
function area(s) {
  if (s.type === "circle")   return Math.PI * s.r * s.r;
  if (s.type === "square")   return s.side * s.side;
  // …edit this function for every new shape, forever
}

// WITH polymorphism — each type owns its behavior
class Circle { area(){ return Math.PI*this.r**2; } }
class Square { area(){ return this.side**2; } }
total += shape.area();    // one call dispatches to the right method

Here is dynamic dispatch made visible. The loop below issues a single call — shape.area() — to a collection of different shape objects. Watch the very same message resolve to each type's own implementation, one after another:

Polymorphic dispatch
the same call, sent to different types, runs different code
for (const shape of shapes) result = shape.area();
send message:
One line of calling code, four different implementations. Add a fifth shape and this loop never changes.
Polymorphism is just method overriding.
Overriding is the mechanism; polymorphism is the payoff — that a caller can treat many types uniformly through a shared interface and let each supply its own behavior. The win is code that's open to new types without modification.
A big type-switch and polymorphism are equivalent.
They produce the same output but differ in change cost. The switch is a single choke point every new type must edit; polymorphism distributes behavior so new types are purely additive. That difference is the Open/Closed Principle (Module 07).
Core principleIntermediateModule 05

Composition over inheritance

The most repeated piece of design advice in all of OOP: favor composition over inheritance. Where inheritance models is-a, composition models has-a — an object gains abilities by holding other objects that provide them, rather than by being a subtype. A Car has an Engine; a game character has a set of capabilities. You build behavior by assembling parts.

Why prefer it? Because inheritance forces every variation onto a single tree, and trees explode. Suppose characters can walk, fly, and swim in any combination. With inheritance you need a class for each combination — WalkingFlyer, SwimmingFlyer, WalkingSwimmingFlyer — and the count doubles with every new ability. With composition, each ability is a component you mix in freely, and three abilities are just three small, independent parts.

js// inheritance: a rigid tree, one class per combination — it explodes
class Walker {} class Flyer {} class WalkingFlyer {} // …and on, and on

// composition: abilities are parts you assemble
const walk = () => ({ move: () => "walking" });
const fly  = () => ({ soar: () => "flying" });

function createHero() {
  return Object.assign({ name: "Kestrel" }, walk(), fly()); // has-a, mixed in
}

Feel the difference. On the left, toggle capabilities onto a character by composition — each is an independent part, and any combination just works. On the right is the inheritance tree you'd otherwise have to write: a separate class for every combination, growing exponentially:

Build by composition
mix capabilities freely — vs a class for every combination
✓ composition — toggle has-a parts
✕ inheritance — a class per combination
Composition adds abilities linearly — one part each. Inheritance needs a class for every subset, doubling with each new ability.
"Favor composition" means never use inheritance.
It means reach for composition first. Inheritance is still right for genuine, stable is-a hierarchies with shared substitutable behavior. The rule guards against using it as a default reuse hammer.
Composition is more code, so it's worse.
Slightly more wiring, far less coupling. Parts are independently testable, swappable, and reusable across unrelated objects — and you sidestep the combinatorial explosion and fragile base classes entirely.
AdvancedModule 06

Coupling & cohesion

Two metrics quietly decide whether a codebase is a pleasure or a nightmare to change. Coupling measures how much one module depends on others — how many threads connect it to the rest of the system. Cohesion measures how focused a single module is — whether its parts all serve one clear purpose. The eternal goal is loose coupling and high cohesion: modules that each do one thing well and lean on each other as little as possible.

The reason coupling matters so much is change propagation. When modules are tightly coupled, touching one forces you to touch its neighbors, which forces their neighbors — a small edit triggers a cascade of changes and a cascade of risk. When coupling is loose, changes stay local: you modify a module's insides and nothing outside notices, because everything outside depended only on its stable interface. Every pillar so far — encapsulation, abstraction, polymorphism, composition — is ultimately a tool for keeping coupling low.

The two together

High cohesion causes loose coupling. When each module has a single, well-defined job, it needs little from others and exposes a small, stable surface. A "god module" that does everything is both incohesive and highly coupled — everything depends on it, and it depends on everything. Fix cohesion and coupling tends to fall out for free.

In a tightly coupled system, one change drags most of the codebase with it. Loosen the coupling and changes stay contained.
Splitting code into more files reduces coupling.
Only if you also cut the dependencies between them. Ten files that all reference each other are as coupled as one big file — worse, because the coupling is now hidden. Coupling is about references, not file count.
Some coupling is bad, so zero coupling is the goal.
Modules must collaborate, so some coupling is necessary. The goal is to make it loose and intentional — through narrow interfaces — not to eliminate it. Zero coupling means the parts don't talk, which means they don't form a system.

The SOLID principles

SOLID is five principles of class design that together push code toward loose coupling, easy extension, and safe change. They aren't laws; they're pressure in a direction. Internalize them and good structure starts to feel obvious. Each one has a memorable failure mode and a clean fix.

Tap each letter to see the principle, a common violation, and the fix:

The SOLID explorer
five principles — each a violation and its fix
Each principle reduces a specific kind of coupling. Together they make classes small, focused, and open to change.

None of these stands alone. Single Responsibility keeps classes cohesive; Open/Closed and Liskov make polymorphism safe; Interface Segregation and Dependency Inversion keep the dependencies between classes loose and pointed at abstractions. They're five views of the same goal you've been circling all course: localize change.

SOLID means more classes and interfaces, so more is always better.
Applied blindly, SOLID can over-engineer simple code into a maze of indirection. The principles solve real coupling pains — apply them when you feel the pain, not preemptively to every two-line class.
The Open/Closed Principle means you can never edit existing code.
It means designing so that common kinds of change (a new type, a new rule) are additive rather than invasive. You still fix bugs and refactor; you just avoid architectures where every new case forces edits to shared, fragile code.
AdvancedModule 08

DRY, KISS & YAGNI

Three short slogans carry an enormous amount of practical wisdom about keeping code simple and change cheap.

DRY — Don't Repeat Yourself. Every piece of knowledge should have a single, authoritative home. When the same rule is copy-pasted in five places, a change means finding and fixing all five — and the bug is the one you missed. Centralize it, and there's one place to change and one place to be right.

js// repetitive — the tax rule lives in three places
const a = price + price * 0.2;
const b = total + total * 0.2;   // change the rate → hunt them all down

// DRY — one source of truth
const withTax = (n) => n * 1.2;
const a = withTax(price), b = withTax(total);

KISS — Keep It Simple. Between a clever solution and a clear one, choose clear. Code is read far more often than it's written, and the next reader (often you) pays for every bit of cleverness. Simplicity isn't a lack of sophistication; it's the discipline of solving the problem with the least machinery that works.

YAGNI — You Aren't Gonna Need It. Don't build for imagined futures. The configurable, pluggable, generalized framework you're tempted to add "in case we need it" usually never gets used — and meanwhile everyone pays to carry and understand it. Build what today's requirement needs; generalize when a second real case actually arrives.

When DRY turns on you

DRY is about duplicated knowledge, not duplicated text. Two pieces of code that look identical today but change for different reasons are not true duplication — merging them couples two unrelated things, and the day their rules diverge you'll tear the abstraction apart. As the saying goes: a little duplication is far cheaper than the wrong abstraction. Wait until the pattern is undeniable before extracting it.

DRY means no two lines of code should ever look the same.
It means no piece of knowledge should be duplicated. Coincidentally-similar code that evolves separately should stay separate — forcing it together is the "wrong abstraction" that's harder to undo than the duplication was.
YAGNI is an excuse to ignore the future and write shortsighted code.
It targets speculative features, not good design. Keep the code clean and easy to change (that's the whole course) — then changing it later when you do need something is cheap. YAGNI trusts refactoring over fortune-telling.
AdvancedModule 09

Heuristics & guidelines

Below the big principles sits a layer of smaller rules of thumb — heuristics that nudge individual lines toward clarity and low coupling. None is absolute, but each catches a recurring mistake.

Tell, Don't Ask

Don't pull data out of an object, make a decision, and push a result back. Tell the object what you want and let it decide. Asking for an object's state and acting on it elsewhere drains behavior out of the object — re-creating the "data over here, logic over there" split that encapsulation exists to prevent.

js// ASK: reach in, decide outside, write back
if (account.getBalance() >= amount) {
  account.setBalance(account.getBalance() - amount);
}                       // the rule lives outside the object
js// TELL: hand it the intent, let it enforce its own rules
account.withdraw(amount);   // the object owns the decision

The Law of Demeter don't talk to strangers

A method should only talk to its immediate collaborators — its own fields, its parameters, objects it creates — not reach through them to distant objects. Chains like order.getCustomer().getAddress().getCity() are train wrecks: they couple you to the entire object graph, so a change three objects away breaks your code. Ask your direct neighbor for what you need.

js// reaches through 3 objects — coupled to all of them
const city = order.getCustomer().getAddress().getCity();
js// ask the immediate object; let it delegate internally
const city = order.getShippingCity();   // one dot, one dependency

A few more worth keeping in your pocket: Command–Query Separation — a method should either do something or answer something, never both, so callers can trust that asking has no side effects. Fail fast — validate inputs at the boundary and throw immediately, rather than letting bad data slip deep into the system where the eventual error makes no sense. Prefer immutability — objects that never change after construction can't be corrupted, can't surprise you, and are safe to share freely.

The Law of Demeter bans all method chaining.
Fluent chains that return the same object (a builder, a stream pipeline) are fine. Demeter targets chains that navigate across different objects' internals — coupling you to a structure you don't own.
These heuristics are unbreakable rules.
They're defaults that pay their way. Each prevents a common, specific pain. Break one deliberately when you have a concrete reason — but know which one you're breaking and why.
ExpertModule 10

Code smells & refactoring

A code smell is a surface symptom that often points to a deeper design problem. Smells aren't bugs — the code works — but they signal that change will be harder than it should be. Learning to notice them is half the craft; the other half is refactoring: improving a program's internal structure without changing its behavior, in small safe steps backed by tests.

The most common smells, and what each is really telling you:

SmellWhat you seeThe deeper problem
Long MethodA function that scrolls off the screenIt's doing several jobs; extract them into named methods.
Large Class / God ObjectOne class that knows and does everythingLow cohesion; split by responsibility (Module 07's S).
Feature EnvyA method that uses another object's data more than its ownThe behavior is in the wrong class; move it to the data.
Primitive ObsessionStrings and ints standing in for real concepts (money, email)A missing type; introduce a small value object.
Shotgun SurgeryOne change forces tiny edits across many filesA responsibility is scattered; gather it into one place.
Long Parameter ListA method taking six, seven, eight argumentsHidden data clump; pass an object, or rethink the call.

Here's a small but classic refactor. The "before" is a long method bristling with feature envy and primitive obsession; the "after" pushes behavior to where the data lives and names the concepts. Same behavior, far easier to change:

js// long method + feature envy: pokes at order's internals
function printReceipt(order) {
  let total = 0;
  for (const line of order.lines)
    total += line.qty * line.unitPrice;        // order's job, not ours
  if (order.customer.type === "vip") total *= 0.9; // reaching in
  console.log("Total: $" + total.toFixed(2));
}
js// behavior moved to the objects that own the data
class Order {
  total() {
    const sum = this.lines.reduce((t, l) => t + l.subtotal(), 0);
    return this.customer.applyDiscount(sum);   // tell, don't ask
  }
}
function printReceipt(order) {
  console.log(`Total: $${order.total().toFixed(2)}`);
}
Refactor on green

The discipline that makes refactoring safe is tests. With a passing test suite, you change structure in tiny steps and re-run after each — if the bar stays green, behavior is preserved. Refactoring without tests is just editing and hoping. This is why "make it work, make it right, make it fast" is an order: get a passing test, then improve the design underneath it.

Refactoring means rewriting a messy module from scratch.
That's a rewrite — high-risk and behavior-changing. Refactoring is a sequence of tiny, behavior-preserving transformations, each verified by tests. The system stays working the entire time.
A code smell means the code is wrong and must be fixed now.
A smell is a hint to look closer, not a verdict. Sometimes the simple, slightly-smelly version is the right call. Fix smells when they're getting in the way of a change you're actually making.
ExpertModule 11

Pragmatic OOP

You now have the pillars and the principles. The final lesson is the one that ties them together: they are all means to an end, and the end is always changeability and clarity. Encapsulation, abstraction, SOLID, low coupling — none is valuable for its own sake. Each earns its keep only when it makes the code easier for the next person to understand and safer to change. When a "principle" makes code harder to follow, you've misapplied it.

Which means judgment outranks dogma. A three-line script needs no interfaces. A data-transformation pipeline may be clearer as plain functions than as a class hierarchy — OOP is one paradigm, not the only one, and the best engineers mix object-oriented, functional, and procedural styles to fit the problem. Reach for objects when you have state and the behavior that guards it, when you're modeling a domain of interacting things, when polymorphism lets you add cases without editing old code. Reach elsewhere when you don't.

The whole course in one line

Bundle state with the behavior that protects it, hide what others don't need, depend on stable interfaces instead of concrete details, and keep each piece focused — so that change stays small, local, and safe. Every pillar and principle here is a different angle on that one sentence.

The deepest skill isn't knowing the principles — it's developing the taste for when each one helps. That taste comes from reps: writing code, living with your decisions, feeling which ones made the next change easy and which made it painful, and adjusting. The principles give you the vocabulary; experience teaches you the music.

Practice · design judgment

1. You have a Report class that builds data, formats it as HTML, and emails it. Which SOLID principle does this violate, and how would you fix it?

▸ Show answer

Single Responsibility — it has three reasons to change (data logic, formatting, delivery). Split into a Report (data), a ReportFormatter, and a Mailer. Each can now change independently, and you can format without emailing or test the data logic in isolation.

2. You see a.getB().getC().doThing() in a method. What's the smell, and what's the fix?

▸ Show answer

A Law of Demeter violation (a "train wreck"). The method is coupled to the structure of B and C. Add a method to a that does the work and delegates internally — a.doThing() — so callers depend only on a.

3. (judgment) You notice two functions with nearly identical bodies. Should you immediately extract a shared one?

▸ Show answer

Not necessarily. Ask why they're the same. If they encode the same rule, extract (DRY). If they're similar by coincidence and will change for different reasons, leave them — premature extraction couples them and the wrong abstraction costs more than the duplication. Wait for a third occurrence before trusting the pattern.

Now you can shape the code, not just write it

You started with a single object — state and behavior, together — and built all the way to the principles that keep a large system soft enough to change. The syntax of classes was never the point; the thinking was, and now it's yours.

From here, the growth is in reps: write code, notice which decisions made the next change easy, and refactor toward the principles when the pain shows up. Good design isn't a destination you reach — it's a direction you keep choosing.
