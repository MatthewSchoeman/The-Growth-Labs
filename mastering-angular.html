BeginnerModule 00

What Angular is

Most front-end tools are libraries: a rendering core, and a shopping trip for everything else — router, forms, HTTP, state, build config, all chosen, versioned, and glued by you. Angular is a framework in the full sense: the platform ships whole. One team designs the router, the forms, the HTTP client, the DI system, and the CLI to work together; one ng update moves them together. The trade is explicit — you accept Angular's opinions, and in return the architecture decisions arrive pre-made and coherent.

Modern Angular is also not the Angular of old blog posts. The renaissance versions made standalone components the default (no NgModule ceremony), added signals as a first-class reactivity primitive, and replaced asterisk directives with built-in control flow. It's TypeScript-first, decorator-driven, and DI-native — which is why it reads like home if you arrived through the C# track:

terminal · then app.component.ts 
# the CLI scaffolds, serves, generates, tests, and builds
ng new snip-dashboard      ng serve      ng generate component link-list

@Component({
  selector: 'snip-root',
  template: `<h1>Snip · {{ links().length }} links</h1>`
})
export class AppComponent {
  links = signal([{ code: '4C92', clicks: 812 }]);
}   // standalone by default — a component is a class + a decorator
↳ Front-End Engineering · Mastering C#
This is the sequel: every concept from Front-End Engineering — the loop, components, keys, one-way flow, server state, routing — reappears here wearing Angular syntax. And if Mastering C# was your track, notice the accent: decorators ≈ attributes, DI ≈ the ASP.NET container, strict types everywhere. Angular is the front end .NET developers recognize.
Angular is the old, heavyweight, dying one.
That reputation describes a version, not the framework. Modern Angular — standalone components, signals, built-in control flow, a rebuilt compiler — is lean and current, and it still ships the whole platform: the thing teams actually pay for is fewer glue decisions and one coherent upgrade path.
You must learn NgModules before anything works.
Standalone is the default: a component declares its own imports and just works. NgModules are legacy reading knowledge for older codebases — this course teaches the modern shape and flags the old one where you'll meet it.
ComponentsBeginnerModule 01

Components & bindings

An Angular component is a TypeScript class with a template, and bindings are the wires between them. There are exactly four directions to learn: {{ interpolation }} renders class data as text; [property] pushes data into an element; (event) pulls user actions out; and [(two-way)] — the "banana in a box" — is the two combined. Data down, events up: the contract from the front-end course, now with syntax.

link-editor.component.ts 
@Component({ template: `
  <input [(ngModel)]="code">                <!-- two-way: in and out -->
  <p>preview: snip.ly/{{ code }}</p>        <!-- interpolation -->
  <button [disabled]="!code"                <!-- property binding -->
          (click)="save()">save</button>   <!-- event binding -->
` })
export class LinkEditor { code = ''; save() { /* … */ } }

All four wires, live — type in either input and watch the two-way pair stay in lockstep while every dependent binding follows:

Both inputs bind the same code — typing in one updates the model, and the model updates the other. [( )] is [property] plus (event), nothing more mystical.
Templates can hold real logic — it's all JavaScript anyway.
Template expressions are a deliberately small language: no assignments, no new, no chained side effects. That's a feature — logic lives in the class (or a computed signal), templates stay declarative and inspectable, and the compiler can check every binding.
Two-way binding is framework magic to be avoided.
It's sugar with a receipt: [(x)] desugars to [x] + (xChange) — data down, event up, exactly the discipline you already learned. Use it where it reads well (form controls); expand it whenever you need the two halves separately.
TemplatesBeginnerModule 02

The template toolkit

Real templates branch, loop, and format. Modern Angular builds control flow into the template language: @if / @else for branches, @for for lists — with a required track expression — and @switch for enumerations. Alongside them, pipes transform values at the binding site (clicks | number, created | date), and attribute directives like [class.sel] decorate elements. You'll still read the legacy asterisk forms (*ngIf, *ngFor) in older code; you'll write the new ones.

link-list.component.html 
@if (links().length) {
  @for (l of links(); track l.code) {          <!-- identity = the keys lesson -->
    <article [class.sel]="l.code === sel()">
      snip.ly/{{ l.code }} · {{ l.clicks | number }}
      @switch (l.status) {
        @case ('live')    { <span class="ok">live</span> }
        @case ('paused')  { … }  @default { … }
      }
    </article>
  }
} @else { <p>no links match — create one</p> }
<!-- @defer (on viewport) { <heavy-chart/> } — lazy templates, one keyword -->

↳ Front-End Engineering
That mandatory track expression is Module 06 of the Front-End course — keyed reconciliation — promoted from best practice to required syntax. Angular refuses to let you write the unkeyed list that turns one move into a full rebuild.
*ngIf and *ngFor are how you write Angular.
They're how you read yesterday's Angular. Built-in control flow is the modern default — no directive imports, better type narrowing inside branches, and a faster list algorithm under @for. Learn both directions of the translation.
Pipes are just formatting sugar — do it in the class.
Pipes are memoized transforms at the binding site: a pure pipe re-runs only when its input changes, and reusable ones (date, number, custom) keep formatting out of every component class. The async pipe in Module 06 is a pipe doing subscription management.
SignalsIntermediateModule 03

Signals

Here is modern Angular's heart. A signal is a value that knows who reads it. signal() holds writable state; computed() derives from other signals and tracks its dependencies automatically; effect() runs side effects when its dependencies change. Update a signal and only its actual dependents recompute — the fine-grained version of UI = f(state), and the engine templates now run on.

dashboard.state.ts 
const links  = signal<Link[]>([…]);            // writable state
const filter = signal<'all' | 'paid'>('all');

const visible = computed(() =>                  // derives; tracks deps itself
  links().filter(l => filter() === 'all' || l.paid));
const count = computed(() => visible().length);

effect(() => console.log(`showing ${count()}`)); // side effects only

links.update(ls => [...ls, newLink]);            // .set / .update to write

↳ Front-End Engineering
Signals are UI = f(state) from the parent course with the dependency edges made explicit: instead of re-rendering the whole view and diffing, the graph knows exactly which computation each write invalidates. Same formula, sharper pencil.
Signals replace RxJS — forget Observables.
They divide the work: signals hold state (current values, synchronous, always readable); RxJS models events over time (streams, cancellation, backpressure — Module 06). toSignal/toObservable bridge them. Two tools, two shapes of problem.
effect() is the place to update other state.
Effects are for the outside world — DOM, logging, storage. Deriving state from state is computed()\u2019s job; writing signals inside effects builds hidden cascades and feedback loops the graph was designed to eliminate.
I/OIntermediateModule 04

Component I/O

Components talk through a typed, signal-based surface: input() receives data from the parent (with input.required() when it isn't optional), output() emits events upward, and model() is a two-way signal — both sides can read and write it, and both stay in sync through an explicit change event. The classic @Input()/@Output() decorators do the same jobs in older code.

link-card.component.ts 
export class LinkCard {
  link     = input.required<Link>();          // data ↓ (a signal)
  selected = input(false);
  remove   = output<string>();                // events ↑
  editing  = model(false);                     // two-way signal ⇅
}
<!-- parent: --> <link-card [link]="l" (remove)="drop($event)" [(editing)]="edit"/>

Select flows up as an output and returns as an input; remove shrinks the parent\u2019s list; and the editing checkboxes are one model() worn by two owners — flip either.
Any two components that share data need a shared service.
Parent and child already have a channel: inputs down, outputs up. Services (Module 05) are for state that\u2019s genuinely app-wide or crosses distant branches — reaching for one first turns local wiring into global coupling.
model() is just old ngModel with a new hat.
It\u2019s a two-way signal: typed, decorator-free, readable as a value and writable from either side, with the change event generated for you. The banana-in-a-box syntax survives; the machinery underneath is the Module 03 graph.
DIIntermediateModule 05

Services & DI

Logic that outlives a component — API calls, shared state, cross-cutting helpers — lives in a service: a plain class marked @Injectable. Components never construct services; they ask for them with inject(), and Angular's injector tree supplies an instance. providedIn: 'root' means one app-wide singleton; providing at a component instead scopes a fresh instance to that subtree. A service holding signals is modern Angular's simplest shared store:

snip-store.service.ts 
@Injectable({ providedIn: 'root' })          // one instance, whole app
export class SnipStore {
  readonly links = signal<Link[]>([]);
  add(l: Link)  { this.links.update(ls => [...ls, l]); }
}
// any component, anywhere:
store = inject(SnipStore);                   // ask the tree, don't new()

Prove the tree's behavior — two components inject the store and share every write, until you re-provide it locally and one of them gets a private copy:

inject(SnipStore) → links = 0
both components asked the tree — the root answered with the same singleton
Same badge, same instance: a write from either side is visible to both. Scoping the provider is a deliberate act — and the badges will say so.
↳ ASP.NET Core
This is the same container you configured in Mastering ASP.NET Core: providedIn: 'root' is a singleton lifetime, component-level providers are scoped lifetimes, and constructor/inject() resolution is the identical mental model. One DI brain, both ends of the wire.
DI is enterprise ceremony — just import and new().
new() welds your component to a concrete class. Injection is why Angular code swaps and tests so cleanly — hand the tree a fake SnipStore and every consumer uses it, no component edited. The ceremony is one function call.
providedIn: 'root' is just a global variable.
It\u2019s a singleton with an owner: lazily created, tree-shaken if unused, replaceable in tests, and overridable per-subtree — everything a mutable global isn\u2019t. The lab\u2019s local provider is that override in action.
RxJSAdvancedModule 06

RxJS & HttpClient

Angular's HttpClient returns an Observable — RxJS's stream type. Where a promise is one future value, an observable is any number of values over time, lazy until subscribed and — crucially — cancellable. Streams compose through operators, and one pipeline is the canonical demonstration: the typeahead. Keystrokes are a stream; debounceTime quiets the burst, distinctUntilChanged drops repeats, and switchMap maps each query to an HTTP call while cancelling the previous one — the stale-response bug, deleted by an operator:

search.component.ts 
results$ = this.query.valueChanges.pipe(
  debounceTime(250),                        // the perf-course lesson
  distinctUntilChanged(),
  switchMap(q => this.http.get<Link[]>(`/api/links?q=${q}`))
);                                          // switchMap cancels the stale call
<!-- template: no subscribe, no leak -->
@for (l of results$ | async; track l.code) { … }

↳ ASP.NET Core · Advanced SQL
The endpoint being debounced into is yours: ASP.NET routing the request, the Advanced SQL queries answering it. HttpClient is the typed, testable, interceptable last leg of a pipeline this curriculum built end to end.
Observables are just promises with extra steps.
Promises can\u2019t cancel, can\u2019t emit twice, and start eagerly. The typeahead needs all three fixed at once — which is why switchMap has no promise translation. Streams are a different shape of value, not a wrapper on the old one.
Call subscribe(), stash the data on a field, done.
Every manual subscribe is a leak waiting for a forgotten unsubscribe. Prefer the async pipe or toSignal() — subscription and teardown handled — and when you must subscribe, tie it to takeUntilDestroyed() so the component\u2019s death is the stream\u2019s.
RoutingAdvancedModule 07

Routing

Angular ships the router. Routes are data — an array mapping paths to components — and everything else follows: <router-outlet> marks where views render, routerLink navigates, :code declares a parameter (bindable straight to a component input()), guards vet navigation, and loadComponent makes any route a lazily-loaded bundle in one line:

app.routes.ts 
export const routes: Routes = [
  { path: 'links',       component: LinkListComponent },
  { path: 'links/:code', component: LinkDetailComponent },  // :code → input()
  { path: 'settings',
    canActivate: [() => inject(Auth).isAdmin() || inject(Router).parseUrl('/links')],
    loadComponent: () => import('./settings.component') },   // lazy bundle
  { path: '', redirectTo: 'links', pathMatch: 'full' }
];
Routes, params & a guard
the config above, running
Click a link row and the :code parameter arrives in the detail component as an input. Try /settings — the guard has opinions.
Route guards are the app's security layer.
Guards are UX — they keep users out of screens that would only error. Security lives on the server (your ASP.NET auth), which must reject the request no matter what the client renders. A guard is a courtesy; the API is the law.
Lazy loading is an optimization for later.
Route-level splitting is one line (loadComponent) and permanently shrinks the first load — the settings bundle simply isn\u2019t downloaded until someone goes there. Deferred cost, immediate architecture: do it from the first route.
FormsAdvancedModule 08

Reactive forms

Angular offers two form systems. Template-driven forms lean on ngModel — fine for the trivial. Reactive forms build the form as an object in the class: FormControls composed into a FormGroup, validators attached as functions, validity computed continuously, and every keystroke available as an observable through valueChanges. Typed, testable without a DOM, and composable — the moment a form has rules, this is the system:

create-link.component.ts 
form = new FormGroup({
  code: new FormControl('', [ Validators.required,
    Validators.minLength(3), Validators.pattern(/^[a-z0-9-]+$/) ]),
  url:  new FormControl('', [ Validators.required,
    Validators.pattern(/^https?:\/\//) ]),
});                                     // typed: form.value.code is string
// the form is also a stream:
this.form.valueChanges.subscribe(v => …)   // Module 06, again
Validity as a model
the form object drives everything you see
INVALID
form.valueChanges — the form as a stream
Errors appear per rule, the pill tracks group validity, submit follows the pill — one form object, and the template merely reflects it. UI = f(form).
Template-driven forms are simpler — always start there.
Simpler until the first real rule. Reactive forms put the model in the class: typed values, validators as plain functions, testable with no DOM, and valueChanges feeding straight into Module 06 pipelines. Rules present → reactive.
Disabling the submit button is the validation.
The button follows the validity model; it isn\u2019t the model. Show errors when fields are touched, keep the messages next to their rules — and validate on the server regardless, because Module 07 already told you who enforces the law.
Change detectionAdvancedModule 09

Change detection

The hardest Angular question is the quietest: after something happens, how does the screen know? Classic Angular answers with Zone.js — a patch over every async API that says "something may have changed" — after which Angular checks every component's bindings, top to bottom. Correct, and wasteful. OnPush tightens it: a component is checked only when its inputs change, an event fires inside it, or a signal it reads updates — so a change checks a path, not the world. Signals finish the journey: with fine-grained dependencies, Angular can run zoneless, updating exactly the views that read what changed.

The lifecycle hooks live in this rhythm too: ngOnInit after inputs first arrive, ngOnDestroy (or a DestroyRef) for teardown — which is where takeUntilDestroyed from Module 06 plugs in.

stats-bar.component.ts 
@Component({ changeDetection: ChangeDetectionStrategy.OnPush, … })
export class StatsBar {
  links = input.required<Link[]>();   // new reference in → check me
  count = computed(() => this.links().length);  // signal read → tracked
}
// the zoneless future, one provider:
bootstrapApplication(App, { providers: [provideZonelessChangeDetection()] });

Default checks all six components for any event anywhere. OnPush checks the event\u2019s path. A signal under fine-grained reactivity touches exactly one. That\u2019s the whole performance story, counted.
↳ Big O
Change detection is a complexity story: Zone-default is O(components) per event; OnPush prunes to a path; signals approach O(affected views). The Big O instinct — count the work as n grows — is exactly how you reason about a 500-component app staying at 60 fps.
Zone.js watches my data and knows what changed.
It knows only that time passed: a click, a timer, an XHR completed. It cannot see what changed — hence "check everything." Signals invert this: the data itself reports its readers, which is why they make zoneless possible.
OnPush randomly breaks working apps.
OnPush breaks hidden mutation: editing an array in place and expecting the child to notice. New references for inputs and signals for state — the immutability discipline from the front-end course — and OnPush is not a risk but a guarantee.
ExpertModule 10

Architecture

Angular's opinions extend to the shape of the app. The convention that scales: organize by feature, not by file type — each feature a folder of standalone components behind a lazy route, split internally into smart components (inject services, own state) and presentational ones (inputs in, outputs out, no injections). Shared plumbing — the store from Module 05, interceptors, guards — lives in core/; dumb reusable pieces in shared/.

src/app/ · and one interceptor 
core/       snip-store.ts · auth.interceptor.ts · api.service.ts
features/
  links/    links.routes.ts (lazy) · link-list · link-card · link-detail
  settings/ settings.routes.ts (lazy, guarded)
shared/     badge.component.ts · time-ago.pipe.ts

// cross-cutting HTTP concerns are functions in the DI pipeline:
export const authInterceptor: HttpInterceptorFn = (req, next) =>
  next(req.clone({ setHeaders: { Authorization: inject(Auth).token() } }));

Two more conventions earn their keep everywhere: keep components thin (logic migrates to services and computed signals, which is also what makes TestBed tests small — the Testing in .NET instincts transfer directly), and let the CLI generate so names and structure stay uniform. An interceptor, you'll notice, is ASP.NET middleware wearing a TypeScript jacket — the pipeline idea, on the client.

A components/ folder and a services/ folder — organized!
Type-sorted folders scatter every feature across the tree. Feature folders keep a slice\u2019s route, components, and state together — lazily loaded as a unit, reviewable as a unit, and, best of all, deletable as a unit.
Guards, interceptors, resolvers — too much framework magic.
Each is a plain function registered with DI and called at a documented moment — the middleware pattern from ASP.NET Core, client-side. The magic is just a pipeline you can read, test, and order deliberately.
ExpertModule 11

The dashboard, assembled

The front-end course ended with this component in pseudocode. Here it is in Angular — every line an earlier module:

dashboard.component.ts 
@Component({                                        // m01 · a class + a decorator
  changeDetection: ChangeDetectionStrategy.OnPush,  // m09 · check the path
  template: `
    <stats-bar [links]="store.links()"/>          // m04 · input ↓
    @if (store.links().length) {                     // m02 · control flow
      @for (l of store.links(); track l.code) {
        <link-card [link]="l" (remove)="drop($event)"/>  // m04 · output ↑
      }
    } @else { <create-link-form/> }                // m08 · reactive form
    <router-outlet/>                                 // m07 · /links/:code
  ` })
export class DashboardComponent {
  store = inject(SnipStore);                        // m05 · ask the tree
  constructor() {
    inject(SnipApi).stats$()                       // m06 · HttpClient stream
      .pipe(takeUntilDestroyed())                   // m09 · lifecycle-safe
      .subscribe(ls => this.store.links.set(ls));   // m03 · signals
  }
}
One framework, one loop

Underneath the platform sits the same loop the parent course taught: signals hold state, templates render it, events update it. Angular's contribution is everything around the loop — a DI tree so parts swap, a router so the URL is state, typed forms so validity is a model, RxJS so time is composable, and a CLI so none of it needs assembling by hand. Learn the loop once; Angular is the loop with batteries and opinions included — and after fourteen courses, you know exactly why each opinion exists.
