BeginnerModule 00

What we're building

Snip is a URL shortener. Its whole job is two endpoints: POST /shorten takes a long URL and returns a short one like snip.ly/4C92, and GET /4C92 looks that code up and redirects to the original, counting the click on the way. Tiny surface, but building it well pulls in every course in this series.

The shape is a clean layered app: an ASP.NET Core minimal API at the edge, a ShortenerService holding the logic, a repository over EF Core for persistence to Postgres, and an in-memory cache in front of reads. A short code is just the row's Id rendered in base 62. Everything else — the patterns, the complexity choices, the tests — hangs off that skeleton.

OOPBeginnerModule 01

The domain model

Start where the design should start: the domain type. A ShortLink is the row we store and the object the app reasons about. The instinct to make it a bag of public setters is the one to resist — instead we encapsulate, exposing only the operations that keep the object valid. A click is registered through a method, not by letting anyone assign the counter.

Domain/ShortLink.cs 
public class ShortLink
{
    public long Id { get; private set; }          // surrogate key → the short code
    public string Code { get; private set; } = "";
    public string LongUrl { get; private set; } = "";
    public DateTime CreatedAt { get; private set; }
    public long Clicks { get; private set; }

    private ShortLink() { }                          // for EF

    public ShortLink(Uri longUrl, DateTime createdAt)
    {
        LongUrl = longUrl.ToString();            // a valid Uri can't be malformed
        CreatedAt = createdAt;
    }

    public void RegisterClick() => Clicks++;   // the only way Clicks changes
    public void AssignCode(string code) => Code = code;
}

Two small choices carry a lot of the design. Taking a Uri in the constructor makes an invalid URL unrepresentable — the type system enforces the invariant instead of a runtime check scattered everywhere. And a private setter on Clicks means the count can only move through RegisterClick(), so there's exactly one place that behavior lives.

↳ OOP + C#
This is encapsulation and "make illegal states unrepresentable" from OOP & Coding Principles, expressed with C# properties and a private constructor for EF. A rich domain object like this — behavior with its data — is the antidote to the anemic model the OOP course warns about.
Entities are just data — public getters and setters on everything.
That's the anemic model: data with no guardrails, so invariants leak into every caller. Encapsulating mutation behind methods keeps the rules in one place and the object always valid — the whole point of OOP.
Store the URL as a plain string and validate it later.
Accepting a Uri pushes validation to the boundary — by the time you hold a ShortLink, the URL is known-good. Types that can't hold bad data beat checks you have to remember to run.
BitsBeginnerModule 02

The short code

Where does 4C92 come from? It's the row's numeric Id written in base 62 — the digits 0-9, then A-Z, then a-z. Base 62 packs a big number into few characters: id 1,000,000 becomes just four symbols. It's the same positional-notation idea as binary or hex from the bits course, only with 62 symbols instead of 2 or 16.

Encoding/Base62.cs 
public static class Base62
{
    const string Alphabet = "0123456789ABCDEFGHIJKLMNOPQRSTUVWXYZabcdefghijklmnopqrstuvwxyz";

    public static string Encode(long n)
    {
        if (n == 0) return "0";
        var sb = new StringBuilder();
        while (n > 0)
        {
            sb.Insert(0, Alphabet[(int)(n % 62)]);   // least-significant digit first
            n /= 62;
        }
        return sb.ToString();
    }

    public static long Decode(string code)
    {
        long n = 0;
        foreach (var c in code)
            n = checked(n * 62 + Alphabet.IndexOf(c));  // checked → throws on overflow
        return n;
    }
}

Type the id below and watch it convert digit by digit — then decode straight back, because the two must be exact inverses:

Base-62 encode & decode
a number becomes a short code — and back
Repeated division by 62 peels off one digit at a time; decoding folds them back. Round-trip must be lossless.
↳ Bits + Big O
Base 62 is positional notation — the Bits, Pointers & Overflow idea behind binary and hex. Using long (not int) matters: int overflows near 2.1 billion rows, and checked turns a silent wrap into an exception. Encoding is O(log₆₂ n) — the code grows one character per 62× more links.
Just use a random GUID as the short code.
A GUID is 32+ characters — the opposite of short — and needs collision handling. Encoding a unique Id is compact and collision-free by construction: distinct ids always map to distinct codes.
int is plenty for the id.
An int caps at ~2.1 billion; a popular shortener blows past that. long reaches ~9.2 quintillion, and checked arithmetic makes any overflow loud instead of silently corrupting a code — exactly the overflow lesson from the bits course.
SQLBeginnerModule 03

The database

One table carries Snip. Each row is a link: a bigserial primary key (the number we base-62 encode), the generated code, the original URL, a timestamp, and a click counter. The key decision is the unique index on code — every redirect is a lookup by code, so it's the hottest query in the system and must not be a table scan.

db/schema.sql 
CREATE TABLE links (
    id          bigserial     PRIMARY KEY,
    code        text          NOT NULL,
    long_url    text          NOT NULL,
    created_at  timestamptz   NOT NULL DEFAULT now(),
    clicks      bigint        NOT NULL DEFAULT 0
);

-- the redirect path: WHERE code = $1, on every single visit
CREATE UNIQUE INDEX ix_links_code ON links (code);

With that index, Postgres finds a code by walking a B-tree — a handful of steps even across millions of rows — instead of scanning the table. The uniqueness constraint does double duty: it enforces that no two links share a code and provides the fast path. The redirect query itself is boring, which is exactly what you want on the hot path:

db/redirect.sql 
SELECT long_url FROM links WHERE code = '4C92';   -- index seek, not a scan
↳ SQL & PostgreSQL
This is the indexing lesson from SQL & PostgreSQL applied to the one query that matters most. The unique B-tree index turns an O(n) scan into an O(log n) seek — the difference between a shortener that scales and one that grinds to a halt as links pile up.
Add an index on every column to be safe.
Indexes cost write time and storage. Index the columns you actually filter or join on — here, code. Snip almost never queries by long_url, so indexing it would be pure overhead, as the SQL course explains.
Count clicks by inserting a row per visit and running COUNT(*).
That turns every redirect into a growing aggregate query. A single clicks counter updated in place is O(1) to read. (If you later need per-visit analytics, that's a separate append-only table — a deliberate trade, not the default.)
EF CoreIntermediateModule 04

EF Core mapping

EF Core connects the ShortLink object to the links table so we write LINQ, not SQL strings. The DbContext declares the set; the Fluent configuration mirrors the schema — the unique index especially, so EF's migrations create exactly the index we designed by hand.

Data/SnipContext.cs 
public class SnipContext(DbContextOptions<SnipContext> options) : DbContext(options)
{
    public DbSet<ShortLink> Links => Set<ShortLink>();

    protected override void OnModelCreating(ModelBuilder b)
    {
        var link = b.Entity<ShortLink>();
        link.ToTable("links");
        link.Property(x => x.Code).HasMaxLength(16);
        link.HasIndex(x => x.Code).IsUnique();      // → CREATE UNIQUE INDEX
    }
}

The repository's read is a single LINQ line that EF translates to the indexed WHERE code = … seek from the last module. Because Code is indexed and unique, FirstOrDefaultAsync stops at the first match:

Data/LinkRepository.cs 
public Task<ShortLink?> FindByCodeAsync(string code) =>
    _db.Links.FirstOrDefaultAsync(x => x.Code == code);   // → SELECT ... WHERE code = @code
↳ EF Core
The DbSet, Fluent HasIndex, and LINQ-to-SQL translation are straight from EF Core. EF writes the SQL, but the index you configured is why that SQL is fast — the ORM doesn't excuse you from knowing what query runs underneath.
EF Core means you never think about SQL again.
EF writes the query, but you still choose the indexes and shape that make it fast. Configure the wrong index and FirstOrDefaultAsync quietly becomes a full scan — the ORM hides the SQL, not the cost.
Load the links and filter with LINQ in memory.
Links.ToList().First(x => x.Code == code) drags the whole table into the app. Keeping the predicate in the IQueryable lets EF push WHERE to Postgres and use the index — the translation lesson from the EF course.
Data StructuresIntermediateModule 05

Making lookups fast

A popular short link gets hit constantly, and going to Postgres every time — even for an indexed seek — adds up. So Snip puts an in-memory cache in front of reads: a Dictionary keyed by code, which is an O(1) hash lookup. This is the classic three-tier speed story: a full scan is O(n), the B-tree index is O(log n), and a cache hit is O(1).

Slide the number of links up and watch how many steps each strategy needs to find one code:

links stored:
As the table grows, the scan explodes, the index barely moves, and the cache stays flat at one step.

In code it's the cache-aside pattern: check the dictionary, and only on a miss go to the repository and remember the answer. An LRU bound keeps memory finite by evicting the least-recently-used entries.

Caching/LinkCache.cs 
public async Task<string?> ResolveAsync(string code)
{
    if (_map.TryGetValue(code, out var url)) return url;  // O(1) hit
    var link = await _repo.FindByCodeAsync(code);           // miss → O(log n) index seek
    if (link is null) return null;
    _map.Set(code, link.LongUrl);                          // LRU-bounded
    return link.LongUrl;
}
↳ Data Structures + Big O
The dictionary is the hash map from Data Structures — average O(1) — and the index is a B-tree, O(log n); both beat the O(n) scan from Big O. Snip layers all three so the common case is constant time and the rare miss is still logarithmic.
Postgres is fast — a cache is premature optimization.
For a hot link served thousands of times a second, even a 1 ms indexed query dominates the response. An O(1) in-memory hit is orders of magnitude cheaper, and the data structures course shows why the hash map wins here.
Just cache everything and never evict.
Unbounded caches exhaust memory. An LRU bound keeps the hot set and drops the cold — trading a little hit rate for a hard memory ceiling, the eviction trade-off from the data structures course.
PatternsIntermediateModule 06

Patterns in play

Three patterns do real work in Snip, and no more. The Repository hides EF behind an interface so the service depends on "somewhere links live," not on DbContext. The Strategy pattern makes code generation swappable — sequential today, random tomorrow — without touching the service. And Dependency Injection wires it together: the service asks for interfaces and receives whatever's registered.

Services/ICodeStrategy.cs 
public interface ICodeStrategy
{
    string CodeFor(ShortLink link);
}

// default: the id in base 62 — compact and collision-free
public class SequentialCodeStrategy : ICodeStrategy
{
    public string CodeFor(ShortLink link) => Base62.Encode(link.Id);
}

The service is the composition point. It takes its collaborators as interfaces — the repository, the strategy, a clock — so every dependency can be swapped, and (crucially for Module 09) replaced with a test double:

Services/ShortenerService.cs 
public class ShortenerService(
    ILinkRepository repo, ICodeStrategy strategy, IClock clock)
{
    public async Task<ShortLink> ShortenAsync(Uri url)
    {
        var link = new ShortLink(url, clock.UtcNow);
        await repo.AddAsync(link);          // assigns Id
        link.AssignCode(strategy.CodeFor(link));
        await repo.SaveAsync();
        return link;
    }
}
↳ Design Patterns + OOP
Repository, Strategy, and DI all come from Design Patterns, and they rest on the Dependency Inversion principle from OOP: depend on abstractions. Taking an IClock instead of calling DateTime.UtcNow is what makes time controllable in tests — the payoff lands in Module 09.
A well-designed app uses lots of patterns.
Patterns are tools for specific problems, not a quality score. Snip uses three because each removes a real coupling; adding a factory or observer here would be ceremony. The design patterns course is about knowing when, not applying all of them.
Strategy is overkill when there's only one algorithm.
It isolates the one decision most likely to change and makes it testable in isolation. Even with a single strategy today, the seam costs almost nothing and pays off the moment you need random codes or a custom-alias scheme.
ASP.NETIntermediateModule 07

The Web API

Now the edge. Two minimal-API endpoints expose Snip over HTTP: one to create a link, one to follow it. Dependency injection hands each endpoint the services it needs, model binding turns the request body into a typed value, and the right status codes carry the meaning — 201 Created for a new link, a 301 redirect to follow one, 404 when the code is unknown.

Program.cs 
var builder = WebApplication.CreateBuilder(args);
builder.Services.AddDbContext<SnipContext>(o => o.UseNpgsql(cs));
builder.Services.AddScoped<ILinkRepository, LinkRepository>();
builder.Services.AddScoped<ShortenerService>();
builder.Services.AddSingleton<ICodeStrategy, SequentialCodeStrategy>();
builder.Services.AddSingleton<ILinkCache, LinkCache>();     // one shared cache
var app = builder.Build();

app.MapPost("/shorten", async (ShortenRequest req, ShortenerService svc) =>
{
    var link = await svc.ShortenAsync(new Uri(req.Url));
    return Results.Created($"/{link.Code}", new { shortUrl = $"snip.ly/{link.Code}" });
});

app.MapGet("/{code}", async (string code, ILinkCache cache) =>
{
    var url = await cache.ResolveAsync(code);
    return url is null ? Results.NotFound() : Results.Redirect(url, permanent: true);
});

↳ ASP.NET Core
Minimal APIs, the DI container with its AddScoped/AddSingleton lifetimes, model binding, and correct status codes are all from ASP.NET Core. The cache is a singleton (shared across requests) while the DbContext is scoped (one per request) — exactly the lifetime lesson from that course.
Return 200 with the URL in the body for a redirect.
A redirect is a 301/302 with a Location header — that's what browsers follow automatically. Returning 200 makes the client do extra work and breaks the one thing a shortener must do. Status codes are part of the contract.
Put the shortening logic straight in the endpoint.
The endpoint should bind input, call a service, and shape the response — nothing more. Keeping logic in ShortenerService makes it reusable and testable without HTTP, the thin-controller idea from the ASP.NET course.
BitsIntermediateModule 08

Scale & edge cases

A real service needs per-link options — disabled, one-time-use, private — and storing five booleans as five columns is wasteful. Instead we pack them into a single integer with a [Flags] enum, the bitmask from the bits course. Each flag is one bit; the set of them is one int, combined and tested with bitwise | and &.

Domain/LinkFlags.cs 
[Flags]
public enum LinkFlags
{
    None    = 0,
    Disabled = 1 << 0,   // 0001
    OneTime  = 1 << 1,   // 0010
    Private  = 1 << 2    // 0100
}

// combine with OR, test with AND — one int holds them all
var flags = LinkFlags.OneTime | LinkFlags.Private;   // 0110
bool isPrivate = (flags & LinkFlags.Private) != 0;   // true

The other edge cases are about not corrupting data at scale. The click counter uses checked arithmetic so a wildly popular link can never silently wrap a long; and if you ever switch to the random code strategy, codes can collide, so a write must detect the unique-index violation and retry:

Services/ShortenerService.cs (random-strategy path) 
for (int attempt = 0; attempt < 5; attempt++)
{
    link.AssignCode(strategy.CodeFor(link));
    try { await repo.SaveAsync(); return link; }
    catch (UniqueViolationException) { /* collision → try a new code */ }
}
↳ Bits, Pointers & Overflow
The [Flags] bitmask, the shift operators, and checked arithmetic are all from Bits, Pointers & Overflow. Packing booleans into bits and guarding against silent overflow are exactly the low-level concerns that course drilled — here keeping Snip compact and correct under load.
Store each on/off option as its own boolean column.
A [Flags] enum packs many booleans into one integer — less storage, one field to read, and set operations for free via | and &. That's the bitmask technique from the bits course, and it scales to dozens of flags in a single int.
Random codes are unique — no need to check.
Randomness makes collisions unlikely, not impossible (the birthday problem bites as the table fills). The unique index is the source of truth: catch the violation and retry. Sequential codes sidestep this entirely — a design trade worth making on purpose.
TestingAdvancedModule 09

Testing the app

Snip earns confidence at three levels. A property test pins the encoder: for any id, decoding the code must return the id — a [Theory] over many values. A unit test with a mocked repository and clock checks the service's logic in isolation. And an integration test drives real HTTP through WebApplicationFactory, exercising routing, DI, and EF together.

Tests/Base62Tests.cs 
[Theory]
[InlineData(1)] [InlineData(125)] [InlineData(1_000_000)] [InlineData(long.MaxValue)]
public void Decode_is_the_inverse_of_Encode(long id)
    => Assert.Equal(id, Base62.Decode(Base62.Encode(id)));   // round-trip must be lossless

↳ Testing + Bits
Property tests, mocking, and WebApplicationFactory are from Testing in .NET; the bug they catch is the int overflow from Bits. The IClock and ILinkRepository seams built back in Module 06 are what make the service unit-testable without a database or a real clock.
Clicking through the app by hand is enough testing.
Manual checks don't run on every change. This suite re-verifies encoding, the service, and the endpoints in seconds, so the overflow bug can't sneak back — the change-safety argument from the testing course.
Write tests for EF and ASP.NET themselves.
Trust the frameworks; test your logic — the encoder, the service's decisions, your endpoints' contracts. One focused integration test confirms the wiring; you don't re-test what Microsoft already tests.
Big OExpertModule 10

Complexity review

Step back and cost the whole system. Snip is designed so the operation that runs most — the redirect — is O(1) on a cache hit, and nothing on any path is worse than logarithmic. Encoding grows with the number of digits (O(log₆₂ n)); the indexed lookup is O(log n); a cache hit is flat. Slide N and watch the step counts diverge:

The redirect — Snip's busiest operation — stays constant. The rest are logarithmic. Nothing on the hot path is linear.
↳ Big O & Complexity
This is Big O applied to a real system: identify the hot path, make it the cheapest, and keep everything else sub-linear. The whole architecture — index plus cache — exists to keep the redirect O(1), because that's the operation multiplied by every visitor.
Optimize the encoder — it runs on every shorten.
Encoding is O(log n) over a handful of digits — already negligible. The operation that scales with traffic is the redirect lookup, so that's where the cache and index go. Optimize the hot path, not the cheap one — the Big O course's core discipline.
Big O is academic; real perf is just constant factors.
Constants matter at the margins, but growth decides whether Snip survives a million links. An O(n) redirect would be fine in testing and fatal in production. Getting the complexity class right first is what makes the constants worth tuning later.
