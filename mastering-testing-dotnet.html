BeginnerModule 00

Why we test

It's tempting to think tests exist to catch bugs. They do — but that undersells them badly. The real payoff of a test suite is confidence to change code. A good suite is an executable specification: it states what your code is supposed to do, and the instant a change breaks that, it tells you — automatically, in seconds, before the bug reaches anyone. Without it, every edit is a gamble and every refactor is scary.

Think about the alternative. With no tests, the only way to know you didn't break something is to re-check it by hand, every time, forever. That cost makes people afraid to touch working code, so it rots. Tests flip that: they make change cheap and safe, which is exactly why teams that test well move faster over the life of a project, not slower.

A test is just ordinary code that runs your code and checks the result. You'll write them in the same language, run them with one command, and get back a wall of green when all is well or a precise red failure when it isn't. This course builds from that first [Fact] up to test-driven development, using the very C#, ASP.NET, and EF code the rest of the track produced.

Testing is QA's job — developers write code, testers test it.
The tests this course covers are a developer tool, written alongside the code. They're about design and change-safety, not a separate QA phase. Manual QA still has a place, but it can't give you the fast, automatic regression net that unit tests do.
Writing tests slows me down.
They cost time up front and repay it many times over — every future change is verified in seconds instead of re-checked by hand. The slow projects are usually the untested ones, where nobody dares refactor and bugs reappear endlessly.
AnatomyBeginnerModule 01

Your first test

In .NET you write tests in a separate test project using a framework like xUnit. A test is a method marked [Fact], and the convention that keeps tests readable is Arrange-Act-Assert: set up what you need, perform the one action you're testing, then check the outcome. Three phases, in order, every time.

c#[Fact]
public void Discount_reduces_price_by_the_given_percent()
{
    // Arrange
    var cart = new Cart(price: 100m);
    // Act
    var total = cart.ApplyDiscount(percent: 20);
    // Assert
    Assert.Equal(80m, total);
}

Step through the three phases below — each does one job, and keeping them separate is what makes a test easy to read and easy to trust:

Arrange · Act · Assert
the shape of nearly every test
Most tests follow this rhythm: arrange the inputs, act once, assert the outcome.
A real test needs a lot of setup and ceremony.
Start tiny. A [Fact], three lines, one assertion is a complete, valuable test. If a test needs heavy setup, that's often a hint the code under test has too many dependencies — a design smell the test just surfaced.
One test method can verify a whole feature at once.
Aim for one behavior per test. When a focused test fails, its name tells you exactly what broke. A sprawling test that checks ten things gives you a vague failure and a puzzle instead of an answer.
AssertionsBeginnerModule 02

Assertions

The Assert phase is where a test actually decides pass or fail. xUnit gives you a focused vocabulary — Assert.Equal, Assert.True, Assert.Throws, Assert.Contains, Assert.Null, and more. The art is asserting the outcome you actually care about, and picking the assertion that produces the clearest failure message when it's wrong.

Try different assertions below — change the expected or actual value and watch it pass or fail, with the message xUnit would print:

What an assertion checks
pick one, then make it pass or fail
A passing assertion is silent; a failing one prints expected-vs-actual so you can see what went wrong.
More assertions per test is more thorough.
A test should make one logical assertion. Cramming many unrelated checks in means the first failure hides the rest, and the test name can't describe what it verifies. Split distinct expectations into distinct, well-named tests.
Assert.True(a == b) is just as good as Assert.Equal(a, b).
They behave the same until they fail. Assert.True can only say "expected true, got false"; Assert.Equal prints both values, so you see Expected: 80, Actual: 75 at a glance. Pick the assertion that fails most informatively.
QualityBeginnerModule 03

What makes a good test

A useful shorthand for a good unit test is FIRST: Fast (milliseconds, so you run thousands without thinking), Isolated (independent of other tests and of order), Repeatable (same result every run, on any machine), Self-validating (it asserts pass/fail itself — no reading output by eye), and Timely (written close to the code, ideally first). Tests that violate these turn into a liability you stop trusting.

The deepest principle, though, is to test behavior, not implementation. Assert on what the code does from the outside — its return values, its observable effects — not on how it does it internally. Behavior-focused tests keep passing when you refactor the internals; implementation-coupled tests break every time you tidy the code, which trains people to delete them.

c#// ✗ couples to implementation — breaks on any refactor
Assert.Equal(1, cart._internalDiscountCallCount);   // who cares?

// ✓ asserts behavior — survives refactors
Assert.Equal(80m, cart.ApplyDiscount(20));    // the outcome that matters
↳ OOP course
This is encapsulation from OOP & Coding Principles applied to tests: assert against the public contract, not the private internals. A test that reaches into private state is as coupled as code that does — and just as painful to change.
Tests can share state and run in a set order.
Each test must be independent — runnable alone, in any order, in parallel. Shared mutable state between tests creates failures that appear and vanish depending on run order, the most maddening kind to debug (Module 10).
100% code coverage means the code is well tested.
Coverage measures which lines ran, not whether you asserted anything meaningful about them. You can execute every line and verify nothing. Coverage is a useful gap-finder, a useless quality target (Module 10).
CasesIntermediateModule 04

Parameterized tests

When the same logic should hold across many inputs, you don't copy-paste the test. xUnit's [Theory] runs one test method once per data row, supplied by [InlineData] (or [MemberData] for richer sets). This is your best tool for hammering boundaries and edge cases — and boundaries are exactly where bugs hide.

Below, a [Theory] checks an IsAdult rule across several ages. The implementation has a subtle off-by-one — run the cases and see which one catches it:

One test, many cases
boundaries are where bugs live
Each row is a separate run of the same test. The boundary case is the one that earns its keep.
↳ Big O & Data Structures
The off-by-one at the boundary is the same hazard from Data Structures and Big O — the bug that lives at ==, the first index, the empty case. Parameterized tests let you pin every boundary cheaply, so an off-by-one can't slip through unnoticed.
Just write one test for a typical input.
The typical input is the one least likely to break. The value comes from the edges: zero, empty, the boundary, the max, the negative. A [Theory] lets you cover them all without ten near-identical test methods.
If the happy path passes, the feature works.
Most real bugs live off the happy path — at limits, on bad input, in the empty case. Testing only the obvious scenario gives a green suite and a fragile feature. Make the edges first-class test cases.
DoublesIntermediateModule 05

Test doubles

To test a unit in isolation, you replace its real collaborators with stand-ins called test doubles. This keeps a test fast and deterministic — no real database, no real email, no clock you can't control — and lets you set up exactly the situation you want to verify. The doubles have names worth knowing because they do different jobs.

A stub returns canned answers to feed the unit input (you check the unit's resulting state). A mock carries expectations about interactions — you assert it was called, how, and how often. A fake is a real but lightweight working implementation (an in-memory list standing in for a database). Dummies and spies round out the family, but stub-vs-mock is the distinction that matters most: state verification versus interaction verification.

c#// the unit depends on an abstraction, not a concrete service
public class Checkout(IInventory inventory, IEmailSender email)
{
    public Result Place(Order o)
    {
        if (!inventory.InStock(o.Sku)) return Result.OutOfStock;
        email.SendReceipt(o);            // an interaction we may want to verify
        return Result.Placed;
    }
}
↳ Design Patterns + OOP
Doubles only work because Checkout depends on interfaces, not concrete classes — the Dependency Inversion principle from OOP, enabling the injection you saw in Design Patterns and ASP.NET. Testable code and well-decoupled code turn out to be the same thing.
Stubs and mocks are two words for the same thing.
A stub feeds input and you assert on the unit's state; a mock records calls and you assert on the interaction. "Did the total come out right?" wants a stub; "was the receipt actually sent?" wants a mock. Mixing them up leads to muddled tests.
Replace every dependency with a double.
Over-mocking produces tests that mirror the implementation and break on every change. Use a real collaborator when it's fast and deterministic; reach for a double when the real thing is slow, external, or non-deterministic. Fewer doubles, sturdier tests.
MockingIntermediateModule 06

Mocking & dependencies

Hand-writing doubles gets tedious, so .NET has mocking libraries — Moq, NSubstitute, FakeItEasy — that build a stub or mock of an interface in a line or two. You configure what a method returns, inject the double where the real collaborator would go, run the unit, and then verify the interactions you care about. Because the code already takes its dependencies through constructor injection, swapping in a double is trivial.

The Checkout from the last module takes an inventory and an email sender. Stub the inventory to set up the scenario, then verify whether the receipt email was sent — watch the state assertion and the interaction check:

↳ ASP.NET course
The constructor injection that makes this testable is the very same AddScoped/AddSingleton wiring from ASP.NET Core. Code built for dependency injection is, for free, code built for testing — the double slots into the exact seam the DI container uses.
You need a mocking framework to write tests.
A framework just removes boilerplate. A hand-written class implementing the interface is a perfectly good double, and is sometimes clearer than a pile of mock-setup calls. Reach for Moq when it saves real effort, not by reflex.
Verify every call the unit makes.
Verifying incidental calls couples the test to implementation detail and makes it brittle. Verify the interactions that are part of the contract — "a receipt is sent on success" — and ignore the rest. Over-verification is over-mocking by another name.
DataIntermediateModule 07

Testing data access

Testing EF code forces a choice about fidelity versus speed. You have three broad options. The in-memory provider is fastest but isn't a relational database — it skips real SQL, constraints, and transactions, so it can pass on code a real database would reject. SQLite in-memory runs genuine SQL in-process and catches far more, a strong default for repository tests. A real test database is the most faithful and the slowest, best reserved for true integration coverage.

c#// SQLite in-memory: real SQL, real constraints, still fast
var conn = new SqliteConnection("DataSource=:memory:");
conn.Open();
var options = new DbContextOptionsBuilder<ShopContext>()
    .UseSqlite(conn).Options;

using var db = new ShopContext(options);
db.Database.EnsureCreated();
db.Products.Add(new Product { Name = "Lamp", Price = 35 });
db.SaveChanges();

Assert.Equal(1, db.Products.Count(p => p.Price < 50));   // real query
↳ EF Core course
This is the DbContextOptions and provider-swapping from EF Core put to work: the same context, pointed at SQLite for tests instead of your production database. And the repository seam EF debates is exactly what lets a test substitute one cleanly.
The in-memory provider tests my real database behavior.
It doesn't run SQL or enforce relational rules, so unique constraints, foreign keys, and transaction semantics all go untested. It's convenient for simple logic, but a query that works in-memory can still fail against a real database. Prefer SQLite when fidelity matters.
Every data test must hit a real database.
That's slow and brittle for the bulk of cases. Use the cheapest fidelity that catches the bugs you care about — SQLite for most repository tests, a real database for a focused set of integration tests. Match the tool to the risk.
IntegrationIntermediateModule 08

Integration testing

Unit tests check pieces in isolation; integration tests check that the pieces work together. For ASP.NET Core, WebApplicationFactory<T> boots your whole app in-memory and lets you send real HTTP requests through it — exercising routing, the middleware pipeline, model binding, dependency injection, and EF, all in one test. No network, no deployed server, but the real wiring end to end.

c#public class TodoApiTests(WebApplicationFactory<Program> factory)
    : IClassFixture<WebApplicationFactory<Program>>
{
    [Fact]
    public async Task Get_todos_returns_200_and_a_list()
    {
        var client = factory.CreateClient();          // the app, in-memory
        var res = await client.GetAsync("/todos");    // a real request
        res.EnsureSuccessStatusCode();                  // 200
        var todos = await res.Content.ReadFromJsonAsync<Todo[]>();
        Assert.NotEmpty(todos);
    }
}
↳ ASP.NET + EF
This one test drives everything the C# track built: the ASP.NET request pipeline and routing, the endpoint's C# logic, and the EF query behind it. Integration tests are where the whole stack is verified as one working system.
Integration tests can replace unit tests.
They answer a different question — "do the parts connect?" not "is this piece correct?". Integration tests are slower and pinpoint failures less precisely, so you want many fast unit tests and fewer integration tests: the testing pyramid (Module 11).
An integration test should use all the real external services.
Boot the real app, but swap the truly external edges — third-party APIs, email, payment gateways — for test doubles or in-memory equivalents. Otherwise tests become slow, flaky, and dependent on systems you don't control.
TDDAdvancedModule 09

Test-driven development

Test-driven development inverts the usual order: you write the test first, then the code to satisfy it. The loop is three beats — Red: write a small failing test for the next bit of behavior. Green: write the minimum code to make it pass. Refactor: clean up now that the tests have your back. Then repeat, in tiny increments.

TDD is as much a design technique as a testing one. Writing the test first means you're the first user of your own API, so awkward, hard-to-call designs get felt and fixed immediately — and the code is testable by construction, because it was literally built to a test. ↳ OOP & Design Patterns
Because you call your API before it exists, TDD pressures you toward the small, focused, dependency-injected designs that OOP and Design Patterns argue for. Hard-to-test usually means hard-to-use — TDD makes that pain immediate instead of letting it accumulate.
TDD means writing every test before any code.
It's the opposite of big up-front test-writing — it's tiny red-green cycles, one small behavior at a time, with code and tests growing together in minutes-long loops. You're never far from a green bar.
TDD is purely a testing practice.
Its biggest payoff is design feedback. Writing the test first forces good seams and small units; the comprehensive test suite is almost a side effect. Teams adopt TDD as much for the designs it produces as for the safety net.
PitfallsExpertModule 10

Coverage, flakiness & pitfalls

Two traps catch experienced teams. The first is treating code coverage as a goal. Coverage tells you which lines executed during the tests — useful for spotting code nothing touches — but it says nothing about whether you asserted anything. Chase a coverage number and you get tests that run everything and verify little: green theater.

The second, more corrosive trap is the flaky test — one that passes or fails non-deterministically. The usual culprits are reading the real clock (DateTime.Now), unseeded randomness, shared mutable state across tests, and timing in async code. A flaky test is worse than no test, because it trains the team to ignore red. Toggle the causes below and run the suite a few times:

Why a test goes flaky
non-determinism makes red meaningless
Remove every source of non-determinism — inject a clock, seed the RNG, isolate state — and the test passes every single run.
A test that passes most of the time is fine.
An intermittently failing test is broken. People learn to re-run until green and stop reading failures — so the one real regression hides among the noise. Fix flakiness immediately or quarantine the test; never normalize ignoring red.
Aim for 100% coverage.
The last stretch costs the most and buys the least — testing trivial getters and unreachable branches. Target meaningful tests on logic that matters; let coverage be a map of gaps, not a score to max out.
ExpertModule 11

Putting it together

A healthy strategy balances the kinds of tests by their cost and breadth — the testing pyramid. A wide base of unit tests (many, fast, precise) sits under a smaller band of integration tests (fewer, slower, broader), topped by a thin slice of end-to-end tests (few, slowest, most realistic). Invert that — mostly slow end-to-end tests, few units — and you get the "ice-cream cone": slow feedback, flaky runs, and failures that don't tell you where the bug is.

The testing pyramid
shape your suite for fast, precise feedback
Most tests should be fast unit tests; reserve slow, broad tests for the few things only they can check.
The whole track, made changeable

Step back and the four courses form a complete engineering loop. C# gave you the language; ASP.NET Core turned it into a web server; EF Core backed that server with a database; and testing is what lets you change all of it without fear. A unit test pins your C# logic, a mock isolates a service, a SQLite test exercises your EF queries, and a WebApplicationFactory drives a real request through the whole pipeline. Build it, then build the tests that let you keep building.

Change the code. Trust the tests.

From your first [Fact] through assertions, doubles, integration tests, and the red-green-refactor loop, one idea ran underneath: tests aren't a tax on shipping — they're what makes shipping safe. A good suite is a specification that runs, a safety net that catches regressions in seconds, and a design force that pushes your code toward small, decoupled, callable units.

That's the quiet superpower behind every codebase people enjoy working in. Not the absence of bugs, but the freedom to change things and find out instantly if you were wrong. Write the test, watch it fail, make it pass — and keep the code yours.
