It's all just
bits in a box.

Every value your program touches is a pattern of bits sitting in a box in memory. A type decides how to read those bits; a variable names the box; a pointer holds the address of another box; and overflow is what happens when a value won't fit. Strip away the syntax and it's all the same handful of ideas — and you can poke at every one of them right here on the page.

BeginnerModule 00

Bits, bytes & binary

Underneath every number, letter, image, and instruction is the same raw material: the bit. A bit is the smallest possible piece of information — a single thing that is either 0 or 1, off or on. A computer is, at bottom, an enormous field of switches, and a bit is one switch. Everything in this course is a story about how we arrange and interpret those switches.

One bit alone says very little, so we group them. Eight bits make a byte, and the byte is the fundamental unit memory is measured and addressed in. With eight switches you can make 2⁸ = 256 distinct patterns, from 00000000 to 11111111 — enough for a small number, one character, or one slice of a bigger value.

To read a binary number, you use place values, exactly like decimal — except each place is a power of two instead of ten. The rightmost bit is worth 1, the next 2, then 4, 8, 16, and so on, doubling each step. A bit that is on contributes its place value; a bit that is off contributes nothing. Add up the on-bits and you have the value.

binary  1   0   1   0   1   0   1   0     // the bits
128  64  32  16   8   4   2   1     // place values (powers of two)
128 +     32 +     8 +     2        = 170

Binary is correct but hard for humans to read, so we lean on hexadecimal (base 16) as shorthand. The trick: one hex digit (0–9, then A–F for 10–15) encodes exactly four bits. So a byte is always two hex digits, and a messy 10101010 becomes a tidy 0xAA. Hex isn't a different kind of value — it's just a more compact way of writing the same bits, which is why addresses and byte dumps are almost always shown in hex.

Binary is a special weird thing computers do to numbers.

BeginnerModule 01

Primitive types — the atoms of data

Primitive types are the handful of built-in types a language and its hardware understand directly, with no assembly required. They're the atoms every other data structure is built from. A primitive has two defining properties: it occupies a fixed number of bits, and it's typically stored by value — the bits are the data, sitting right where the variable lives, not off somewhere behind a reference.

Here's the idea that unlocks everything else: a type is an interpretation. The bits in memory have no inherent meaning. The pattern 01000001 is the integer 65, the character 'A', or part of a larger number — depending entirely on what type you've told the machine to read it as. The type is the lens; the bits are just the bits.

The two questions a type answers

How many bits is this value? — and — how should those bits be interpreted? Fix those two answers and you have a primitive type. Everything from int to double to char is just a different pair of answers.

Most languages offer a small, familiar menu of primitives. The exact names vary, but the categories are universal: whole numbers (integers) in several widths and in signed or unsigned flavors; floating-point numbers for fractions and huge ranges; a one-bit-of-meaning boolean; and a character or raw byte. Width matters because it sets both the range of values and the memory cost.

Fixed width is the whole story

Because a primitive is a fixed number of bits, it can hold only a fixed number of distinct values — and that single fact is the seed of nearly everything that follows. It's why integers overflow, why floats can't represent every fraction, and why a value fits neatly into a known-size box in memory that a pointer can later refer to. Fixed width is a feature and a trap at once.

A bigger type is always safer, so just use 64-bit everything.
Wider types cost more memory and bandwidth, which matters in arrays, caches, and hot loops. The skill is choosing a width that comfortably fits your real range — not reflexively maxing it out, and not cutting it so fine that you overflow.
A character and a small integer are fundamentally different things.
At the bit level they're the same — a char is just a small integer that some code chooses to interpret as a letter via an encoding like ASCII or UTF-8. 'A' simply is 65.
IntegersBeginnerModule 02

Signed integers & two's complement

An unsigned integer is the easy case: read the bits straight as a binary number. Eight bits run from 0 to 255, sixteen from 0 to 65,535, and in general n bits cover 0 to 2ⁿ−1. No surprises. But how do you store a negative number when all you have is on/off switches and no minus sign?

The answer almost every machine uses is two's complement, and it's one of the most elegant tricks in computing. The idea: give the top bit (the most significant) a negative place value. In an 8-bit signed integer, the leftmost bit is worth −128 instead of +128; the rest stay positive. So 10000000 is exactly −128, and 11111111 is −128 + 127 = −1.

two's complement (8-bit) -128  64   32   16    8    4    2    1    // note the negative top bit
   1    1    1    1    1    1    1    1
-128 + 64 + 32 + 16 + 8 + 4 + 2 + 1            = -1

Two beautiful things fall out of this. First, there's exactly one zero (00000000), unlike schemes that waste a bit pattern on "negative zero." Second, and more importantly, ordinary binary addition just works across the sign boundary — the same adder circuit that computes 3 + 5 computes (−3) + 5 correctly, with no special handling. That's why two's complement won: the hardware gets simpler, not more complex.

To negate a number in two's complement, you flip every bit and add one (−x = ~x + 1). The same top bit that represents the largest negative also explains the slightly lopsided range: an 8-bit signed integer runs from −128 to +127, because zero eats one of the "positive" slots.

That scheme (called sign-magnitude) exists but is rarely used, because it has two zeros and needs special-case arithmetic. Two's complement bakes the sign into the place values so plain addition handles negatives for free.
An unsigned and a signed integer with the same bits are the same number.
Same bits, different value. 11111111 is 255 read as unsigned but −1 read as signed. The bits don't change — only the interpretation does, which is exactly why type matters.
OverflowIntermediateModule 03

Integer overflow & wraparound

A fixed number of bits can hold only a fixed range of values. So what happens when a calculation produces a result too big to fit? The value doesn't grow a ninth bit or politely raise an error — it wraps around. Think of a car's odometer rolling from 999999 back to 000000, or a clock passing 12 back to 1. The extra carry simply falls off the top and is lost.

For an 8-bit unsigned integer, the maximum is 255 (11111111). Add one and the carry ripples off the end, leaving 00000000 — you're back at 0. This is exact modular arithmetic: results are taken mod 2ⁿ. For a signed integer it's stranger: 127 (01111111) plus one becomes 10000000, which in two's complement is −128. The largest positive number suddenly becomes the most negative.

8-bit wraparound  unsigned:   255 + 1  =  0      // 11111111 + 1 → 00000000
  signed:     127 + 1  =  -128   // 01111111 + 1 → 10000000
Signed overflow is undefined behavior in C/C++

Unsigned overflow is defined to wrap. But signed overflow is undefined behavior — the compiler is allowed to assume it never happens and optimize accordingly. A loop like for (int i = 0; i >= 0; i++) may be "proven" infinite and have its exit deleted. This is why overflow bugs are so dangerous: the result isn't just a wrong number, it can be code that does something the source never said.

This isn't academic. A 32-bit signed counter caps at about 2.1 billion — which is why YouTube had to rebuild its view counter when Gangnam Style blew past it, and why the Year 2038 problem looms for systems counting seconds since 1970 in a 32-bit int. Ariane 5 flight 501 was destroyed by a 64-bit float crammed into a 16-bit integer; Pac-Man's level 256 "kill screen" is an 8-bit counter overflowing. Fixed width, finite range, real consequences.

By default it silently wraps — 255 becomes 0, 127 becomes −128 — and execution continues with the wrong value. No crash, no cap, no warning. The damage happens downstream, far from the overflow.
Overflow only matters for huge numbers.
It strikes at each type's edge, however small. An 8-bit counter overflows at 256; a 16-bit one at 65,536. Multiplying two modest ints can overflow even when each fits comfortably — the product is what blows the range.
Floating pointIntermediateModule 04

Floating point — fractions in bits

Integers can't hold ½ or π, and they can't reach the very large or very small. Floating point solves both by storing numbers in binary scientific notation: a sign, a fraction (the mantissa), and an exponent that floats the binary point left or right. Just as 6.022×10²³ packs a huge number into few digits, a float packs an enormous range into 32 or 64 bits.

The near-universal standard is IEEE 754. A 32-bit float divides its bits into three fields: 1 sign bit, 8 exponent bits, and 23 mantissa bits. A 64-bit double uses 1 + 11 + 52, buying far more precision and range. The value is roughly (−1)sign × 1.mantissa × 2exponent.

The catch is precision. With a fixed number of mantissa bits, only a fixed set of values can be represented exactly; everything else is rounded to the nearest one. And just as 1/3 has no exact decimal, 0.1 has no exact binary — it's a repeating fraction in base two. So 0.1 is stored as something microscopically off, and the famous result follows:

js> 0.1 + 0.2
0.30000000000000004   // not a bug — 0.1 and 0.2 aren't exactly representable

Never store money in a float

Because most decimal fractions can't be represented exactly, floats accumulate tiny rounding errors that compound — catastrophic for currency. Use integer cents, or a decimal type built for exact base-ten arithmetic. Floats are for measurement and science, where a relative error of 10⁻⁷ is invisible; they are wrong for anything that must balance to the penny.

0.1 + 0.2 == 0.3 failing is a bug in the language.
It's correct behavior of the IEEE 754 standard, identical across nearly every language. None of 0.1, 0.2, or 0.3 is exactly representable in binary, so the rounded sum misses the rounded target by one tiny step. Compare floats with a tolerance, never with ==.
Floats have a fixed number of decimal places.
They have a fixed number of significant bits, so precision is relative. Near zero, floats are densely packed; out at billions, the gap between consecutive floats can exceed 1, so adding a small number to a huge one can change nothing at all.
MemoryIntermediateModule 05

Variables & memory

Now we can say precisely what a variable is. Memory is one gigantic array of bytes, and every byte has a numeric address — its index in that array. A variable is three things bound together: a name you type, a box of memory of the type's size, and the address where that box lives. The name is a convenience for you; the machine only ever deals in addresses.

cint x = 42;        // reserve a 4-byte box, label it x, store 42's bits
char c = 'A';      // reserve a 1-byte box, label it c, store 65
double d = 3.14;   // reserve an 8-byte box, label it d, store the IEEE-754 bits

Declaring a variable allocates a box big enough for its type and binds the name to that location. Reading the variable fetches the bits from the box; assigning to it overwrites them. The name x doesn't appear anywhere in the running program — the compiler has already translated it into "the four bytes at address 0x7ffe…".

Value semantics

For a primitive, assignment copies the bits. b = a reads a's box and writes a duplicate into b's box. They are now two independent boxes that merely happen to hold equal values — change one and the other is untouched. This "copy by value" behavior is the default for primitives in essentially every language, and it's the thing that makes pointers and references (next) feel so different.

Where do these boxes live? Local variables typically sit on the stack — a fast region that grows and shrinks automatically as functions are called and return. A local's box exists only while its function is running; when the function returns, the box is reclaimed and its address may be reused. That lifetime — when a box is valid — becomes critical the moment you start keeping addresses of boxes, which is exactly what pointers do. We'll return to the stack, and its longer-lived sibling the heap, in Module 08.

A variable is its value.
A variable is a named box at an address; the value is the bits currently inside it. The distinction seems pedantic until you take the address of the box (a pointer) or give the box a second name (a reference) — then the box and its contents are clearly different things.
Assigning b = a links the two variables.
For primitives it copies — you get two independent boxes. Later changes to a don't touch b. Linking only happens when something shares a box (a reference) or holds the other's address (a pointer), which is a deliberate, different act.
IndirectionAdvancedModule 06

Pointers — a value that is an address

If every box has an address, then an address is just a number — and a number can be stored in a box. A pointer is exactly that: a variable whose value is the address of another box. It doesn't hold the data; it holds where the data lives. That one extra hop — going through an address to reach a value — is called indirection, and it's the foundation of nearly every interesting data structure.

Two operators run the show. Address-of (&x) asks "where does x live?" and yields its address. Dereference (*p) does the reverse: "follow this address and give me the box there." Writing through a dereferenced pointer reaches across the indirection and changes the original:

cint x = 42;
int *p = &x;     // p holds the ADDRESS of x  (p → x)

printf("%d", *p);  // 42  — dereference: follow p to x's box
*p = 99;          // writes 99 into x, through the pointer
printf("%d", x);   // 99  — x changed, though we never named it

A pointer is itself an ordinary box — on a 64-bit machine it holds a 64-bit address, whatever it points to. A pointer that points at nothing is set to null (address 0); dereferencing it is the classic crash (a "segmentation fault" or null-reference error), because address 0 is off-limits. And because a pointer knows the type it points to, pointer arithmetic moves in whole elements: p + 1 advances by one int (4 bytes), not one byte — which is precisely how arrays and iteration work under the hood.

A pointer and the thing it points to are the same variable.
They're two separate boxes. The pointer's box holds an address; the target's box holds the value. Reassigning p changes which box it aims at; writing *p changes the box it currently aims at. Different boxes, different operations.
Pointers are an obsolete C thing modern languages removed.
They renamed and tamed them. Java and Python references, Go's pointers, C++ smart pointers, even a CPU's program counter are all addresses-to-boxes. The indirection never left; safer languages just stop you from doing arithmetic on it or dereferencing null.
IndirectionAdvancedModule 07

References & aliasing

A reference is a second name for an existing box. Where a pointer is a separate box that holds an address you must explicitly dereference, a reference simply is the original box under another label — the indirection is hidden and automatic. When two names refer to one box, they are aliases, and a write through either name is instantly visible through the other.

This distinction is the heart of pass-by-value versus pass-by-reference. Pass a primitive by value and the function receives a copy in a fresh box — scribbling on it leaves the caller's variable untouched. Pass by reference (or hand over a pointer) and the function operates on the same box, so its changes persist after the call. One is a photocopy; the other is the original document.

cvoid addOne(int n)  { n++; }       // by value: n is a copy, caller unchanged
void addOne(int &n) { n++; }       // by reference: n aliases the caller's box

Aliasing is where the famous surprises live. In languages like Java, Python, and JavaScript, variables that hold objects, lists, or arrays actually hold references. So b = a copies the reference, not the object — now a and b point at one shared box, and mutating through a changes what you see through b. This is the difference between a shallow copy (duplicate the reference, share the contents) and a deep copy (duplicate the contents too).

See copy versus alias side by side. Choose how b = a behaves, then change a and watch whether b follows. Value semantics gives two boxes; reference semantics gives one box with two names:

Copy or alias?
the same assignment, two very different outcomes
In value mode, b is an independent copy. In reference mode, a and b alias one box.
Passing an object to a function copies the object.
In reference-based languages it copies the reference — the function shares your actual object and can mutate it. To protect the original you must deliberately make a copy. (Primitives, by contrast, are usually passed by value.)
A reference and a pointer are completely different mechanisms.
Underneath, a reference is usually implemented as a pointer — an address to a box. The difference is ergonomics: a reference auto-dereferences and can't be null or do arithmetic, so it's a safer, narrower tool for the common case of "another name for this box."
MemoryAdvancedModule 08

The stack & the heap

A program's boxes live in two very different regions, and knowing which is which prevents a whole category of bugs. The stack is fast, automatic, and tied to function calls: each call pushes a frame holding that function's local variables, and returning pops it. Allocation is nearly free (just move a pointer), but a local's box vanishes the instant its function returns. The heap is a large pool for boxes whose size or lifetime isn't known in advance; you request space explicitly (malloc, new) and it lives until it's explicitly released or garbage-collected.

The lifetime traps

Mixing addresses with lifetimes is where memory bugs breed. Return a pointer to a local and you get a dangling pointer — it aims at a stack box that no longer exists. Free a heap box but keep using the pointer and that's a use-after-free. Free it twice, a double free. Never free it at all, a memory leak. Every one of these is really the same mistake: a pointer and the box it names disagreeing about whether the box is still alive.

This is exactly the problem different languages attack in different ways. Garbage collectors track which heap boxes are still reachable and reclaim the rest, trading some performance and predictability for safety. Rust's borrow checker proves at compile time that no pointer outlives its box. C hands you the keys and the responsibility. Same underlying hazard — addresses and lifetimes — managed along a spectrum from "you do it" to "the system does it."

The stack and heap are different physical hardware.
Both are just regions of the same RAM. The difference is the allocation discipline: the stack is a strict last-in-first-out frame per call; the heap is a general pool you carve up on demand. Same memory, different bookkeeping.
Garbage collection means you can ignore memory entirely.
It removes manual free, not thinking. You can still leak by keeping references to things you no longer need, still alias and mutate shared objects by surprise, and still pay GC pauses. It automates reclamation, not understanding.
ExpertModule 09

Memory layout & safety

Zoom in on how multi-byte values actually sit in memory and a few subtle, important details appear. The first is endianness: the order in which a value's bytes are laid out across consecutive addresses. Big-endian stores the most significant byte first (the way we write numbers); little-endian stores the least significant byte first. Most machines you use (x86, most ARM) are little-endian, which looks backwards in a memory dump but simplifies some hardware.

It usually doesn't matter — until bytes cross a boundary. Send an integer over a network, write it to a binary file, or reinterpret a buffer as a different type, and both sides must agree on byte order or the value scrambles. That's why network protocols define a canonical "network byte order" (big-endian) that everyone converts to and from.

Watch the same 32-bit value, 0x12345678, laid out both ways. Toggle the byte order and see how the four bytes rearrange across the addresses:

Byte order
one value, two layouts in memory
value: 0x12345678 (stored in 4 bytes)
Little-endian puts the least significant byte at the lowest address — it looks reversed.

Two more layout facts shape real programs. Alignment: the hardware prefers (or requires) a value to sit at an address that's a multiple of its size, so compilers insert invisible padding bytes inside structures to keep fields aligned — which is why a struct can be larger than the sum of its fields. And arrays are simply boxes laid end to end in one contiguous block, so arr[i] is really "start address + i × element size" — array indexing is pointer arithmetic in disguise.

Buffer overflow — when writing past the box corrupts the next one

Because arrays are contiguous and most languages don't check bounds at the hardware level, writing past the end of an array spills into whatever box comes next in memory — an adjacent variable, a saved return address, anything. That's a buffer overflow, and deliberately overwriting a return address to hijack control flow is one of the oldest and most damaging classes of security exploit. It is the dark side of "a value is just bits in a box": the boxes are right next to each other, and the bits don't know where one ends.

A struct's size is just the sum of its fields' sizes.
Often it's larger, because of alignment padding inserted to keep each field on a friendly address. Reorder fields from largest to smallest and you can shrink a struct without removing a single field — a real optimization in memory-heavy code.
Endianness affects how I write numbers in my source code.
No — 0x12345678 means the same value everywhere. Endianness only governs the byte layout in memory, so it surfaces only when you inspect raw bytes, serialize, or cast a pointer to a different type. The numeric value itself is unaffected.

How languages handle all this

Every idea so far is universal, but languages expose it along a spectrum from "raw and dangerous" to "managed and safe." Understanding where your language sits tells you which bugs you must guard against yourself and which the language handles for you.

A few patterns are worth pulling out. Primitives versus references: most managed languages keep primitives as by-value boxes but make objects references, so the same = copies a number but aliases a list — the single biggest source of "why did both variables change?" confusion. Fixed versus arbitrary width: Python integers simply grow more bytes as needed, so they never overflow (at the cost of speed), while JavaScript has no integer type at all — every number is a double, which is why it loses precision past 2⁵₃.

And the memory-safety story is the defining axis of modern language design. C trusts you completely. Garbage-collected languages remove manual freeing but keep aliasing and leaks. Rust's ownership system is the notable third path: it proves pointer safety at compile time, getting C-like control without a garbage collector — a direct response to decades of the lifetime traps from Module 08.

* JavaScript added BigInt for arbitrary-precision integers, but its default number is always a 64-bit float.

High-level languages don't have any of these low-level concerns.
They hide them, they don't remove them. Aliasing bugs, float imprecision, and silent integer wrap (or surprise big-int slowdowns) all show up in Python and JavaScript too. The bits-in-a-box model still explains the surprises — it's just one layer down.
A memory-safe language can't be fast.
Rust disproves it: ownership and borrowing are checked at compile time, so safety costs nothing at runtime. Safety and speed trade off against convenience (you do more up front), not necessarily against each other.
ExpertModule 11

Putting it together

Every topic in this course collapses into one mental model: a value is a pattern of bits in a box, and a name, pointer, or reference is a way of getting to that box. Hold that picture and the bugs stop being mysterious. They become predictable consequences of three questions you can ask about any variable.

How wide is the box? The width fixes the range, and the range is where overflow and float imprecision live. Multiplying two ints that each fit can still overflow; adding a small float to a huge one can do nothing. Pick widths that fit your real range, watch the edges of arithmetic, compare floats with a tolerance, and keep money in exact integer or decimal types — never a float.

Is this a copy or a shared box? Value semantics give you an independent duplicate; references and pointers give you a shared box where a change through one name is seen through every other. Most "why did that other variable change?" bugs are an unexpected alias. When you hand data to a function or assign it, know whether you're passing the document or a photocopy.

Who owns this box's lifetime? The moment you keep an address — a pointer, a reference, an index into something that can move — you've taken on a question about when that box is valid. Dangling pointers, use-after-free, and leaks are all lifetime disagreements. Garbage collectors, ownership systems, and plain discipline are just different answers to "when does this box die?"

The through-line

Width, copy-or-share, and lifetime aren't three topics — they're three views of the same box-of-bits. The reason this layer is worth understanding even in a language that hides it is that the hidden machinery still leaks through, as a wrong sum, a surprise mutation, or a value that vanished. Knowing what's underneath turns "spooky" into "oh, of course."

You began with a single bit — one switch, on or off — and built all the way up to the way entire languages manage memory and safety. The syntax of int, *, and & was never the point; the model underneath was. Now when a number wraps, a float drifts, or two variables move in lockstep, you won't reach for superstition. You'll reach for the boxes.

Now you can see the boxes

Primitive types, pointers, references, and overflow are not four separate subjects — they're four faces of one idea: bits in boxes, and the names we use to reach them. You can read a memory dump, predict an overflow, explain an aliasing bug, and reason about a lifetime, because you know what's actually there.

The growth from here is in noticing. Every wrong-looking number, every shared-mutation surprise, every crash on a stray address is the model asking to be applied. Keep asking how wide, copy or share, and who owns the lifetime — and the machine stops being magic.
