Across decades of software, the same design problems keep recurring — and a handful of proven solutions keep solving them. Design patterns are those solutions, given names so we can reach for them and talk about them. They aren't code to copy; they're shapes of thinking. This course teaches the classic catalog from the ground up, with the recurring problem made concrete and the pattern shown solving it — live, where watching it run is the point.

BeginnerModule 00

What is a design pattern?

Suppose you need part of your program to react whenever some value changes — a dashboard updating when data arrives, an email firing when an order ships. You could wire it up by hand. But this exact problem — "when one thing changes, notify many others" — has appeared in millions of programs, and someone has already worked out a clean, flexible solution to it. That solution has a name: Observer. Knowing the name means you can reach for the proven approach instead of reinventing it, and say one word to a teammate instead of explaining the whole design.

That's all a design pattern is: a named, reusable solution to a problem that recurs in software design. Each pattern captures the distilled experience of countless developers who solved the same problem and converged on the same shape. Patterns are a vocabulary for design as much as they are solutions.

Where they came from

The idea was borrowed from architecture — Christopher Alexander's pattern language for buildings — and brought to software in 1994 by four authors (the "Gang of Four") whose book cataloged 23 patterns. Their framing still holds: a pattern is a solution to a problem in a context. The context matters as much as the solution; a pattern that fits one situation is wrong in another.

The most important thing to understand first

A pattern is not a snippet of code to copy and paste. It's a template — a description of how classes and objects relate to solve a problem — that you adapt to your situation. Two implementations of the same pattern can look quite different in code while sharing the same underlying structure and intent. Learn the intent, not a canonical listing.

Three families

The classic catalog divides into three groups by what kind of problem each pattern addresses. This taxonomy organizes the entire course, and we'll color-code by it throughout:

Creational
How objects get created. They make a system independent of how its objects are made, freeing you from hard-coding concrete classes. Factory, Builder, Singleton…
Structural
How objects are composed into larger structures. They assemble objects and classes into bigger arrangements while keeping them flexible. Adapter, Decorator, Composite…
Behavioral
How objects communicate and divide responsibility. They concern the assignment of responsibilities and the flow of messages between objects. Strategy, Observer, State…

Creational patterns are about making, structural about arranging, behavioral about interacting. Keep that one-word summary in mind and most patterns slot naturally into place.

Patterns are tools, not trophies

The goal is never "use a pattern." The goal is to solve a problem well — and sometimes the right answer is plain, direct code. Reaching for patterns where they aren't needed ("patternitis") adds indirection and complexity for nothing. The skill this course builds is recognizing when a problem matches a pattern's intent, and having the judgment to leave patterns on the shelf when they don't. We'll return to this discipline in the final module.

This is the third course in The Growth Lab, alongside the ones on Big O complexity and data structures. Patterns operate on the structures from that course, and "favor composition" — coming up next — echoes how those structures were built by combining simpler pieces.

BeginnerModule 01

The principles underneath

Before any specific pattern, it's worth understanding the principles patterns are built on — because once you internalize these, many patterns feel less like things to memorize and more like the obvious consequence of applying a principle. Almost every pattern is one of these ideas, made concrete.

Encapsulate what varies

The single most important principle: identify the parts of your design that change, and separate them from the parts that stay the same. When you isolate what varies behind a stable interface, you can change or extend it without disturbing everything around it. Nearly every pattern is a specific technique for encapsulating some particular kind of variation — the algorithm (Strategy), the object being created (Factory), the behavior tied to state (State), and so on.

Program to an interface, not an implementation

Depend on abstractions — interfaces or abstract types — rather than concrete classes. When your code talks to an interface, any implementation of that interface can be swapped in without the code noticing. This is what makes substitution, testing with fakes, and runtime flexibility possible.

typescript// Coupled to a concrete class — rigid
class Report {
  save() { new MySQLDatabase().write(this); }   // locked to MySQL forever
}

// Programmed to an interface — flexible
interface Database { write(data): void; }
class Report {
  constructor(private db: Database) {}      // any Database works
  save() { this.db.write(this); }              // MySQL, Postgres, an in-memory fake for tests…
}

Favor composition over inheritance

This one deserves special attention, because it underlies a huge fraction of patterns. Inheritance ("a Dog is-an Animal") is rigid: it's fixed at compile time, a subclass is tightly bound to its parent, and trying to vary behavior along several dimensions causes a combinatorial explosion of subclasses. Composition ("a Car has-an Engine") assembles behavior from interchangeable parts you can swap at runtime.

the inheritance trap// Want characters × weapons. With inheritance, every combination is a class:
QueenWithSword, QueenWithBow, KnightWithSword, KnightWithBow, KnightWithAxe, ...
// 5 characters × 4 weapons = 20 classes. Add one weapon → 5 more classes.

// With composition, behavior is a part you plug in:
class Character { constructor(public weapon: Weapon) {} }
character.weapon = new Bow();   // swap at runtime, 5 + 4 = 9 classes total

Composition is more flexible because the relationship is established at runtime and can change; the pieces are decoupled and independently testable. The cost is a bit more indirection. This principle is the engine behind Strategy, Decorator, State, Bridge, and more — wherever you see an object holding a reference to another object that it delegates to, you're seeing "composition over inheritance" at work.

SOLID — five principles for maintainable OO

A widely used checklist that captures much of what makes object-oriented code resilient to change:

LetterPrincipleIn one sentence
SSingle ResponsibilityA class should have one reason to change.
OOpen / ClosedOpen for extension, closed for modification — add behavior without editing existing code.
LLiskov SubstitutionSubtypes must be usable anywhere their base type is expected.
IInterface SegregationMany small, specific interfaces beat one fat one — don't force clients to depend on methods they don't use.
DDependency InversionDepend on abstractions, not concrete implementations.
How this connects to everything ahead

Watch for these principles surfacing in every pattern. The Open/Closed principle is the explicit goal of Strategy and Decorator (extend behavior without touching existing code). Dependency Inversion is why Factory and Abstract Factory exist (depend on a product interface, not concrete products). Single Responsibility motivates Command and Mediator (give each responsibility its own object). Patterns are largely these principles crystallized into reusable structures — which is why understanding the principles first makes the patterns click.

CreationalBeginner → IntermediateModule 02

Factory & Abstract Factory

The simplest way to make an object is new ConcreteThing() — and it's also a quiet source of rigidity. The moment you write new, your code is welded to one specific class. Factory patterns exist to break that weld, letting you decide what to create without your client code knowing the concrete type.

Factory Method creational

Intent

Define an interface for creating an object, but let the decision of which class to instantiate be made elsewhere — so client code depends only on the product's interface.

The idea: instead of scattering new calls for concrete classes throughout your code, funnel object creation through a single factory that returns the common interface. Clients ask the factory for a product and work with the abstraction; swapping or adding product types touches only the factory.

Factory lab
ask for a product by type — the factory makes the right one
shapeFactory.create( )
Click a product type. Your client code calls one method and never says new Circle() itself.
typescriptinterface Shape { draw(): void; }
class Circle implements Shape { draw() { ... } }
class Square implements Shape { draw() { ... } }

class ShapeFactory {
  create(kind: string): Shape {        // returns the INTERFACE, not a concrete type
    switch (kind) {
      case "circle": return new Circle();
      case "square": return new Square();
    }
  }
}
// Client never mentions Circle or Square — only Shape and the factory.
const shape = factory.create(userChoice);
shape.draw();

Abstract Factory creational

Intent

Provide an interface for creating families of related objects without specifying their concrete classes — so the whole family swaps together consistently.

Where Factory Method makes one product, an Abstract Factory makes a coordinated set. The classic example is cross-platform UI: a MacFactory produces a Mac-styled button, checkbox, and scrollbar; a WindowsFactory produces the Windows-styled versions. Client code asks the factory for "a button" and "a checkbox," and because they come from the same factory, they always match. Switching the entire look means swapping one factory.

Factory Method vs Abstract Factory

The difference is scope. Factory Method creates one product through a single method. Abstract Factory creates a family of related products through several methods, guaranteeing the pieces are compatible. Both share the same goal: decouple client code from concrete classes by depending on interfaces — a direct application of the Dependency Inversion principle from the last module.

CreationalIntermediateModule 03

Builder, Prototype, Singleton

Three more creational patterns, each addressing a different awkwardness in object creation: complex construction, expensive copying, and controlled instantiation.

Builder creational

Intent

Separate the construction of a complex object from its representation, building it step by step — so the same process can produce different configurations.

When an object has many parts or options, a constructor with ten parameters (the "telescoping constructor") becomes unreadable and error-prone. Builder lets you assemble the object incrementally with named, readable steps, often as a fluent chain, and produce the finished product at the end. Configure one below and watch it assemble:

assemble a complex object one readable step at a time
fluent construction
resulting product
Toggle parts to add them. Each call returns the builder, so they chain — then build() yields the finished object.

Prototype creational

Intent

Create new objects by cloning an existing instance rather than building from scratch — useful when construction is expensive or you want copies of a pre-configured object.

Instead of new and re-running costly setup, you keep a fully configured prototype and call clone() on it. This shines when an object is expensive to create (heavy initialization, database loads) or when you want many near-identical copies of a complex configuration. The key subtlety is deep vs shallow copy: a shallow clone shares nested objects with the original, which can cause surprising aliasing bugs, while a deep clone duplicates the whole graph.

Singleton creational

Intent

Ensure a class has exactly one instance, and provide a single global point of access to it.

The classic use is for something there should genuinely be only one of — a configuration registry, a connection pool, a logger. The implementation hides the constructor and exposes a static accessor that lazily creates the instance once and returns it thereafter.

Singleton is the most controversial pattern

Use it sparingly. A Singleton is global mutable state wearing a nicer outfit, and global state brings real problems: it creates hidden dependencies (a class secretly reaches for the Singleton instead of declaring what it needs), it makes testing hard (you can't easily substitute a fake, and state leaks between tests), and it can cause issues under concurrency. Many experienced developers consider it an anti-pattern in most cases. The modern alternative is dependency injection — create one instance and explicitly pass it to the code that needs it, keeping the "one instance" benefit without the hidden global. Reach for Singleton only when global access is truly the point.

StructuralIntermediateModule 04

Adapter & Facade

Structural patterns are about composing objects into larger, more useful arrangements. The first two are about interfaces: Adapter makes incompatible things work together, and Facade makes complicated things simple to use.

Adapter structural

Intent

Convert the interface of a class into another interface that clients expect — letting classes work together that otherwise couldn't because of incompatible interfaces.

The everyday metaphor is a power-plug adapter: your laptop's plug doesn't fit a foreign socket, so a small adapter sits between them, translating one shape into another. In code, an Adapter wraps an existing class (often third-party or legacy, which you can't change) and exposes the interface your code already speaks, translating calls across the gap.

typescript// Your code speaks this interface:
interface JsonLogger { log(msg: string): void; }

// But a third-party library only offers this incompatible one:
class LegacyXmlLogger { writeXml(xml: string) { ... } }

// The adapter implements YOUR interface and translates to THEIRS:
class XmlLoggerAdapter implements JsonLogger {
  constructor(private legacy: LegacyXmlLogger) {}
  log(msg: string) { this.legacy.writeXml(toXml(msg)); }   // translate across the gap
}

Facade structural

Intent

Provide a single, simplified interface to a complex subsystem — so clients can use the common cases easily without wrestling with all the moving parts.

A complex subsystem might involve a dozen classes that must be used together in just the right order. A Facade is a single class that wraps that complexity behind a handful of simple methods. Think of startCar() hiding the fuel pump, ignition, starter motor, and electrical system. The subsystem is still there and still usable directly for advanced needs — the Facade just offers an easy front door for the common path.

Adapter vs Facade — a clean distinction

Both wrap things, but for opposite reasons. Adapter changes an interface to match what a client already expects — it's about compatibility, and usually wraps a single object. Facade invents a new, simpler interface over a whole subsystem — it's about convenience, and usually wraps many objects. Adapter makes the incompatible compatible; Facade makes the complicated simple.

StructuralIntermediate → AdvancedModule 05

Decorator & Composite

Two structural patterns built on the "favor composition" principle, both achieving with object-wrapping what would otherwise need an explosion of subclasses.

Decorator structural

Intent

Attach additional responsibilities to an object dynamically by wrapping it in another object with the same interface — a flexible alternative to subclassing for extending behavior.

A decorator implements the same interface as the thing it wraps, holds a reference to it, and adds behavior before or after delegating to the wrapped object. Because decorators share the interface, they stack: each wraps the previous, and the composed stack still looks like a single object to the client. Build a drink below — each condiment is a decorator wrapping the one beneath it, and the cost accumulates through the layers:

Toggle condiments. cost() at the outer layer calls inward — each wrapper adds its price to whatever it wraps.
typescriptinterface Drink { cost(): number; desc(): string; }
class Coffee implements Drink { cost() { return 2.00; } desc() { return "Coffee"; } }

// A decorator wraps a Drink and IS a Drink:
class Milk implements Drink {
  constructor(private inner: Drink) {}            // holds what it wraps
  cost() { return this.inner.cost() + 0.50; }       // add, then delegate inward
  desc() { return this.inner.desc() + ", Milk"; }
}
let drink = new Whip(new Milk(new Coffee()));   // stacked wrappers
drink.cost();  // 2.00 + 0.50 + 0.60, computed through the layers

This is exactly how Java's I/O streams work — new BufferedReader(new InputStreamReader(new FileInputStream(...))) stacks buffering and decoding decorators around a raw file stream. Adding a new behavior means writing one new decorator, never touching the existing classes — the Open/Closed principle in action.

Composite structural

Intent

Compose objects into tree structures to represent part-whole hierarchies, and let clients treat individual objects and whole compositions uniformly.

The key move: both the "leaf" (an individual item) and the "composite" (a container of items) implement the same interface. So a client can call getSize() on a file or a folder without caring which it is — a folder simply asks each of its children and sums the results, recursing all the way down. A file system is the perfect example. Click below to compute the total size and watch the operation recurse through the tree:

Composite lab

A folder's size is the sum of its children's — the same getSize() call works on every node, recursing to the leaves.
Decorator vs Composite — same skeleton, different intent

Both build recursive object structures where wrappers share the wrapped interface. But Decorator wraps one object to add behavior (a single chain), while Composite holds many children to model a part-whole tree. Decorator is about layering responsibilities; Composite is about treating a hierarchy as a single thing.

StructuralAdvancedModule 06

Proxy, Bridge, Flyweight

Three more structural patterns, each a sharper, more specialized tool: controlling access, splitting two axes of variation, and sharing memory at scale.

Proxy structural

Intent

Provide a surrogate or placeholder for another object to control access to it — sharing the real object's interface so clients can't tell the difference.

A proxy implements the same interface as the real object and stands in front of it, adding a layer of control. The variations are defined by what that control does: a virtual proxy delays creating an expensive object until it's actually needed (lazy loading), a protection proxy checks permissions before forwarding, a remote proxy represents an object living on another machine, and a caching proxy stores results to skip repeated work.

typescriptinterface Image { display(): void; }

class RealImage implements Image {        // expensive: loads from disk on creation
  constructor(private file: string) { loadFromDisk(file); }
  display() { ... }
}
class ImageProxy implements Image {       // same interface, defers the cost
  private real?: RealImage;
  constructor(private file: string) {}
  display() {
    if (!this.real) this.real = new RealImage(this.file);  // load only on first real use
    this.real.display();
  }
}
Proxy vs Decorator vs Adapter — the wrapper family

All three wrap an object behind the same or a different interface, which is why they blur together. The intent separates them: Adapter changes the interface (compatibility); Decorator keeps the interface and adds behavior, stackably (enhancement); Proxy keeps the interface and controls access, usually without changing what the operation does (access control). Same shape, three different jobs.

Bridge structural

Intent

Decouple an abstraction from its implementation so the two can vary independently — avoiding a combinatorial explosion when there are two separate dimensions of change.

When you have two independent axes — say shapes (circle, square) and rendering APIs (vector, raster) — inheritance forces you to create a class for every combination (VectorCircle, RasterCircle, …). Bridge splits them into two hierarchies connected by composition: the shape has-a renderer. Now m × n combinations need only m + n classes, and you can add a shape or a renderer without touching the other side. It's the "favor composition" principle applied to a two-dimensional problem.

Flyweight structural

Intent

Use sharing to support huge numbers of fine-grained objects efficiently, by separating the state they can share from the state unique to each.

When you need millions of similar objects, the memory adds up fast. Flyweight splits each object's state into intrinsic state (shared, context-independent — stored once and reused) and extrinsic state (unique per use — passed in by the client). A text editor doesn't store a font and glyph shape for every single character; it shares one flyweight per character type, and the position is passed in. A forest of 100,000 trees stores a handful of shared tree-models (mesh, texture) plus a lightweight per-tree position and scale. The savings can be enormous.

BehavioralIntermediateModule 07

Strategy & Observer

Behavioral patterns govern how objects communicate and divide responsibility. These two are arguably the most-used patterns in all of software — worth knowing cold.

Strategy behavioral

Intent

Define a family of interchangeable algorithms, encapsulate each one, and make them swappable — letting the algorithm vary independently from the code that uses it.

Instead of baking an algorithm into a class (or branching through a giant if/else to pick one), you extract each algorithm into its own object behind a common interface, and let the context hold a reference to whichever one it should use. Swapping behavior becomes swapping an object — at runtime, even. Below, the checkout context stays fixed while you swap the pricing strategy:

Strategy lab
same context, swap the algorithm, different result
cart subtotal: $120.00  ·  pick a pricing strategy:
The Checkout never changes — only the strategy injected into it does. New pricing rule = new strategy class, nothing else touched.
typescriptinterface PricingStrategy { apply(total: number): number; }
class MemberPricing implements PricingStrategy { apply(t) { return t * 0.9; } }
class VipPricing    implements PricingStrategy { apply(t) { return t * 0.75; } }

class Checkout {
  constructor(private pricing: PricingStrategy) {}   // holds a strategy
  process(total) { return this.pricing.apply(total); }    // delegates to it
}
checkout.pricing = new VipPricing();   // swap behavior at runtime
In modern languages, Strategy is often just a function

Where languages have first-class functions, a full strategy interface and classes are frequently overkill — you can pass the algorithm as a plain function or lambda: checkout.process(total, t => t * 0.9). The pattern (encapsulate and inject the varying algorithm) is identical; the ceremony melts away. This is a recurring theme we'll examine in the modern-world module.

Observer behavioral

Intent

Define a one-to-many dependency so that when one object (the subject) changes state, all its dependents (observers) are notified and updated automatically.

The subject keeps a list of observers and, on any change, calls a notify method on each. Observers can subscribe and unsubscribe at runtime, and the subject neither knows nor cares what they do with the news — it's completely decoupled from them. Change the temperature below and watch the update fan out to every subscriber; toggle subscribers on and off to see the dynamic relationship:

Observer lab
one change, broadcast to many — added or removed at will
subscribers:
Click "Weather changes" — the subject notifies every subscriber at once. Unsubscribe one and it stops getting updates, with zero change to the subject.
typescriptinterface Observer { update(temp: number): void; }

class WeatherStation {
  private observers: Observer[] = [];
  subscribe(o: Observer)   { this.observers.push(o); }
  unsubscribe(o: Observer) { /* remove from list */ }
  setTemp(t: number) {
    this.observers.forEach(o => o.update(t));   // notify all — subject doesn't know who they are
  }
}
Observer is the heart of reactive software

This pattern underlies an enormous amount of modern code: UI event systems (a click notifies its listeners), the Model-View-Controller architecture (the model notifies views to re-render), spreadsheet cells recalculating when a dependency changes, and reactive libraries like RxJS. Whenever you've written addEventListener or .subscribe(), you've used Observer.

BehavioralAdvancedModule 08

State & Command

Two powerful behavioral patterns: one lets an object's behavior change with its internal mode, the other turns actions themselves into objects you can store, queue, and undo.

State behavioral

Intent

Allow an object to alter its behavior when its internal state changes — so it appears to change its class, and sprawling conditionals disappear.

Without the pattern, an object with modes is riddled with if (state === ...) checks in every method. The State pattern extracts each state into its own object that knows how to handle events in that state, including which state to transition to next. The context simply delegates to its current state object. The result below is a media player: notice how the same button does different things — or nothing — depending on the current state.

State machine lab
the same action behaves differently per state
media player
STOPPED
Press Play from Stopped → it plays. Press Play while already Playing → nothing. Each state object decides how to respond.
typescriptinterface PlayerState { play(p): void; pause(p): void; stop(p): void; }

class Stopped implements PlayerState {
  play(p)  { p.state = new Playing(); }   // transition by swapping the state object
  pause(p) { /* ignored — can't pause when stopped */ }
  stop(p)  { /* already stopped */ }
}
class Player {
  state: PlayerState = new Stopped();
  play()  { this.state.play(this); }     // delegate to current state — no if/else
}
Strategy vs State — the classic confusion

They share an identical structure — a context delegating to a swappable object — but their intent differs sharply. With Strategy, the client chooses the algorithm, and strategies are independent and unaware of one another. With State, the states themselves drive transitions to other states based on internal logic, modeling a lifecycle the object moves through. Strategy is about how to do something; State is about what mode you're in. Same skeleton, opposite purpose.

Command behavioral

Intent

Encapsulate a request as an object — letting you parameterize, queue, log, and (crucially) undo operations.

Turning an action into an object sounds abstract until you see what it buys you. A command object bundles a receiver and an operation, with execute() and often undo() methods. Because actions are now objects, you can keep a history of them — which is exactly how undo/redo works. Run some operations below and watch the command history build into an undo stack:

Where Command shows up

Undo/redo in every editor, menu and button actions decoupled from their handlers, task queues and job schedulers (a command is a unit of work to run later), transaction logs, and macro recording. Pairing Command with the Memento pattern (which snapshots state) is the standard recipe for robust undo.

BehavioralAdvancedModule 09

The rest of the behavioral set

The behavioral family is the largest. Here's the remainder — some you'll use often, some you'll meet rarely but should recognize. We'll go deeper on the most practical and lighter on the niche.

Chain of Responsibility behavioral

Intent

Pass a request along a chain of handlers; each either handles it or forwards it to the next — decoupling the sender from whoever ultimately handles it.

Rather than one object deciding who handles a request, you string handlers together and let the request travel down the line until someone takes it. Below, an expense approval flows through an authority chain — each handler approves it if it's within their limit, otherwise passes it up:

Chain of Responsibility lab
a request travels until a handler can deal with it
expense:
Submit an amount and watch it pass up the chain — Team Lead ($1k), Manager ($10k), Director ($100k), CEO — until someone can approve it.

This is the pattern behind middleware (each layer in an Express or ASP.NET pipeline handles or forwards a request), event bubbling in the DOM, and tiered approval or logging systems.

The remaining patterns, briefly

Template Method
Define an algorithm's skeleton in a base method, letting subclasses fill in specific steps without changing the overall structure. Uses inheritance where Strategy uses composition — the steps are fixed at compile time. The backbone of many frameworks' "override these hooks" designs.
Iterator
Access the elements of a collection sequentially without exposing its internal structure. So fundamental it's now built into most languages (for…of, forEach, generators) — you use it constantly without naming it.
Mediator
Centralize complex communication between objects in one mediator, so they don't reference each other directly. Turns a tangle of many-to-many links into a clean hub-and-spoke. Think a chat room coordinating users, or a dialog coordinating its widgets.
Memento
Capture an object's internal state so it can be restored later, without exposing its internals. The snapshot half of undo — typically paired with Command.
Visitor
Add new operations to an object structure without modifying its classes, by moving the operation into a visitor object (via double dispatch). Powerful for stable hierarchies needing many operations — e.g., walking an abstract syntax tree. One of the more complex patterns.
Interpreter
Define a grammar for a simple language and an interpreter that evaluates sentences in it. Niche — used for things like query languages or rule engines, and usually superseded by parser tools.
Template Method vs Strategy, one more time

They solve the same goal — varying part of an algorithm — by opposite means. Template Method uses inheritance: a subclass overrides specific steps, chosen at compile time. Strategy uses composition: a strategy object is plugged in, swappable at runtime. The trade is the familiar one from Module 01 — inheritance is simpler but rigid; composition is flexible but adds indirection.

ExpertModule 10

Patterns in the modern world

The classic catalog was written in 1994, for the object-oriented languages of that era. Much of it remains essential — but expert-level understanding means knowing how patterns have shifted: some dissolved into language features, some became the bones of the frameworks you use daily, and some are best avoided.

When a pattern becomes a language feature

Several GoF patterns existed partly to work around limitations of 1990s languages. As languages gained features, the patterns didn't disappear — their intent remained vital — but the heavy machinery melted away:

Classic patternModern form
Strategy / CommandA first-class function or lambda — just pass the behavior directly, no interface or class needed.
IteratorBuilt into the language — for…of, generators, yield.
ObserverNative events and reactive streams (addEventListener, RxJS, signals).
SingletonA module (imported once) or a DI container managing a single instance.
PrototypeBuilt-in cloning / spread syntax; JavaScript's object model is literally prototypal.
The distinction that matters

Separate the intent of a pattern from its 1994 implementation. "Encapsulate an interchangeable algorithm and inject it" is timeless wisdom; doing it with a five-class hierarchy when a one-line lambda suffices is cargo-culting. Modern developers apply the thinking behind patterns while using the lightest construct the language offers.

Patterns hiding in your frameworks

You're already using patterns constantly, often without the labels. They're baked into the tools:

  • React is a masterclass in "composition over inheritance" — components compose rather than inherit. Hooks and render props are Strategy-like (inject behavior); Context is part Observer (components react to changes), part Dependency Injection (supply values down the tree); higher-order components and wrappers are Decorator-shaped.
  • Dependency injection frameworks (Spring, Angular, NestJS) are the industrial-strength answer to Factory and Singleton — they construct and supply your dependencies so your code depends only on interfaces.
  • Middleware in web frameworks is Chain of Responsibility; ORMs use Proxy for lazy loading; streams use Decorator and Iterator.

Architectural patterns — patterns at scale

Beyond the object-level GoF catalog sit larger architectural patterns that organize whole applications, often built out of the smaller patterns. MVC and MVVM separate data, presentation, and logic (and lean heavily on Observer to keep views in sync). Publish-subscribe and event-driven architectures scale Observer across services. Repository abstracts data access; dependency injection wires systems together. The same principles — encapsulate what varies, depend on abstractions, favor composition — operate at every level, from a single class to a distributed system.

The anti-patterns to avoid

Knowing patterns includes knowing their failure modes — the "patterns" that are really warnings:

  • Patternitis / over-engineering — forcing patterns where simple code would do, adding layers of indirection that obscure rather than clarify. The most common mistake of developers who just learned patterns.
  • The Golden Hammer — "when all you have is a hammer, everything looks like a nail." Applying your favorite pattern to every problem.
  • God Object — one class that knows and does everything, the opposite of Single Responsibility.
  • Premature abstraction — building elaborate flexibility for variation that never arrives. Abstractions have a cost; earn them with real, present need.
ExpertModule 11

Choosing & combining patterns

You now know the catalog. The expert skill is selection and restraint — recognizing when a problem genuinely matches a pattern's intent, telling apart the patterns that look alike, and combining them when one isn't enough. This final module turns the whole course into judgment.

A decision process

  1. Name the real problem first. What is actually varying or causing pain — object creation, an incompatible interface, tangled communication, a sprawling conditional? The problem points to the family before the pattern.
  2. Match intent, not shape. Patterns are indexed by intent. "I need to swap an algorithm" → Strategy. "I need behavior to change with mode" → State. "I need to notify many things of a change" → Observer.
  3. Prefer the simplest thing that works. A function beats a Strategy hierarchy; a plain object beats a Builder; direct code beats a pattern you don't yet need. Add a pattern when the pain is real and present, not hypothetical.
  4. Expect to combine. Real designs use several patterns together — a Factory producing Strategy objects, Command with Memento for undo, Composite whose nodes are Decorated.

Telling the look-alikes apart

The patterns most often confused share structure but differ in intent. This table is worth committing to memory:

These look similar…but differ in intent
Strategy vs StateStrategy: client picks an interchangeable algorithm. State: states transition between themselves over a lifecycle.
Adapter vs Decorator vs ProxyAdapter changes the interface; Decorator keeps it and adds behavior (stackable); Proxy keeps it and controls access.
Decorator vs CompositeDecorator wraps one object to add behavior; Composite holds many children as a part-whole tree.
Strategy vs Template MethodStrategy injects behavior via composition (runtime); Template Method overrides steps via inheritance (compile time).
Factory Method vs Abstract FactoryFactory Method creates one product; Abstract Factory creates a whole family of related products.
Facade vs AdapterFacade simplifies a whole subsystem (convenience); Adapter converts one interface to another (compatibility).

Misconceptions that survive to senior level

More patterns = better code
Patterns add indirection. The best code uses the fewest that clearly solve the problem — often none.
A pattern is specific code to copy
It's a template adapted to context. Two correct implementations can look very different.
GoF patterns are universal truth
They reflect 1990s OOP. In modern languages some are trivial or unnecessary; the intent endures, the ceremony doesn't.
Singleton is a good default
It's global state in disguise — usually an anti-pattern. Prefer dependency injection.
Strategy and State are different shapes
Same structure, opposite intent: client-chosen algorithm vs self-driven lifecycle.
Inheritance is the OO tool
Composition is usually more flexible. Most patterns prove it.

Capstone problems

Reason through each before revealing the answer. If all four feel natural, you have the design judgment this course set out to build.

Problem 1 · the sprawling conditional

An order object has a method full of if (status === "pending") … else if (status === "shipped") … repeated across many methods, and a new status keeps breaking things. Which pattern, and why?

▶ Reveal answer

State. The behavior changes with an internal mode and the same conditionals are duplicated across methods — the textbook signal. Extract each status into a state object that handles the order's operations for that status and knows the valid transitions. The order delegates to its current state; the conditionals vanish, and adding a status means adding one class (Open/Closed) rather than editing every method.

Problem 2 · the payment plugins

Your checkout must support credit card, PayPal, and crypto today, with more payment methods added by different teams over time. You want to add methods without touching checkout code. Pattern?

▶ Reveal answer

Strategy. Each payment method is an interchangeable algorithm behind a common PaymentMethod interface; checkout holds whichever one the user selected and delegates to it. A new method is a new strategy class — checkout never changes (Open/Closed, Dependency Inversion). In a modern language, each strategy could even be a function. Note it's Strategy, not State: the client picks the method, and the methods don't transition between each other.

Problem 3 · the legacy integration

You're integrating a third-party analytics SDK whose method names and data shapes don't match the clean Tracker interface the rest of your app uses — and you can't modify the SDK. Pattern?

▶ Reveal answer

Adapter. Wrap the SDK in a class that implements your Tracker interface and translates each call into the SDK's vocabulary. Your app keeps talking to Tracker, oblivious to what's behind it, and if you switch analytics vendors later you write a new adapter instead of touching app code. (If instead you wanted to simplify a whole complicated subsystem behind one easy interface, that would be Facade — but here the issue is interface incompatibility, which is Adapter's job.)

Problem 4 · the editor with undo

You're building a graphics editor. Users perform many kinds of actions (add shape, move, recolor, delete) and need unlimited undo and redo. Which pattern (or combination)?

▶ Reveal answer

Command, often combined with Memento. Model every action as a command object with execute() and undo(); push executed commands onto a history stack. Undo pops and reverses; redo replays. For actions whose reversal is hard to compute, pair with Memento to snapshot the relevant state before the change and restore it on undo. This is the canonical demonstration of the course's recurring lesson — real solutions combine patterns: Command supplies the action-as-object and the history, Memento supplies reliable restoration.
